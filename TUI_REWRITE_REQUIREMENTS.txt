TUI REWRITE REQUIREMENTS
========================
Comprehensive functional requirements for a complete rewrite of the
review-agent-orchestrator TUI using correct React patterns and Ink library
best practices.

Generated from a full codebase audit of the current implementation.
Date: 2025-03-06


================================================================================
PART 1: ARCHITECTURE & PATTERNS
================================================================================

1.1  COMPONENT MODEL
---------------------
REQ-ARCH-001: All components MUST be functional components using hooks.
             No class components. The current codebase uses class components
             for App, DetailView, HomeView, and IssueListPicker — all must be
             rewritten as functional components.

REQ-ARCH-002: State MUST be managed via useState/useReducer hooks, never via
             instance variables with forceUpdate(). The current codebase stores
             state as class instance fields (e.g., DetailView.inputValue,
             DetailView.threadStack, DetailView.selectedMessage) and calls
             this.forceUpdate() — this is an anti-pattern.

REQ-ARCH-003: Side effects MUST use useEffect with proper dependency arrays.
             The current codebase has no useEffect usage; lifecycle logic is
             in componentDidMount or inline in event handlers.

REQ-ARCH-004: Memoization MUST be applied where appropriate using useMemo and
             useCallback. The current codebase creates new arrow functions in
             render props on every render cycle (e.g., onSelect={() => ...} in
             App.render()).

REQ-ARCH-005: Input handling MUST use Ink's useInput hook directly in the
             component that owns the behavior, not via bridge components. The
             current codebase uses a DetailInputBridge wrapper component solely
             to call useInput and forward to a class method — this is
             unnecessary with functional components.

REQ-ARCH-006: Focus management SHOULD use Ink's useFocus and useFocusManager
             hooks where possible, instead of the current manual focus ring
             implementation (DetailView.cycleFocus with manual index math).

REQ-ARCH-007: All text content MUST be wrapped in <Text> components. No raw
             strings outside <Text>. No <Box> inside <Text>.

REQ-ARCH-008: Layout MUST use Ink's <Box> component with flexbox properties.
             No manual string padding or space calculation where flexbox can
             achieve the same result.


1.2  STATE MANAGEMENT
---------------------
REQ-STATE-001: Navigation state (view stack) MUST be managed via useReducer
              with a well-defined action type union. Actions: PUSH_VIEW,
              POP_VIEW, REPLACE_VIEW, GO_HOME.

REQ-STATE-002: A React Context MUST provide navigation actions to all views,
              eliminating prop drilling of onBack/onHome/onNavigate/onQuit
              through every component.

REQ-STATE-003: Thread navigation state (threadStack) MUST be managed via
              useState or useReducer within the DetailView component, not as
              a mutable instance array.

REQ-STATE-004: Selected message index MUST be managed via useState within
              DetailView, not as a class instance variable.

REQ-STATE-005: The "saved selected message" map (persists cursor position when
              navigating away from a Detail view) MUST be managed via useRef
              or lifted into the navigation context.

REQ-STATE-006: Global application state (unread counts, active agent count,
              thread info for header/footer) SHOULD use a shared context
              rather than prop threading from App to Header/Footer.

REQ-STATE-007: Mock data references MUST NOT be mutated in place. State updates
              MUST produce new objects/arrays (immutable updates). The current
              codebase mutates MOCK_ISSUES, MOCK_DETAIL_DATA arrays directly
              and relies on forceUpdate to re-render.


1.3  INK BEST PRACTICES
------------------------
REQ-INK-001: Use incremental rendering (incrementalRendering option) to avoid
            full terminal rewrites on every frame.

REQ-INK-002: Respect maxFps throttling (default 30). Do not trigger unnecessary
            re-renders.

REQ-INK-003: Use useApp() hook for exit handling instead of process.exit().

REQ-INK-004: Use useStdout() for terminal dimension access instead of
            process.stdout directly.

REQ-INK-005: Screen clearing MUST NOT use raw ANSI escape sequences
            (process.stdout.write('\x1B[2J\x1B[H')). Use Ink's built-in
            clear mechanisms or component remounting.

REQ-INK-006: Use <Static> component for any persistent output that should
            remain visible above dynamic content (e.g., completed operation
            logs).

REQ-INK-007: Use <Spacer> for flexible spacing instead of manual dash/space
            string construction where applicable.

PART 2: VIEW SYSTEM & NAVIGATION
================================================================================

2.1  VIEW TYPES
---------------
REQ-VIEW-001: The application MUST support the following view types:
             - Home (issue dashboard)
             - Detail (issue detail with conversation)
             - NewIssue (issue creation form)
             - AgentStatus (agent monitoring table)
             - BlockingMap (dependency graph visualization)
             - GroupView (container/group browser)
             - IssuePicker (issue selection overlay for blockedBy/blocks)

REQ-VIEW-002: Views MUST be defined as a discriminated union type:

             type View =
               | { type: 'Home' }
               | { type: 'Detail'; inum: number }
               | { type: 'NewIssue' }
               | { type: 'AgentStatus' }
               | { type: 'BlockingMap' }
               | { type: 'GroupView' }
               | { type: 'IssuePicker'; mode: 'blockedBy' | 'blocks'; inum: number }


2.2  NAVIGATION STACK
---------------------
REQ-NAV-001: The app MUST maintain a view stack for back-navigation.

REQ-NAV-002: Push: navigateToView(view) pushes a new view onto the stack.

REQ-NAV-003: Pop: goBack() pops the top view. MUST NOT pop past the root view.

REQ-NAV-004: Replace: replaceCurrentView(view) replaces the top of the stack
            without adding history. Used for adjacent issue navigation
            (Shift+Left/Right).

REQ-NAV-005: Home: goHome() resets the stack to [{ type: 'Home' }].

REQ-NAV-006: Screen MUST clear on every navigation transition.

REQ-NAV-007: Footer focus index MUST reset to null on every navigation.

REQ-NAV-008: When navigating away from DetailView, the current selectedMessage
            index MUST be saved (keyed by inum) and restored when returning.


2.3  GLOBAL KEYBOARD SHORTCUTS
-------------------------------
REQ-KEY-001: The following global shortcuts MUST be available from all views:
            - Escape: go back (pop view stack)
            - q: quit the application
            - s: navigate to AgentStatus (unless already there)
            - b: navigate to BlockingMap (unless already there)
            - g: navigate to GroupView (unless already there)
            - n: navigate to NewIssue

REQ-KEY-002: Global shortcuts MUST NOT push duplicate views onto the stack.
            If the current view matches the target, the shortcut is a no-op.

REQ-KEY-003: Global shortcut handling MUST be a reusable hook or utility
            consumed by each view, not duplicated logic.

PART 3: LAYOUT COMPONENTS
================================================================================

3.1  APP SHELL
--------------
REQ-SHELL-001: The app shell MUST render three zones vertically:
              1. Header (fixed height)
              2. Content area (flexible, fills remaining space)
              3. Footer (variable height based on shortcuts)

REQ-SHELL-002: The shell MUST constrain total height to terminal rows.

REQ-SHELL-003: Terminal resize events MUST trigger re-render with new
              dimensions.


3.2  HEADER
-----------
REQ-HDR-001: Header height is exactly 3 lines.

REQ-HDR-002: Line 1: Centered title with horizontal rule characters.
            Format: "--- Review Agent Orchestrator - <ViewLabel> ---"
            The view label changes per active view:
            - Home: "Dashboard"
            - Detail: "I-<inum> Detail"
            - NewIssue: "New Issue"
            - AgentStatus: "Agent Status"
            - BlockingMap: "Blocking Map"
            - GroupView: "Groups"
            - IssuePicker: "Issue Picker"

REQ-HDR-003: Line 2: Status bar showing:
            - "Agents: <active>/<max>" (colored green if any active)
            - "Unread: <count>" (colored yellow if > 0)

REQ-HDR-004: Line 3: Context-sensitive subtitle (dimmed):
            - Home: "All issues and orchestration state"
            - Detail (not in thread): empty or hint text
            [CHANGED] - Detail (in thread): "(add a response)"
              ("^R to resolve" removed from subtitle)
            - NewIssue: "Create a new issue"
            - AgentStatus: "Active agent sessions"
            - BlockingMap: "Issue dependency graph"
            - GroupView: "Issue groups and sprints"

REQ-HDR-005: Header MUST be memoized — re-render only when its props change.


3.3  FOOTER
-----------
REQ-FTR-001: Footer displays keyboard shortcuts applicable to the current view.

REQ-FTR-002: Shortcuts are rendered as "[key] label" chips.

REQ-FTR-003: Shortcuts MUST wrap into multiple rows if they exceed terminal
            width. Row computation MUST account for chip width + spacing.

REQ-FTR-004: Each view type defines its own shortcut set:

            Home (9 shortcuts):
              [Return] Select, [n] New, [a] Activate, [d] Defer, [r] Resolve,
              [s] Agents, [b] Blocking, [g] Groups, [q] Quit

            Detail - InputBox focused (10+ shortcuts):
              [Up/Dn] Scroll, [Shift+L/R] Prev/Next Issue, [Tab] Cycle Focus,
              [Return] Send/Open, [Ctrl+R] Resolve Issue, [Esc] Back,
              [Alt+H] Home, [d] Defer (disabled), [b] Block (disabled),
              [w] Rebase (disabled), [s] Show Agent pane (disabled)

            [CHANGED] Detail - Response selected (context-sensitive footer):
              [Ctrl+Shift+->] Create Thread / View Thread
                (label is "Create" if selected response has no replies,
                 "View" if it has replies)
              [Alt+Q] Quote
              Optional: [Ctrl+Alt+Q] Show quoted
                (shown only if selected response is quoting another response)
              Optional: [Ctrl+Alt+R] Resolve Thread
                (shown only if selected response has replies)

            [CHANGED] Detail - Thread mode (reduced shortcuts):
              [Up/Dn] Scroll, [Ctrl+Shift+L] Exit Thread,
              [Ctrl+Shift+R] Enter Sub-thread, [Tab] Cycle Focus,
              [Return] Send
              (Removed: [Ctrl+R], [Esc], [Alt+H] — these keys still function
               via useInput but are no longer shown in the thread footer.)

            NewIssue (2 shortcuts):
              [Return] Create, [Esc] Cancel

            AgentStatus (3 shortcuts):
              [Return] Focus Pane, [Esc] Back, [q] Quit

            BlockingMap (3 shortcuts):
              [Return] View Issue, [Esc] Back, [q] Quit

            GroupView (4 shortcuts):
              [Return] Select, [Esc] Back, [g] Toggle, [q] Quit

            IssuePicker (4 shortcuts):
              [Return] Toggle, [Ctrl+V] View, [Esc] Close, [q] Quit

REQ-FTR-005: Shortcuts with an "action" property are tabbable/focusable.
            The focused shortcut MUST render with inverse+bold styling.

REQ-FTR-006: Disabled shortcuts MUST render with dimmed color.

REQ-FTR-007: Pressing Return while a focusable footer shortcut has focus
            MUST dispatch that shortcut's action.

REQ-FTR-008: Footer height MUST be dynamically computed and communicated
            to the content area for correct space allocation.

PART 5: DETAIL VIEW
================================================================================

5.1  LAYOUT
-----------
REQ-DTL-001: DetailView has 3 vertical zones within the content area:
            1. Context header (issue metadata or thread parent)
            2. Conversation (scrollable response chain)
            3. Input box (text entry)

REQ-DTL-002: Conversation height = rows - header(3) - contextHeader - input(3)
            - footer. Must be at least 1 line.


5.2  ISSUE CONTEXT HEADER
--------------------------
REQ-DTL-010: When NOT in a thread, display the Issue Header (5 lines):
            [CHANGED] Line 1: LabeledSeparator with status-colored
              "Issue - I-<#>" text (replaces plain bold title)
            Line 2: "Status: <status>  Group: <group>"
              - Group field is tabbable (inverse+bold when focused)
            Line 3: "Blocked by: I-1, I-3  Blocks: I-7"
              - Both fields are tabbable (inverse+bold when focused)
            [CHANGED] Line 4: Hint text (dimmed):
              "([Tab] cycle through selectable components, [Enter] to choose)"
              (replaces old "(Tab to change Group, Blocked By, Blocks)" text)
            Line 5: Separator line

REQ-DTL-011: Pressing Return while a header field (Group/BlockedBy/Blocks)
            is focused MUST open the corresponding overlay/picker.

REQ-DTL-012: Group field opens GroupPicker overlay (inline).
            BlockedBy/Blocks fields navigate to IssuePicker view.


5.3  THREAD CONTEXT HEADER
---------------------------
REQ-DTL-020: When IN a thread, display:
            [CHANGED] Line 1: LabeledSeparator("[ Thread ]" or
              "[ Thread -- Resolved ]") — uses new LabeledSeparator component.
              "^R to resolve" hint text removed from this separator.
            Lines 2-N: Thread parent ResponseContainer (dimmed, not selectable)
            Final line: Replies separator ("[ Replies ]")

REQ-DTL-021: Thread parent MUST be rendered with isThreadParent=true styling
            (gray/dimmed colors).

REQ-DTL-022: The resolved label MUST show "Resolved" in white within gray
            separator text.


5.4  RESPONSE CHAIN (CONVERSATION)
-----------------------------------
REQ-RSP-001: The conversation area renders a vertical chain of
            ResponseContainer components within a fixed-height viewport.

REQ-RSP-002: The chain is derived from a linked list (Response.response
            pointers). A flattenToArray() utility converts this to an indexed
            array for rendering.

REQ-RSP-003: Viewport management with smart scrolling:
            - On cold start or resize: center the selected message
            - On cursor movement: stable anchor scrolling — only shift the
              viewport when the selection leaves the visible range
            - Track lastFirstVisible and lastHeight to detect reflows

REQ-RSP-004: The viewport MUST detect chain changes (thread enter/exit) and
            reset the scroll anchor.

REQ-RSP-005: A "[ New Replies ]" separator MUST appear before the first
            message with a timestamp after userLastViewedAt.

REQ-RSP-006: [CHANGED] Up/Down arrow keys scroll through a cyclical ring
            composed of the response list and the InputBox:
            - Down past last message: wraps to first message (index 0) and
              moves focus to InputBox with cursor at end of any existing text.
            - Up from first message (index 0): wraps to last message.
            The ring is: InputBox -> Response[0] -> ... -> Response[N] -> InputBox.
            (Replaces previous behavior where selection stopped at array bounds.)


5.5  RESPONSE CONTAINER
------------------------
REQ-RSP-010: Each message renders inside a bordered container with:
            - Top border with author name
            - Word-wrapped body with side borders
            - Bottom border with contextual action hint

REQ-RSP-011: Color coding by author:
            - User messages: cyan borders/author
            - Agent messages: green borders/author
            - Thread parents: gray (dimmed) borders/author
            - Selected messages: white/bold

REQ-RSP-012: Continuation messages (is_continuation=true) render with
            minimal borders — no author header, just the continued body
            text within the existing container visual.

REQ-RSP-013: Reply badge on top border (right-aligned):
            - "[N replies]" if has replies, no new ones
            - "[N replies (M new)]" in red+bold if has new unread replies
            - "[N replies (resolved checkmark)]" if thread is resolved

REQ-RSP-014: [CHANGED] Bottom border action hints:
            - No replies: plain bottom border, no action hint.
              ("reply to this" removed.)
            - Has replies, not focused: "view replies (N)" or "(N new)"
            - Has replies, focused: inverse+bold "view replies" label

REQ-RSP-015: ResponseContainer MUST expose a static/pure function to
            compute its rendered line count given body text and column width.
            This is needed by the viewport manager.

REQ-RSP-016: Word wrapping MUST respect terminal width minus border characters.


5.6  INPUT BOX
--------------
REQ-INP-001: Input box is 3 lines tall:
            Line 1: Separator (dashes)
            Line 2: Text input with prompt "Enter response: > "
            Line 3: (padding)

REQ-INP-002: Uses ink-text-input for text entry.

REQ-INP-003: The input receives keyboard focus when no header field,
            view-replies button, or footer shortcut is focused.
            [CHANGED] TextInput.enabled MUST track focus state:
              TextInput.enabled = (selectedComponent === InputBox).
              When a Response in the conversation list is selected (via
              Up/Down), TextInput becomes disabled (no cursor, no input).

REQ-INP-004: Pressing Return with non-empty text calls onSend(text) and
            clears the input.
            [CHANGED] After sending, InputBox retains focus — the cursor
            remains in the TextInput. Focus does NOT move to the conversation.

REQ-INP-005: Pressing Return with empty text and a focused header field
            opens the corresponding overlay.

REQ-INP-006: [NEW] On first load of a DetailView, InputBox MUST be the
            focused component with cursor visible. If Ink's TextInput supports
            a showCursor prop, enable it.

REQ-INP-007: [NEW] Quoting banner: when a quote is active (see REQ-QUO-*),
            a banner line appears between the separator and the TextInput:
              "Responding to <first 40 chars of quoted message>"
            The banner uses the same color as the original quoted message
            (cyan for user, green for agent), dimmed.


5.7  THREAD NAVIGATION
-----------------------
REQ-THR-001: Any response with replies can be entered as a thread.

REQ-THR-002: Enter thread: Ctrl+Shift+Right arrow (or Return on focused
            "view replies" button).
            - Pushes current {parent, savedSelectedIndex} onto threadStack
            - Displays the reply chain of the selected message
            - Moves cursor to last message in reply chain
            - Marks all replies as seen

REQ-THR-003: Exit thread: Ctrl+Shift+Left arrow, or Escape.
            - Pops threadStack
            - Restores savedSelectedIndex
            - Marks replies as seen
            - If threadStack is empty, exits the DetailView entirely

REQ-THR-004: Thread nesting is unlimited. Each enter pushes, each exit pops.

REQ-THR-005: Thread state changes MUST notify the parent App (for
            header subtitle and footer shortcut updates).

REQ-THR-006: Thread resolution:
            - Ctrl+R toggles thread_resolved_at on the thread parent
            - When in a thread: resolves current thread's parent
            - When on main chain: resolves selected message's thread
              (only if selected message has replies)

REQ-THR-007: Issue resolution (separate from thread resolution):
            - Ctrl+R when NOT on a message with replies toggles issue status
              between Resolved and Active


5.8  FOCUS CYCLING
------------------
REQ-FOC-001: Tab cycles focus through a ring of interactive elements:
            1. Input box (text entry)
            2. Group field (issue header, not in thread)
            3. BlockedBy field (issue header, not in thread)
            4. Blocks field (issue header, not in thread)
            5. "View replies" button (only if selected message has replies)
            6. Focusable footer shortcuts

REQ-FOC-002: Shift+Tab cycles in reverse.

REQ-FOC-003: The focus ring dynamically adjusts:
            - In thread mode: skip header fields (Group/BlockedBy/Blocks)
            - No replies on selected: skip "view replies"
            - Different footer shortcuts per view mode

REQ-FOC-004: Focus state MUST reset on thread enter/exit.

REQ-FOC-005: The currently focused element MUST have a clear visual
            indicator (inverse+bold or equivalent).


5.9  ADJACENT ISSUE NAVIGATION
-------------------------------
REQ-ADJ-001: Shift+Left: navigate to previous issue in the list.
REQ-ADJ-002: Shift+Right: navigate to next issue in the list.
REQ-ADJ-003: Uses replaceCurrentView (no stack push).
REQ-ADJ-004: Bounds-checked — no-op at first/last issue.


5.10 KEY BLOCKING
-----------------
REQ-BLK-001: Ctrl+R MUST NOT echo the 'r' character into the text input.
            The current implementation uses a suppressNextInput flag. The
            rewrite should handle this cleanly, possibly by checking for
            modifier keys before passing input to TextInput.


5.11 OVERLAYS
-------------
REQ-OVR-001: GroupPicker renders as a full-content-area overlay replacing the
            normal DetailView content.

REQ-OVR-002: GroupPicker features:
            - List of existing containers (groups)
            - First item: "+ Create new group" (switches to create mode)
            - Up/Down (j/k): navigate list
            - Return: select group or enter create mode
            - In create mode: TextInput for new group name
            - Escape: close overlay
            - Selection calls onGroupChange(containerId) or onGroupCreate(name)

PART 8: BLOCKING MAP VIEW
================================================================================

REQ-BLK-001: Display issue dependency graph as an ASCII tree.

REQ-BLK-002: Tree construction algorithm:
            1. Build adjacency map from dependencies
            2. Find root nodes (in graph but not blocked by anything)

            4. Append isolated issues (no dependencies) at bottom

REQ-BLK-003: Tree connector characters:
            - Branch: "--- " (with unicode box-drawing)
            - Last branch: "--- " (corner)
            - Continuation: "|   "
            - Leaf: "    "

REQ-BLK-004: Each node displays: I-<inum> <title> (<status>)
            Status is color-coded per standard mapping.

REQ-BLK-005: Sections:
            - "Dependency Graph" header
            - Tree of connected issues
            - "No dependencies:" header (if isolated issues exist)
            - Flat list of isolated issues

REQ-BLK-006: Up/Down (j/k): navigate cursor.
            Selected row: bold white with triangle cursor.

REQ-BLK-007: Return: navigate to selected issue's DetailView.

REQ-BLK-008: Global shortcuts available (Esc/q).

PART 12: DATA LAYER (UNCHANGED IN REWRITE)
================================================================================

The following modules are NOT being rewritten. They form the stable data layer
that the new TUI will consume. Documented here for interface reference.

12.1 TYPES (src/types.ts)
--------------------------
The rewrite MUST use these exact types:

Enums:
  IssueStatus { Active, Awaiting, Blocked, Deferred, Resolved }
  ResponseType { Question, Implementation, Clarification, Analysis, Fix, Other, None }
  AuthorType { User, Agent }

Interfaces:
  Issue { inum, title, description, status, created_at, resolved_at,
          issue_revision, agent_last_read_at, user_last_viewed_at }
  Message { author, type, body, timestamp, seen }
  Response { id, content: Message, responding_to, response, replying_to, reply,
             is_continuation, thread_resolved_at,
             [NEW] quoted_response_id: number | null }
  ResponseRow { id, inum, author, type, body, created_at, responding_to_id,

PART 20: SUGGESTED NEW ABSTRACTIONS
================================================================================

These are recommended new patterns for the rewrite:

20.1 CUSTOM HOOKS
-----------------
- useNavigation(): returns { navigateTo, goBack, replaceView, goHome,
  currentView, viewStack }. Backed by useReducer.

- useGlobalKeys(currentViewType, inputActive): handles global shortcuts
  (s/b/g/q/Esc/n). Uses useInput internally. When inputActive is true,
  single-char shortcuts (q, s, b, g, n) are suppressed so they don't
  conflict with text entry. Modifier shortcuts (Ctrl+*, Alt+*) still work.

- useViewport(messages, selectedIndex, height, columns): manages scroll
  position, returns { firstVisible, lastVisible, visibleMessages }.
  Encapsulates the current ResponseChain viewport logic.

- useThreadNavigation(rootResponse): manages threadStack, currentMessages,
  enterThread, exitThread, resolveThread. Returns current chain and state.

- useFocusRing(slots): manages circular focus cycling with dynamic slot count.
  Returns { focusedIndex, cycleFocus, resetFocus }.

- useSavedSelection(): persists selected message per inum across navigations.

- [NEW] useQuote(): manages quote state for the DetailView. Returns
  { quotedResponse, startQuote(response), clearQuote(), isQuoting }.
  Handles Alt+Q trigger, Escape cancel, and quote inclusion on send.

20.2 CONTEXTS
-------------
- NavigationContext: provides useNavigation() values to all descendants.

- AppStateContext: provides unreadCount, activeAgentCount, maxAgents,
  threadInfo for Header/Footer consumption without prop drilling.

20.3 COMPONENT COMPOSITION (View Configuration Pattern)
--------------------------------------------------------
- AppShell: wraps Header + children + Footer. Manages terminal dimensions.
  Owns footerOptions state and passes setFooterOptions callback down through
  ViewRouter to views. Header reads currentView from NavigationContext and
  derives its own subtitle — no view configuration needed for Header.
  Footer receives viewType (from NavigationContext) + footerOptions (from
  AppShell state) and calls getFooterShortcuts(viewType, options) to derive
  the correct shortcut set. footerOptions resets to {} on view change.

- ViewRouter: renders the correct view component based on currentView.type.
  A simple switch/map, extracted from App.render(). Passes setFooterOptions
  to views that need it (currently only DetailView).

- Views communicate footer configuration upward via the setFooterOptions
  callback. Most views don't call it (their shortcuts are static per
  ViewType). DetailView calls setFooterOptions({ inThread, responseSelected,
  hasReplies, isQuoting }) in a useEffect when its internal focus state
  changes. Each view owns its own useInput for key handling.
