# Plan: Question Review Orchestrator — v2

## Context

The v1 file-based question system didn't work reliably — `is_blocked()` failed, agents spawned despite unresolved blockers, and the XML tag format was brittle. The v2 redesign replaces question files entirely with a **SQLite database** and a **TUI** (terminal user interface) for user interaction.

The system has outgrown bash. **Rewriting as a Node.js/TypeScript app** gives type safety, native SQLite access via `better-sqlite3`, a TUI framework, proper error handling, and testability.

The result is a **repo-level ticketing system** — like Jira but offline, portable, and travels with the repo. Clone the repo, run setup, and all questions/responses/dependencies are immediately available.

## Design Decisions (consolidated)

- **SQLite database** — single source of truth for questions, responses, dependencies, and status
- **SQL dump tracked in git** — `questions.dump.sql` is the portable, diffable artifact. The `.db` file is gitignored (runtime artifact rebuilt from dump)
- **Auto-export** — daemon writes `questions.dump.sql` after every scan cycle that changes the DB
- **TUI** — primary user interface. No more editing question files directly
- **No question files** — question content lives in the database, not markdown files on disk
- **qnum as primary key** — stable, embedded in DB rows, never changes
- **`lastQuestionCreated`** — metadata row tracking the highest Q number assigned
- **Agents interact via CLI tool** — `qr-tool read <qnum>`, `qr-tool respond <qnum> "..."`, etc.
- **`.pending/` queue** — agents write pending actions as files; daemon is the sole DB writer
- **Groups** — optional grouping of related questions. A group is resolved when ALL members are resolved
- **`blocked_by`/`blocks`** — reference individual qnums. To block on a group, expand to all qnums in that group
- **`blocked_by` is purely DB-managed** — agents call `qr-tool block-by`, no file-level annotations
- **Lockfiles**: qnum-based (`Q7.lock`)
- **Worktrees**: qnum-based (`.claude/worktrees/Q7`, branch `worktree-Q7`)
- **Logs**: `[review] Spawned agent pane: Q7 — Migrate ServerDerivedFields`
- **Scan loop**: DB-driven (query entries with status=Active)
- **Pane titles**: removed (never worked in v1)
- **Statuses**: `Awaiting` (unblocked backlog) → `Active` (agent working) → `Resolved` / `Deferred` (user-deferred or blocked)
- **Auto-promotion**: daemon moves unblocked `Awaiting` questions to `Active` up to `MAX_AGENTS`
- **Enforce blocked invariant**: blocked questions cannot be `Awaiting` or `Active`; daemon enforces every cycle
- **Auto-unblock**: `Deferred` questions whose blockers are all `Resolved` auto-move to `Awaiting`
- **User-deferred vs system-blocked**: distinguished by presence of `blocked_by` entries. System-blocked auto-unblock; user-deferred stay until manually moved

## Database Schema

```sql
CREATE TABLE metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Seed: INSERT INTO metadata VALUES ('lastQuestionCreated', '0');

CREATE TABLE questions (
    qnum        INTEGER PRIMARY KEY,
    title       TEXT NOT NULL,          -- short label (e.g., "migrate_ServerDerivedFields")
    description TEXT NOT NULL DEFAULT '',  -- first ~200 chars, used in TUI list views
    "group"     TEXT,                   -- optional group name for related questions
    status      TEXT NOT NULL DEFAULT 'Awaiting'
                CHECK (status IN ('Awaiting', 'Active', 'Deferred', 'Resolved')),
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    resolved_at TEXT
);

CREATE TABLE responses (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    qnum       INTEGER NOT NULL REFERENCES questions(qnum),
    author     TEXT NOT NULL CHECK (author IN ('user', 'agent')),
    body       TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
CREATE INDEX idx_responses_qnum ON responses(qnum);

CREATE TABLE dependencies (
    blocker_qnum INTEGER NOT NULL REFERENCES questions(qnum),
    blocked_qnum INTEGER NOT NULL REFERENCES questions(qnum),
    PRIMARY KEY (blocker_qnum, blocked_qnum),
    CHECK (blocker_qnum != blocked_qnum)
);
CREATE INDEX idx_deps_blocked ON dependencies(blocked_qnum);
CREATE INDEX idx_deps_blocker ON dependencies(blocker_qnum);
```

**Why a proper join table for dependencies instead of JSON arrays:**
- SQL can answer "is Q7 blocked?" in one query: `SELECT 1 FROM dependencies d JOIN questions q ON d.blocker_qnum = q.qnum WHERE d.blocked_qnum = 7 AND q.status != 'Resolved' LIMIT 1`
- No parsing, no hash lookups, no stale references — foreign keys enforce integrity
- Group expansion is a query: `INSERT INTO dependencies SELECT q.qnum, ? FROM questions q WHERE q."group" = ?`

## Scan Cycle Pipeline

```
1. Process pending queue     (.pending/ → DB writes)
2. Enforce blocked           (Awaiting/Active with unresolved blockers → Deferred, kill agent)
3. Auto-unblock              (Deferred with all blockers Resolved → Awaiting)
4. Auto-promote              (Awaiting with pending agent response → Active, up to MAX_AGENTS)
5. Spawn/re-prompt agents    (Active questions: spawn new agents, re-prompt on new responses)
6. Cleanup                   (stale lockfiles, finished panes)
7. Export dump if changed    (questions.dump.sql written only when DB was modified this cycle)
```

**Key invariant**: Blocked questions cannot be `Awaiting` or `Active`. The daemon enforces this every cycle.

## Git Tracking: SQL Dump

The SQLite `.db` file is a runtime artifact — gitignored, rebuilt from the dump on setup.

**What's tracked in git:** `Questions/questions.dump.sql`
- Text-based SQL dump (INSERT statements) — diffable, human-readable
- Generated by: `sqlite3 questions.db .dump > questions.dump.sql`
- Imported by: `sqlite3 questions.db < questions.dump.sql` (in `setup.sh`)

**Auto-export flow:**
1. Daemon runs a scan cycle
2. If any DB writes occurred (pending queue processed, status changed, response added), set `dirty = true`
3. At end of cycle, if `dirty`: export dump via `sqlite3 questions.db .dump > questions.dump.sql`
4. The dump is ready to be committed whenever the user commits

**On clone / new machine:**
```bash
git clone <repo>
cd repo
./setup.sh    # imports questions.dump.sql → questions.db, installs deps, builds
```

**`.gitignore`** (in `Questions/`):
```
questions.db
questions.db-journal
questions.db-wal
```

**Merge strategy**: `.gitattributes` with `Questions/questions.dump.sql merge=ours` — local state wins on merge conflicts. Alternatively, use a custom merge driver that merges SQL INSERTs by qnum (future enhancement).

### `pipeline.ts` — Scan Pipeline

```typescript
function enforceBlocked(db: DB, agents: AgentManager): void;
// SELECT qnum FROM questions WHERE status IN ('Awaiting','Active')
// For each: if isBlocked(qnum) → updateStatus('Deferred'), kill agent if Active

function autoUnblock(db: DB): void;
// SELECT DISTINCT d.blocked_qnum FROM dependencies d
//   JOIN questions q ON d.blocked_qnum = q.qnum
//   WHERE q.status = 'Deferred'
// For each: if NOT isBlocked(qnum) → updateStatus('Awaiting')
// (Only auto-unblock questions that HAVE dependency entries;
//  user-deferred questions without dependencies stay in Deferred)

function promoteAwaiting(db: DB, maxAgents: number): void;
// SELECT qnum FROM questions WHERE status = 'Awaiting' ORDER BY qnum
// Move to Active up to (maxAgents - currentActiveCount)

function runPipeline(db: DB, agents: AgentManager, maxAgents: number): void;
// enforceBlocked → autoUnblock → promoteAwaiting
```

Used by agents and users. Agents call it from their tmux pane.

```

Commands:
  read <qnum>                        Show question + full response history
  respond <qnum> <body>              Submit a response (writes to .pending/)
  create <title> <description>       Create a new question (writes to .pending/)

  block-by <blocked> <blocker>       Mark qnum as blocked by another (writes to .pending/)
```

Read-only commands query the DB directly. Write commands create files in `.pending/`.

## Edge Cases

- **Legacy migration**: `migrate-v1.ts` handles all existing `Q####_*.md` files. Post-migration, files can be archived
- **Concurrent agent writes**: Agents write to `.pending/` queue (one file per action). Daemon processes sequentially. No DB contention
- **Auto-export idempotent**: If no DB changes occurred in a scan cycle, no dump is written
- **First run on clone**: `setup.sh` imports `questions.dump.sql`. If no dump exists, creates empty DB from `schema.sql` + `seed.sql`
- **User-deferred vs system-blocked**: Both have status `Deferred`. Distinguished by presence of rows in `dependencies` table. `autoUnblock` only moves questions that HAVE dependency entries
- **Group partially resolved**: `isGroupResolved()` checks all members via SQL aggregate
- **Agent responds to resolved question**: Pending action is processed but daemon skips questions with status `Resolved`
- **Stale lockfile**: Daemon checks if tmux pane still exists; removes lockfile if pane is dead
- **DB corruption**: Rebuild from `questions.dump.sql` (the git-tracked source of truth)
- **Large response bodies**: SQLite handles TEXT columns up to 1GB. No practical limit
- **Node.js not installed**: `setup.sh` checks for `node` and `npm`, exits with error

## Decision Log

| # | Decision | Answer |
|---|----------|--------|
| D1 | Lockfile naming | Qnum-based (`Q7.lock`) |
| D2 | Worktree naming | Qnum-based (`.claude/worktrees/Q7`) |
| D3 | Log format | `Q<num> — <title>` |
| D4 | Scan loop | DB-driven (query Active entries) |
| D5 | Agent interaction | Agents use `qr-tool` CLI; writes go to `.pending/` queue |
| D6 | Legacy filenames | One-time migration to DB; files archived |
| D7 | Counter name | `lastQuestionCreated` (metadata table) |
| D8 | Multi-question entries | One row per question, with `group` column. Group resolved when all members resolved |
| D9 | Pane titles | Remove all pane title infrastructure |
| D10 | Git tracking | SQL dump tracked in git; `.db` gitignored |
| D11 | `blocked_by` management | Purely DB-managed via `dependencies` table |
| D12 | Question creation | Through `qr-tool create` or TUI |
| D13 | Implementation language | TypeScript (Node.js) |
| D14 | Storage backend | SQLite (`better-sqlite3`) — scales to 800+ questions |
| D15 | User interface | TUI (Ink/React) — replaces direct file editing |
| D16 | Portability | SQL dump in git, `.db` rebuilt on clone via `setup.sh` |
| D17 | Auto-export | Daemon writes `questions.dump.sql` after scan cycles with DB changes |
