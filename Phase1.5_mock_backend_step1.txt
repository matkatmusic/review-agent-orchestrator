Phase 1.5 Step 1: Wire Missing App View Render Cases
=====================================================

Context
-------

app.tsx's render switch only handles ViewType.Home and ViewType.Detail. The four
remaining views (NewIssue, AgentStatus, BlockingMap, GroupView) have fully implemented
components but no render cases, so navigating to them shows nothing. 5 app.test.tsx
tests fail because of this. This must be fixed before any Phase 1.5 callback wiring
(onSend, status actions, group/block changes) since those callbacks are untestable on
views that don't render.

Secondary issue: AgentStatus, BlockingMap, and GroupView do not call handleGlobalKey,
so even once rendered, cross-view navigation shortcuts (s, b, g, n, q, Esc) won't work
from those views. The tests explicitly exercise multi-level navigation
(Home -> AgentStatus -> BlockingMap -> back -> back) which requires global keys.

Part A: Add 4 render cases to app.tsx
-------------------------------------

File: src/tui/app.tsx

Add cases to the switch in render() after the ViewType.Detail case. All four follow
the same pattern as the existing Home and Detail cases — instantiate the component
with navigation callbacks.


ViewType.NewIssue:

  case ViewType.NewIssue:
      content = (
          <NewIssue
              onCreated={(data) => {
                  // Step 1: just navigate back.
                  // Actual issue creation is Phase 1.5 Step 4.
                  this.goBackToPreviousView();
              }}
              onCancel={() => this.goBackToPreviousView()}
              onNavigate={(view) => this.navigateToView(view)}
              onBack={() => this.goBackToPreviousView()}
              onQuit={() => this.props.onExit?.()}
          />
      );
      break;

NewIssue handles Escape internally (calls onCancel). It's a text-entry view so no
global key wiring needed (same reasoning as Detail view).


ViewType.AgentStatus:

  case ViewType.AgentStatus:
      content = (
          <AgentStatus
              onNavigate={(view) => this.navigateToView(view)}
              onBack={() => this.goBackToPreviousView()}
              onQuit={() => this.props.onExit?.()}
          />
      );
      break;


ViewType.BlockingMap:

  case ViewType.BlockingMap:
      content = (
          <BlockingMap
              onNavigate={(view) => this.navigateToView(view)}
              onBack={() => this.goBackToPreviousView()}
              onQuit={() => this.props.onExit?.()}
          />
      );
      break;


ViewType.GroupView:

  case ViewType.GroupView:
      content = (
          <GroupView
              onBack={() => this.goBackToPreviousView()}
              onSelectIssue={(inum) => this.navigateToView({ type: ViewType.Detail, inum })}
              onNavigate={(view) => this.navigateToView(view)}
              onQuit={() => this.props.onExit?.()}
              groupMode={this.groupMode}
              onGroupModeChange={(mode) => this.setGroupMode(mode)}
          />
      );
      break;


Part B: Add handleGlobalKey to three view components
-----------------------------------------------------

The tests exercise cross-view navigation (e.g., 's' from Home -> AgentStatus, then
'b' from AgentStatus -> BlockingMap, then Esc -> back, Esc -> back). This requires
each non-text-entry view to call handleGlobalKey as a fallback in its useInput.


AgentStatus (src/tui/agent-status.tsx):

  - Import handleGlobalKey from ./global-keys.js and ViewType from ./views.js
  - At the end of useInput, after the existing j/k/Enter handlers, add:

      handleGlobalKey(input, key, ViewType.AgentStatus,
          { onBack, onQuit, onNavigate });


BlockingMap (src/tui/blocking-map.tsx):

  - Import handleGlobalKey from ./global-keys.js (ViewType already imported)
  - At the end of useInput, after the existing j/k/Enter handlers, add:

      handleGlobalKey(input, key, ViewType.BlockingMap,
          { onBack, onQuit, onNavigate });

  - Remove the existing manual key.escape -> onBack?.() handling since
    handleGlobalKey covers it.


GroupView (src/tui/group-view.tsx):

  - Import handleGlobalKey from ./global-keys.js and ViewType from ./views.js
  - In useInput, for list mode: replace the existing manual
    key.escape -> onBack?.() with a fallback to:

      handleGlobalKey(input, key, ViewType.GroupView,
          { onBack, onQuit, onNavigate });

  - In issues mode: keep key.escape going back to list mode (this is internal
    state, not view-level back). Add handleGlobalKey as fallback for other keys
    like q.


NewIssue: No changes. Text-entry view — single-char globals would interfere
with typing. Same reasoning as Detail view.
