## PRIOR DEBATE RESOLUTIONS (D1-D15)

**D1. Scan cycle blocking** — RESOLVED: Single-flight guard (boolean flag, skip if previous running) + debounced dump export (5s) + synchronous better-sqlite3 (<1ms). Git/tmux calls stay sync for now; migrate to async if TUI sluggish.

**D2. aidi CLI phase ordering** — RESOLVED: Moved from Phase 13 to Phase 0.5. Built alongside CRUD layer.

**D3. Resolve flow git mechanics** — RESOLVED: V1's stable approach was rebase-then-format-patch but v1 amended to one commit. V2.1 agents have multiple commits. Phase 11 evaluates 4 strategies empirically: (A) rebase+diff, (B) rebase+format-patch, (C) rebase+pre-squash+format-patch, (D) rebase+format-patch+3way. Agent handles its own rebase conflicts (has context), escalates to user only if it fails.

**D4. Signal reliability** — RESOLVED: Two-part fix. (1) cancel_and_send ensures signal delivered to idle agent. (2) Agent runs `aidi ack <inum>` after reading issue.md, setting `agent_last_read_at`. Orchestrator retries if ack not received within N seconds.

**D5. issue.md race condition** — RESOLVED: cancel_and_send (C-c + poll until idle) before rewriting file. Agent guaranteed stopped during rewrite. No atomic writes needed.

**D6. Dependency graph edge cases** — RESOLVED: (1) Cycle detection via recursive CTE at addBlock time. (2) Auto re-block: reopened blocker re-blocks dependents if dependency row exists. (3) Deferred blocker treated as unresolved.

**D7. Phase 5 bare agent spawn** — RESOLVED: Accepted as tmux smoke test. Agent behavioral testing deferred to Phase 10.

**D8. Mock data double work** — RESOLVED: Not an issue — better-sqlite3 is synchronous, no sync-to-async transition.

**D9. Git index.lock contention** — UNRESOLVED. Multiple agents in worktrees share underlying .git directory. Orchestrator running git commands every 1-2s risks collisions.

**D10. Branch accumulation** — RESOLVED: TODOs added for auto-delete on merge detection + manual delete command.

**D11. aidi identity** — RESOLVED: Every aidi command takes inum as explicit argument. issue.md has structured header (inum, title, status, description, blocked_by, blocks).

**D12. Teardown commit protocol** — RESOLVED: Agent handles ALL git ops. Orchestrator signals via cancel_and_send ("commit your work and stop"), polls for clean tree (git status --porcelain). CLAUDE.md instructs routine commits + commit-on-signal.

**D13. Single-flight locking** — RESOLVED: (1) Single-flight guard from D1. (2) BEGIN IMMEDIATE transactions for status transitions.

**D14. SQLite busy_timeout** — RESOLVED: PRAGMA busy_timeout=5000 on both orchestrator and aidi connections.

**D15. Deferred permissions** — RESOLVED: V1 battle-tested baseline (Read, Edit, Write, Glob, Grep, git, ls, find, mv, cp, grep, cat, echo, tmux) + aidi + defaultMode: acceptEdits. Empirical verification in Phase 6.

## UPDATED DESIGN DOCUMENT (v2.1.md) — KEY SECTIONS

### Architecture
- **Single process** — TUI + orchestrator scan loop in one Ink app. `scanCycle` runs on a `useEffect` timer (1-2s interval).
- **Session recovery on startup** — reconcile existing tmux panes against `agent_sessions` DB table, re-adopt alive panes, clean up dead ones.
- **SQLite WAL mode** with `PRAGMA busy_timeout=5000` on both orchestrator and aidi connections.
- **Single-flight scan guard** — skip cycle if previous still running.
- **BEGIN IMMEDIATE transactions** — each status transition wrapped in explicit SQLite transaction.
- **Debounced dump export** — after 5s of no further DB changes.

### Issue Statuses (5 states)
- **Awaiting** — candidate for work. No agent, no worktree.
- **Active** — user activated via TUI. Agent spawned, worktree created. Capped by MAX_AGENTS.
- **Blocked** — depends on another issue. Auto-transitions to Awaiting when blockers resolve.
- **Deferred** — user postponed. Manual move only.
- **Resolved** — done. Pane killed, worktree applied. Can reopen (→ Awaiting, may re-block dependents).

### Status Transitions
- Awaiting → Active: user triggers, disabled at MAX_AGENTS.
- Active → Blocked: orchestrator signals agent to commit and stop, polls for clean tree, removes worktree, kills pane.
- Active → Deferred: same teardown as Blocked.
- Active → Resolved: agent rebases, apply strategy selected in Phase 11, agent deleted.
- Blocked → Awaiting: automatic when all blockers resolve.
- **Auto re-block**: reopened blocker re-blocks dependents if dependency row exists.
- **Cycle detection**: recursive CTE at addBlock time rejects direct and transitive cycles.

### Agent Communication (two channels)
- **Ticket channel**: user writes in TUI → DB → cancel_and_send (C-c + poll until idle) → rewrite issue.md → send signal → agent reads → agent runs `aidi ack <inum>`. Retry if ack not received within N seconds.
- **Direct channel**: user types in tmux pane. NOT logged unless "log that".
- Strict separation. Interrupt-before-rewrite eliminates race conditions.

### Agent Context
- **CLAUDE.md**: behavioral instructions + routine commit instructions (after each logical unit) + commit-on-signal instructions (teardown signal → commit immediately and stop).
- **issue.md**: structured header (inum, title, status, description, blocked_by, blocks) + full conversation history. Full rewrite on every user response.
- **aidi CLI**: respond, read, block (with cycle detection), create, status, ack. Own DB connection with WAL + busy_timeout=5000.

### Worktree Lifecycle
- On activate: create worktree (or resume from existing branch).
- On block/defer: signal agent to commit and stop via cancel_and_send, poll for clean tree (git status --porcelain). Agent handles ALL git ops — orchestrator never touches worktree git state. Clean → remove worktree + kill pane. Dirty → wait. Force-kill escape hatch.
- On resolve: agent rebases onto active HEAD (resolves own conflicts, escalates if fails). Apply using empirically selected strategy. Remove worktree, delete agent.
- Rebase: manual TUI command only.

### Agent Permissions
- Per-worktree settings.json: Read(*), Edit(*), Write(*), Glob(*), Grep(*), Bash(git/ls/find/mv/cp/grep/cat/echo/tmux *), Bash(aidi *), Bash(agent-issue-db-interface *). defaultMode: acceptEdits.

### Database Schema
```sql
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
```
Tables: metadata, issues (5 statuses + agent_last_read_at), responses, dependencies, containers (group/sprint + parent_id), issue_containers, agent_sessions.

## DISCUSSION CONTEXT (key tradeoffs)

- **Single process**: ~50 fewer lines, session recovery mitigates crash risk. Scan cycle blocking mitigated by single-flight guard + sync better-sqlite3.
- **cancel_and_send**: V1's proven pattern — C-c, poll until idle, then send. Eliminates race conditions.
- **aidi ack**: Delivery guarantee + bidirectional unread indicators.
- **Agent handles all git ops**: Agent has context, orchestrator doesn't. Signals and verifies, never directly commits.
- **4 resolve strategies**: Empirical evaluation because v2.1 differs from v1 (multiple commits vs single amended commit).
- **Auto re-block**: Dependency rows are source of truth. Reopened blocker re-blocks dependents automatically.
