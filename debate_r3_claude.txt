R2-8. Spawned agents not receiving initial prompt (daemon.ts, agents.ts)
STATUS: PARTIAL

The fix is structurally correct but has a timing concern:

(a) The prompt content is correct. buildInitialPrompt (agents.ts:56-79)
    includes qnum, title, description, main tree path, and qr-tool path.

(b) sendInitialPrompt is called immediately after spawnAgent at daemon.ts:168
    and 178. spawnAgent uses sendKeys to send the claude CLI command to the
    tmux pane. sendInitialPrompt then sends the actual question prompt via
    another sendKeys call.

The timing concern: spawnAgent sends the claude CLI launch command to the
pane. sendInitialPrompt is called immediately after, which sends the question
text. There is NO delay between these calls. The claude CLI typically takes
several seconds to start and begin reading stdin. The sendKeys sends text to
the tmux pane buffer, which is then read by whatever process is running in
that pane.

However, tmux's send-keys operates on the terminal buffer. If the shell has
already started but the claude CLI has not yet taken over stdin, the prompt
text would be interpreted as a shell command, which would fail harmlessly but
the prompt would be lost. The shell command sent by spawnAgent includes the
claude CLI invocation, which is followed by Enter. The subsequent sendKeys
for the initial prompt also sends Enter. If the shell has not yet executed the
first command, the second sendKeys text could interleave with the command line.

In practice, tmux buffers keystrokes and delivers them sequentially to the
pane's PTY. The first sendKeys sends the launch command + Enter. The second
sendKeys sends the prompt + Enter. Since launch-agent.sh starts the claude
CLI, the prompt text arrives in the stdin buffer and will be read by the CLI
when it starts. This is because tmux's PTY buffer holds the text.

The tests at agents.test.ts:387-401 verify that sendInitialPrompt sends content
to the pane, with a 300ms delay for pane readiness. The daemon does not have
this delay.

Verdict: The fix addresses the core issue (prompt is now sent) and will work
in most cases because tmux buffers keystrokes. But there is no explicit
readiness check or delay, which could cause issues in edge cases where the PTY
buffer is consumed before claude CLI starts reading.

2. ID: 2
   TYPE: NEW
   SEVERITY: MEDIUM
   Location: pending.ts:16 (writePending)
   Issue: writePending uses non-atomic write (writeFileSync without temp+rename)

   writePending writes directly to the final filepath via writeFileSync.
   If the daemon reads the .pending/ directory between the file creation and
   the write completing (or if the process crashes mid-write), a partial JSON
   file could be read by processPendingQueue.

   processPendingQueue does handle invalid JSON gracefully (pending.ts:38-42,
   it catches parse errors and deletes the file), so this is not a crash risk.
   However, a partial write means the pending action is lost permanently.

   The risk is low because writeFileSync is synchronous and writes small JSON
   payloads (< 1KB) that are likely to be atomic at the filesystem level for
   most operating systems and file sizes. But it's not guaranteed.

   Fix: Write to a .tmp suffix first, then rename:
   ```typescript
   const tmppath = filepath + '.tmp';
   writeFileSync(tmppath, JSON.stringify(action), 'utf-8');
   renameSync(tmppath, filepath);
   ```
   renameSync is atomic on POSIX filesystems when source and destination are
   on the same filesystem (which they are, since they're in the same directory).

3. ID: 3
   TYPE: NEW
   SEVERITY: LOW
   Location: daemon.ts:90 (scanCycle)
   Issue: No protection against concurrent daemon instances

   daemon.sh starts a loop that runs "node dist/daemon.js" repeatedly. If a
   second daemon.sh is started (e.g., by a user running it in another terminal),
   two instances would process the same .pending/ files, potentially causing:
   - Duplicate response processing (addResponse called twice for same pending)
   - Race conditions on lockfile creation

   The daemon.sh script does not check for existing instances (no PID file,
   no flock). WAL mode with busy timeout mitigates database-level races, but
   pending file processing has no deduplication -- both daemons would read
   the same file before either deletes it.

   In practice, the processPendingQueue function deletes files after processing
   (pending.ts:48), so the race window is small. And the safeUnlink at line 64
   handles the case where the file was already deleted by another process. But
   addResponse would be called twice, creating duplicate responses.

   Fix: Add a PID file or flock-based guard in daemon.sh:
   ```bash
   LOCKFILE="$PROJECT_ROOT/.daemon.lock"
   exec 9>"$LOCKFILE"
   flock -n 9 || { echo "[review] Daemon already running"; exit 1; }
   echo $$ > "$LOCKFILE"
   ```

   Issue: Detail view writes directly to DB, bypassing pending queue

   When a user submits a response in the detail view (detail.tsx:90), it calls
   addResponse(db, qnum, 'user', trimmed) directly on the database, bypassing
   the .pending/ queue mechanism.

   This means:
   - The response is written immediately (good for UX responsiveness).
   - But it does not go through the daemon's processPendingQueue path, so if
     the daemon is the only writer expected, this creates a dual-write pattern.
   - Since both the TUI and daemon share the same SQLite database with WAL mode,
     this is functionally correct -- SQLite handles concurrent writers with WAL.
   - The dirty flag IS set by db.run(), so the daemon's dump export will pick up
     changes on its next cycle.

   The main concern is consistency: the qr-tool CLI writes via .pending/ (going
   through the daemon), but the TUI writes directly. If the TUI crashes
   mid-transaction, the response is not persisted (but the user can re-submit).

   This is a design choice, not a bug. The TUI's direct write provides better
   UX (instant feedback). But it should be documented as intentional.

   Fix: No code change needed. Add a comment in detail.tsx explaining the
   design decision:
   ```typescript
   // Direct DB write (not through .pending/) for immediate user feedback.
   // The daemon's exportDump will capture this change on its next cycle.
   addResponse(db, qnum, 'user', trimmed);
   ```

    ```

    ```

    If the process receives SIGINT (Ctrl+C) or throws an uncaught exception,
    db.close() may never be called. With WAL mode and better-sqlite3's native
    binding, the database handles this gracefully (WAL is designed for crash

    the DB needs to replay the WAL, and the -wal and -shm files persist on
    disk.

    Fix: Add a process signal handler:
    ```typescript

    ```

Rating: 4 out of 5

The codebase is in good shape after two rounds of fixes. The core architecture
is well-designed with clear separation of concerns (DB layer, questions module,
responses module, dependencies module, pipeline, agents, daemon, TUI). The
pending queue pattern provides a clean boundary between CLI tools and the daemon.

Positive patterns observed:

1. STRONG error handling philosophy. The pending queue's permanent vs transient
   error classification (pending.ts:76-82) is thoughtful. readLockfile and
   listLockfiles gracefully handle corrupted files. The daemon's main loop
   wraps everything in try/catch.

2. GOOD test coverage. ~296 tests cover the core modules thoroughly. The
   ink-testing-library integration tests for TUI components are particularly
   well done, testing user interactions end-to-end.

3. CLEAN transaction boundaries. The addResponse and createQuestion transactions
   are correct. better-sqlite3's synchronous nature eliminates many classes of
   concurrency bugs.

4. PROPER use of SQLite features. WAL mode, foreign keys, CHECK constraints,
   recursive CTEs for cycle detection, and index definitions are all correct.

5. GOOD resilience patterns. The daemon continues after spawn failures. The
   pending queue retries transient errors. Stale lockfile cleanup runs every
   cycle.

Remaining concerns (minor):

1. No concurrent daemon protection (finding #3). Two daemon instances could
   cause duplicate processing. Low probability in practice.

2. Pending file and lockfile writes are not atomic (findings #2, #4). The
   consequence of corruption is handled gracefully, but data loss is possible.

3. scanCycle does not isolate step failures (finding #5). A failure in pipeline
   processing could skip agent management and dump export.

4. Some test coverage gaps: getUnreadQnums lacks unit tests, orphan cleanup
   lacks integration tests, and user-respond CLI path lacks tests.

5. The sendInitialPrompt timing (R2-8) relies on tmux PTY buffering rather
   than an explicit readiness check. This works in practice but is fragile.

None of the remaining issues are critical. The system is production-ready for
its intended use case (single-developer question review orchestration). The
SQLite-backed architecture with WAL mode is a good fit for this workload.
