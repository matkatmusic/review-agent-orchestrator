MISSING (features v1 has that v2 does not yet have)
========================================================================

Compared against v1/src/ inventory. Items are grouped by subsystem.

--- Agent Lifecycle ---

v1 has:
  - spawnAgent() — creates worktree, writes lockfile, spawns claude CLI
  - repromptAgent() — sends "NEW USER RESPONSE" via tmux send-keys
  - isAgentRunning() — checks lockfile + pane alive
  - cleanupStaleLocks() — removes lockfiles for dead panes
  - buildInitialPrompt() — constructs first message to agent
  - buildClaudeCommand() — builds shell cmd with --worktree, --add-dir
  - Lockfile tracking: {paneId, qnum, headCommit}

v2 has: agent_sessions DB table (schema + CRUD), but NO agent spawn,
        reprompt, teardown, or lifecycle code.


--- Daemon / Scan Cycle ---

v1 has:
  - scanCycle() — full orchestration loop:
    1. Process pending queue
    2. Run pipeline (enforce blocked -> auto-unblock -> promote)
    3. Kill agents for enforced/orphaned questions
    4. Spawn/re-prompt agents for Active questions
    5. Detect new commits, send rebase signals
    6. Cleanup stale lockfiles
    7. Export dump if dirty
  - detectNewCommits() — compares HEAD against lockfiles
  - ensureAutocompact() — sets autoCompactEnabled in ~/.claude.json
  - Main loop with configurable interval

v2 has: NOTHING. No scan cycle, no pipeline, no polling loop.


--- Status Pipeline ---

v1 has:
  - enforceBlocked() — moves Awaiting/Active with unresolved blockers
  - autoUnblock() — moves Deferred->Awaiting when all blockers resolve
  - promoteAwaiting() — moves Awaiting->Active up to maxAgents
  - runPipeline() — orchestrates all three in sequence

v2 has: NOTHING. Status transitions are manual only.


--- Pending Action Queue ---

v1 has:
  - writePending() — writes JSON action to .pending/ (atomic rename)
  - processPendingQueue() — reads all .json, applies to DB, deletes
  - applyAction() — dispatcher for 5 action types

v2 does NOT need this: agents write directly to DB via aidi (WAL mode).
This is a deliberate architectural change, not a gap.


--- Tmux Pane Interaction ---

v1 has: sendKeys(), capturePaneTail(), isPaneAlive(), killPane()
v2 has: createSession, splitWindow, listPanes, killSession
v2 missing: sendKeys, capturePaneTail/getPaneOutput, killPane,
            isPaneAlive (individual pane check)


--- Worktree Management ---

v1 has: Claude CLI's --worktree flag handles creation; lockfiles track state
v2 has: NOTHING. No worktree create/remove/apply code.


--- TUI <-> DB Wiring ---

v1 has: TUI queries DB directly (questions, responses, dependencies)
        Auto-refresh via 3s polling
v2 has: ALL mock data. Zero DB queries from TUI.
        onSend, onGroupChange, etc. are stubs.


--- Agent Prompt Template ---

v1 has: prompts/review-agent.md — step-by-step agent instructions
        (RESOLVE | IMPLEMENT | DEFER | RESPOND classification)
v2 has: NOTHING. config.json references agentPrompt but file missing.


--- Setup & Reset ---

v1 has: setup.ts — creates folder structure, copies templates, merges
        VS Code tasks.json, initializes DB, imports dump
        reset.ts — kills tmux, removes worktrees, clears lockfiles
v2 has: NOTHING. No install.sh, no setup, no reset.


--- SQL Dump Portability ---

v1 has: exportDump() called after every dirty mutation
        importDump() on setup if dump exists
v2 has: exportDump/importDump in database.ts, but never called.
        No debounced export, no git tracking of dump.


--- Rebase Handling ---

v1 has: detectNewCommits() — compares HEAD against agent headCommit,
        sends "git rebase <commit>" via tmux send-keys
v2 has: Planned "rebase-worktree" TUI command, but NOT implemented.


--- Session Recovery ---

v1 has: Daemon detects stale lockfiles and cleans up on startup
v2 design: Planned reconciliation of agent_sessions against live tmux
           panes on startup. NOT implemented.
