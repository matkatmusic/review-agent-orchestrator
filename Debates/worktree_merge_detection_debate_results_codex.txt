DEBATE TOPIC: Methods to determine if a git worktree branch has been merged into another branch

CONTEXT:
You are working with git worktrees. The scenario is:
- Branch 'B' exists with commits.
- A worktree 'WT1' was created from a commit on 'B', with its own branch (let's call it 'wt1-branch').
- Commits were added to wt1-branch in the worktree.
- Meanwhile, additional commits were added to 'B' directly.
- At some point, wt1-branch may or may not have been merged back into 'B'.
- Other worktrees (e.g. WT2) also exist with their own branches.

The ASCII state after WT1 is merged looks like:

          W1 ---- W2
         /          \
C0 ---- C1 --------- M1   <-- B (HEAD)
          \
           X1 -- X2 -- X3  <-- wt2-branch (unmerged)

QUESTION:
What is the best method (or combination of methods) to programmatically determine whether 'wt1-branch' has been merged into 'B'?

1. Git plumbing commands vs porcelain commands
2. Reliability across edge cases (fast-forward merges, squash merges, rebase-and-merge, octopus merges)
3. Performance implications for repos with many branches
4. How to handle the case where the merge was done via squash (commits rewritten, no merge commit)
5. Scripting/automation friendliness (exit codes, parseable output)
6. Whether the worktree itself still exists or has been removed

Provide your recommended approach with specific git commands, explain the trade-offs, and note any pitfalls. If multiple methods are needed to cover all merge strategies, explain when to use each.

- ARGUMENTS: Why this method is best, with specific git commands
- EDGE CASES: What it misses and how to handle those
- SCRIPT EXAMPLE: A short bash snippet demonstrating your approach

- POSITION: Use `git merge-base --is-ancestor wt1-branch B` as the primary test, and only if that fails, run a secondary patch-equivalence check with `git rev-list --right-only --cherry-pick --no-merges B...wt1-branch`. Treat squash merges as a separate best-effort case, because Git cannot reliably prove them from commit graph alone.

- ARGUMENTS: `git merge-base --is-ancestor` is the best primary method because it is plumbing, fast, script-friendly, and answers the precise question “is the tip of `wt1-branch` reachable from `B`?”. It returns exit code `0` if merged, `1` if not, and `>1` on error, so there is no fragile text parsing. It correctly handles normal merge commits, fast-forward merges, and octopus merges, because all of those preserve ancestry. It also does not care whether the worktree still exists; worktrees are irrelevant once you have the branch ref or commit SHA.

  `git branch --merged B` or `git branch --contains <commit>` can show similar information, but they are porcelain-oriented and better for humans than scripts. For automation, `merge-base --is-ancestor` is the direct pairwise test and avoids enumerating every branch in large repos.

  If ancestry fails, the next useful test is patch equivalence: `git rev-list --right-only --cherry-pick --no-merges B...wt1-branch`. If that prints nothing, every non-merge commit unique to `wt1-branch` has an equivalent patch somewhere on `B`. That covers rebase-and-merge and cherry-pick style integrations where commit IDs changed but the content landed. It is more expensive than the ancestry test, so it should be the fallback, not the first check.

- EDGE CASES: `merge-base --is-ancestor` misses any workflow that rewrites commits, including rebase-and-merge and squash merges. The `--cherry-pick` fallback helps for rebased or cherry-picked commits when the branch’s commits still map roughly one-to-one by patch. It does not reliably detect many-to-one squash merges, because a squash commit often combines several topic commits into one new patch, so there is no per-commit patch match.

  For squash merges, there is no perfect generic answer after the fact unless you recorded metadata at merge time. Best-effort heuristics exist, but each has failure modes:
  - Compare the cumulative diff from the fork point with a single commit on `B` using `git patch-id --stable`. This can catch a pure squash commit, but fails if the squash included unrelated changes.
  - Simulate whether merging `wt1-branch` into `B` would be a no-op. This can still fail if `B` continued evolving after the squash.
  - Search PR metadata, commit trailers, or notes if your workflow records “squashed from `<branch>`” or source SHAs. This is the only robust way to make squash detection dependable.

  If the worktree was removed, nothing changes as long as the branch ref still exists. If the branch ref was deleted too, you need the old tip SHA. If both the ref and its reflog are gone, Git cannot prove anything from the repository alone.

- SCRIPT EXAMPLE:
```bash
#!/usr/bin/env bash
set -euo pipefail

target=${1:?target branch}
topic=${2:?topic branch}

# 1. Exact history-preserving merge?
if git merge-base --is-ancestor "$topic" "$target"; then
  echo "merged: ancestry"
  exit 0
fi

# 2. Rewritten but patch-equivalent (rebase/cherry-pick)?
if [ -z "$(git rev-list --right-only --cherry-pick --no-merges "$target...$topic")" ]; then
  echo "merged: patch-equivalent"
  exit 0
fi

# 3. Optional heuristic for a pure squash merge
base=$(git merge-base "$target" "$topic")
topic_patch_id=$(
  git diff "$base" "$topic" | git patch-id --stable | awk '{print $1}'
)

if git log --no-merges -p "$base..$target" \
  | git patch-id --stable \
  | awk '{print $1}' \
  | grep -qx "$topic_patch_id"
then
  echo "maybe merged: squash heuristic matched"
  exit 0
fi

echo "not merged (or squash merge not provable)"
exit 1
```

If you want one rule for production automation, use this policy: ancestry test first, patch-equivalence second, and only call squash “merged” if your workflow records explicit metadata for it.
