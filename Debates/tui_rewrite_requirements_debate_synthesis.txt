TUI REWRITE REQUIREMENTS -- UNIFIED SYNTHESIS
================================================
Three-Agent Debate: Gemini (gemini-3.1-pro-preview) | Codex (gpt-5.3-codex) | Claude (security-auditor)
Date: 2026-03-06

This document merges the outputs of all three agents into a single authoritative
requirements specification. Where agents agreed, the requirement stands as consensus.
Where they diverged, the strongest position is selected with rationale noted.


================================================================================
0. NON-NEGOTIABLE CONSTRAINTS (All 3 agents agreed)
================================================================================

N-1. Every class component becomes a functional component with hooks. Zero class
     components in the rewrite. forceUpdate() is forbidden.
N-2. 100% functional parity: same views, same shortcuts, same navigation, same UI.
N-3. All state updates are immutable -- no in-place mutation of arrays, objects,
     Sets, or Maps. Always produce new copies.
N-4. Ink composition rules enforced: all text in <Text>, no <Box> inside <Text>,
     no raw strings outside <Text>.
N-5. Keyboard priority chain: overlay > focused control > view-local > global.
N-6. Tab traversal uses Ink focus primitives or a well-designed custom hook.
N-7. Fixed layout constants preserved: header 3 lines, issue header 5 lines,
     input area 3 lines, footer dynamic wrapping.

1. COMPONENT ARCHITECTURE
================================================================================

1.1 Restructured Component Tree (Codex proposed most complete, Claude refined)

  run.tsx
    AppRoot (functional)
      ErrorBoundary
      StoreProvider (AppStateContext via useReducer)
      NavigationProvider (view stack + saved selections)
      LayoutProvider (columns/rows from useStdout, resize handling)
      AppShell
        Header (functional, memo -- keep existing)
        ViewRouter (switch on currentView.type)
          HomeView
          DetailView
            IssueHeader (functional, add memo)
            ResponseChain (functional, memo)
              ResponseContainer (functional, memo with custom comparator)
            InputBox (functional, add memo)
          NewIssue (functional -- keep)
          AgentStatus (functional, memo -- keep)
          BlockingMap (functional, memo -- keep)
          GroupView (functional -- keep)
        ModalLayer (GroupPicker | IssueListPicker -- rendered above ViewRouter)
        Footer (functional, memo -- keep existing)

  NOTE: Codex proposed a ThemeProvider. All three agents want centralized tokens.
  Implementation: a simple theme module (exported constants) is sufficient over a
  full React context, since theme values don't change at runtime.

1.2 Input Bridge Elimination (Unanimous)

  All four bridge components are deleted:
    DetailInputBridge, HomeViewInputBridge, GroupPickerInputBridge, IssueListInputBridge
  Each parent functional component calls useInput() directly.

1.3 Custom Hooks (Merged from all 3 agents)

  HOOK                        SOURCE    PURPOSE
  useNavigation()             All 3     View stack: push/pop/replace/goHome/goBack
  useGlobalKeys()             All 3     Esc/q/s/b/g/n dispatch, returns consumed bool
  useViewport()               Gemini+   Terminal cols/rows via useStdout + resize
  useThreadNavigation()       All 3     Thread stack enter/exit/resolve + message list
  useFocusRing()              Claude    Dynamic circular Tab ring with slot count
  useSavedSelection()         Codex+    Per-inum selected message memory (useRef<Map>)
  useCursorNavigation()       Claude+   Reusable j/k/Up/Down cursor for list views
  useListCursor()             Codex     (Same as useCursorNavigation -- merge into one)
  useResponseViewport()       Codex+    Scroll anchor logic from ResponseChain
  usePersistentStore()        Codex     Load/save JSON with debounced writes

  Final hook list (deduplicated):
    useNavigation, useGlobalKeys, useViewport, useThreadNavigation,
    useFocusRing, useSavedSelection, useCursorNavigation,
    useResponseViewport, usePersistentStore

1.4 State Management (Consensus with Codex's detail)

  NavigationContext (useReducer):
    - viewStack: View[]
    - Actions: PUSH_VIEW, POP_VIEW, REPLACE_VIEW, GO_HOME
    - savedSelections: Map<number, number> (useRef, not reducer)
    - threadInfo, groupMode (lifted from old App class)

  AppStateContext (useReducer):
    - issues, unreadInums, maxAgents, detailData, containers,
      dependencies, containerIssues
    - Typed actions for: SET_ISSUE_STATUS, TOGGLE_DEPENDENCY, RESOLVE_THREAD,
      ADD_RESPONSE, CREATE_ISSUE, UPDATE_GROUP, MARK_READ, etc.
    - All reducers pure, no side effects

  Ephemeral state stays local (useState in each view):
    - cursor, activeField, overlay, inputValue, etc.

  Derived data via useMemo selectors:
    - Filtered issues, status counts, unread counts, dependency tree,
      response arrays, container progress

  - a: onActivate(inum)
  - d: onDefer(inum)
  - r: onResolve(inum)

--- 2.2 DETAIL VIEW ---

UI:
  Normal mode:
    - IssueHeader (5 lines): title bold, status+group (group inverse when focused),
      blockedBy+blocks (inverse when focused), Tab hint dim, separator rule
    - ResponseChain: scrollable viewport, height = rows - header - context - input - footer
    - InputBox (3 lines): separator, "Enter response: > " cyan, TextInput
    - Overlays: GroupPicker or IssueListPicker replace content when active

  Thread mode:
    - ThreadHeaderSeparator: "[ Thread ]" or "[ Thread -- Resolved ]" centered
    - Thread parent ResponseContainer (gray, isThreadParent=true)
    - "[ Replies ]" separator
    - ResponseChain showing reply chain

Keyboard (priority order, all 3 agents agreed):
  1. Overlay active: all keys to overlay, DetailView suppressed
  2. Alt+H (or U+02D9): navigate home
  3. Esc: close overlay > exit thread > go back
  4. Ctrl+Shift+Right: enter thread (no-op if no replies)
  5. Ctrl+Shift+Left: exit thread
  6. Ctrl+R: resolve issue (Active<->Resolved) or resolve thread
     (Must suppress 'r' echo into TextInput -- use isActive guard or modifier check)
  7. Shift+Left: previous adjacent issue (replaceCurrentView)
  8. Shift+Right: next adjacent issue (replaceCurrentView)
  9. Tab: cycle focus forward through ring
  10. Shift+Tab: cycle focus backward through ring
  11. Enter on viewReplies: toggle thread resolution
  12. Enter on focused field: open overlay (Group->GroupPicker, BlockedBy/Blocks->IssueListPicker)
  13. Enter on focused footer item: dispatch footer action
  14. Up: selectedMessage -1, clamped, clear viewReplies focus
  15. Down: selectedMessage +1, clamped, clear viewReplies focus

Focus Ring (strict order):
  Input -> Group -> BlockedBy -> Blocks -> ViewReplies -> FooterItems -> (wrap to Input)
  In thread mode: Input -> ViewReplies -> FooterItems -> (wrap)
  (Header fields excluded in thread mode, ViewReplies only when selected has replies)

State: inputValue, focusedField, focusedFooterIndex, focusedViewReplies,
  overlay, blockedBySet, blocksSet, selectedMessage, threadStack[], suppressNextInput
Data: inum, issue, rootResponse, blockedBy[], blocks[], group, columns, rows,

3. CROSS-CUTTING CONCERNS
================================================================================

3.1 Navigation (All 3 agreed)
  - View stack array, minimum 1 entry (root = Home)
  - Operations: push, pop, replace, goHome
  - Global keys: Esc back, q quit, s AgentStatus, b BlockingMap, g GroupView, n NewIssue
  - Per-view selections persist across navigations via useRef<Map>
  - Footer focus resets to null on navigation
  - Minimize raw ANSI escape codes for screen clearing

3.2 Focus Management (Claude's analysis most thorough)
  - DetailView focus ring is dynamic (slot count changes per thread mode + reply state)
  - Two viable approaches:
    A) Ink useFocus/useFocusManager: each focusable element is a component
    B) Custom useFocusRing hook: manual index with dynamic slot count
  - RECOMMENDATION: Option B (custom hook). Rationale: the focus ring includes
    non-component targets (header fields, conceptual buttons). A custom hook is
    simpler and avoids conflicts with Ink's global focus system.
  - Modal overlays MUST trap focus and restore on close

3.3 Keyboard Architecture (Codex's tier model, Claude's detail)
  Three tiers:
    Tier 1 - Global: useGlobalKeys() in views that want it
    Tier 2 - View-specific: useInput() per view with own handler
    Tier 3 - Overlay/modal: priority when active, parent handler suppressed
  Hotkey utilities preserved: matchesKey(), matchesKeyCombination(), all enums
  useInput isActive parameter gates handler activation based on overlay state

3.4 Data Layer (All 3 agreed)
  - No direct MOCK_* imports in view components
  - All data through AppStateContext
  - Mutations via reducer dispatch -> immutable updates
  - Mock store remains as persistence layer, loaded on mount
  - Auto-save after mutations (debounced write)
  - Thread resolution: either immutable Response chain relinking or external
    resolution state map keyed by response ID (Codex+Claude both noted this)

3.5 Theming/Styling (Claude most detailed)
  Centralized theme module with tokens:
    Status colors: Active=green, Awaiting=blue, Blocked=red, Deferred=yellow, Resolved=gray
    UI colors: selected=cyan, authorUser=cyan, authorAgent=green, threadParent=gray,
      unread=yellow, error=red, disabled=dim
    Characters: cursor=U+25B8, unread=U+2731, alive=U+25CF, dead=U+25CB,
      rule=U+2500, checkmark=U+2713

3.6 Error Handling (All 3 agreed)
  - Top-level ErrorBoundary with fallback UI and Esc-to-home
  - Graceful empty states for missing data
  - Data load/save failures surface as non-crashing banner
  - Invalid actions are no-ops with hints (e.g., enter thread when no replies)
  - Async side effects handle cancellation on unmount

3.7 Accessibility (Codex most specific)
  - Use useIsScreenReaderEnabled() for detection
  - All color indicators have text alternatives (labels alongside colors)
  - Focused element has non-color indicator (inverse, prefix marker, or label)
  - Keyboard-only covers 100% of functionality
  - Header/footer readable on narrow terminals

3.8 Performance (All 3 agreed, Claude most specific)
  Memoization priorities:
    - ResponseContainer: React.memo with custom comparator (id, selected, hasNewReplies,
      threadResolved, viewRepliesFocused)
    - ResponseChain: memoize message array and viewport calculations
    - flattenToArray(): memoize based on rootResponse identity
    - computeLineCount(): memoize per (body, columns) if profiling shows need
    - buildFlatRows(): already memoized -- keep
    - Header, Footer: already memo'd -- keep
  Scroll anchoring: useResponseViewport hook tracks lastFirstVisible/lastHeight
    via useRef, resets on chain identity change (thread enter/exit)
  Avoid rerendering response chain on unrelated input keystrokes
  All callbacks via useCallback for stability


================================================================================
4. INK-SPECIFIC IMPLEMENTATION
================================================================================

4.1 Box/Text Composition (Unanimous)
  - All text in <Text>, all layout in <Box>
  - Horizontal: <Box flexDirection="row">
  - Vertical: <Box flexDirection="column">
  - Borders on overlays/pickers: <Box borderStyle="single">
  - Response containers: manual Unicode box-drawing (preserved, not Ink borders)
  - <Spacer /> for flexible gaps, gap={} for consistent spacing

4.2 Focus Integration
  - Use useFocus({ autoFocus }) for initial focus per view
  - Use useInput with isActive guard for overlay suppression
  - Custom useFocusRing for DetailView's complex focus ring
  - Modal focus trap via isActive toggling

4.3 useInput Patterns
  Handler structure:
    1. Guard: return if overlay active (or use isActive parameter)
    2. Check modifier combos (Ctrl+Shift+Arrow, Ctrl+R, Alt+H)
    3. Check special keys (Esc, Return, Tab, arrows)
    4. Check character keys (j, k, a, d, r, n, q)
    5. Fallthrough to global handler

4.4 Responsive Layout
  - Terminal dimensions from useStdout + resize listener
  - Width flows through LayoutContext (not prop drilling)
  - Width-sensitive components wrap/truncate based on current columns
  - No hard-coded widths except fixed constants (HEADER_LINES, etc.)

4.5 Border & Spacing Conventions
  - Response container inner width: columns - 4
  - Unicode box-drawing: U+250C, U+2510, U+2514, U+2518, U+2500, U+2502
  - Spacing: gap={2} between tabs, marginBottom={1} section headers,
    paddingLeft/Right={1} bordered boxes

6. BUGS AND ERRATA (Claude's source-code audit)
================================================================================

BUG-1: GroupPicker create-mode has no Esc handler. The InputBridge only renders
  in list mode, so Esc in create mode is unhandled. Rewrite must fix this.

BUG-2: AgentStatus dead color is red in code, but was documented as gray.
  Rewrite uses red.

BUG-3: AgentStatus lacks global key handling (Esc/q not handled). Rewrite
  should add global keys for consistency.

BUG-4: Home view column order in prior docs was wrong. Actual: cursor, inum,
  title, unread, status. Rewrite follows actual code order.

BUG-5: App.render() switch only handles Home, Detail, IssuePicker. NewIssue,
  AgentStatus, BlockingMap, GroupView are navigated to but never rendered.
  Rewrite MUST implement all seven view routes in ViewRouter.
