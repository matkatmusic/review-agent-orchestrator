DEBATE SYNTHESIS: Methods to determine if worktree branch WT1 has been merged into B
================================================================================

VERDICT: UNANIMOUS CONSENSUS
All three agents (Gemini, Codex, Claude) converged on the same layered approach.

================================================================================
AGREED PRIMARY METHOD: git merge-base --is-ancestor
================================================================================

Command:
  git merge-base --is-ancestor wt1-branch B

Why all three chose this:
- Plumbing command (not porcelain) -- returns exit code 0/1 with no stdout
- Traverses the DAG to definitively prove ancestry
- Natively handles: fast-forward merges, standard merge commits, octopus merges
- Performance: graph traversal is heavily optimized in Git, fast even with
  thousands of branches (no file content loading or diff computation)
- Worktree-agnostic: works whether the worktree directory exists or not,
  as long as the branch ref (or commit SHA) is still in .git

================================================================================
AGREED SECONDARY: git branch --merged (batch queries)
================================================================================

Command:
  git branch --merged B | grep -q wt1-branch

Useful for checking many branches at once. Porcelain command, requires output
parsing, but convenient for batch enumeration of all merged branches.

================================================================================
AGREED TERTIARY: git cherry (squash/rebase detection)
================================================================================

Command:
  git cherry B wt1-branch

Lines prefixed with '-' = already in B (by patch-id)
Lines prefixed with '+' = not in B

If all lines are '-', the branch was effectively squash-merged or cherry-picked.

Limitation: many-to-one squash merges (multiple commits squashed into one) break
patch-id matching since the combined patch doesn't match individual patches.

================================================================================
AGREED SCRIPT PATTERN
================================================================================

  #!/usr/bin/env bash
  set -euo pipefail

  is_branch_merged() {
      local branch="${1:?usage: is_branch_merged <branch> [target]}"
      local target="${2:-HEAD}"

      # Layer 1: Graph-based ancestry check (true merge / fast-forward)
      if git merge-base --is-ancestor "$branch" "$target" 2>/dev/null; then
          echo "MERGED (ancestor): $branch is reachable from $target"
          return 0
      fi

      # Layer 2: Patch-id check for squash/cherry-pick merges
      local unmerged_count
      unmerged_count=$(git cherry "$target" "$branch" 2>/dev/null \
                       | grep -c '^+' || true)

      if [ "$unmerged_count" -eq 0 ]; then
          echo "MERGED (squash/cherry-pick): all patches from $branch in $target"
          return 0
      fi

      echo "NOT MERGED: $unmerged_count commit(s) from $branch not in $target"
      return 1
  }

  is_branch_merged "$@"

================================================================================
EDGE CASES (ALL THREE AGREE)
================================================================================

1. Squash merges: merge-base --is-ancestor misses them (commits rewritten).
   git cherry partially catches this but fails on many-to-one squashes.

2. Rebase-and-merge: Similar to squash -- new commit SHAs. git cherry handles
   one-to-one rebased commits via patch-id matching.

3. Deleted branch ref: If the branch ref AND its reflog are gone, Git cannot
   prove anything. Must have the old tip SHA saved externally.

4. Worktree removed: Irrelevant. Branch refs live in .git, not the worktree
   directory. Removing a worktree does not delete its branch.
