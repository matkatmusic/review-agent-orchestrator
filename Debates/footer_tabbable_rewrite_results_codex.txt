**Answers to your 7 questions**

1. **Tab focus ring behavior**
- Issue view order: `Input -> Group -> Blocked By -> Blocks -> Footer item 0..n-1 -> Input` (wrap).
- Thread view order: `Input -> Footer item 0..n-1 -> Input` (no header fields).
- `Shift+Tab` walks the same ring in reverse.
- This directly fixes current thread bug where Tab focuses hidden header fields.

2. **Footer focus state propagation**
- `DetailView` owns keyboard/focus ring.
- `DetailView -> App` via `onFooterFocusChange(index)`.
- `App -> Footer` via `focusedIndex` prop.
- Footer remains purely presentational.

3. **Enter on focused footer item**
- Yes, for actionable items.
- Define actions in footer shortcut data (in `footer.tsx`), but execute in `DetailView` through a switch mapping to existing methods (`exitThread`, `resolveThread`, `onHome`, etc.).
- Non-action items (scroll hint, disabled shortcuts) are focusable for consistency but Enter is no-op.

4. **`FOOTER_LINES` when wrapping**
- Static `1` will cause layout drift when footer wraps.
- Yes, conversation/input viewport budgeting should account for actual footer line count.
- Best approach: dynamic line estimate from shortcut widths and terminal columns.

5. **`flexWrap="wrap"` with spacing**
- Use per-item `marginRight` rather than embedding separator spaces in text.
- This prevents dangling leading spaces on wrapped lines and keeps item boundaries explicit.
- Set `flexShrink={0}` on item boxes so items don’t compress and split internally.

6. **Ink pitfalls with `<Text inverse>` in wrapped boxes**
- Nested styled `<Text>` can fight `inverse`; apply focus style at item root text and minimize conflicting child styles.
- `dimColor + inverse` can become hard to read; define explicit focused-disabled style decision.
- Snapshot text output usually won’t expose inverse state; style assertions need behavioral/structural proxies.
- Very narrow terminals may still force awkward truncation for a single long item; handle gracefully.
