    M.3.2: CHECK MERGE STATUS (R8)
      - Call is_worktree_merged() — see Section 4

    M.3.3: IF MERGED → CLEANUP
      a) Remove scaffold files (.claude/settings.json, .vscode/tasks.json)

      c) If dirty: log WARNING, skip deletion
      d) If clean: kill tmux, git worktree remove --force, git branch -D, git worktree prune

  M.4: sleep 5

## 4. is_worktree_merged() — Merge Commit Parent Scan

Scans ALL merge commits for one that has the worktree branch tip as a non-first
parent. This means someone ran `git merge --no-ff worktree_branch`.

No false positives on freshly-created branches (no merge commit exists yet).
No lockfiles. No grace period.

```bash
is_worktree_merged() {
  local branch_name="$1"
  local repo_root="${2:-${REPO_ROOT}}"
  local branch_tip
  branch_tip=$(git -C "${repo_root}" rev-parse "refs/heads/${branch_name}" 2>/dev/null) || return 1

  while IFS= read -r line; do
    read -ra parts <<< "${line}"
    # parts[0]=merge_hash, parts[1]=first_parent, parts[2..n]=merged-in parents
    for (( i=2; i<${#parts[@]}; i++ )); do
      [ "${parts[$i]}" = "${branch_tip}" ] && return 0
    done
  done < <(git -C "${repo_root}" log --all --merges --format='%H %P' 2>/dev/null)

  return 1
}
```

Edge cases handled:
- Octopus merges: checks all non-first parents
- Commits after merge: tip changes, old merge commit no longer matches → correct (not merged)
- Branch deleted: rev-parse fails → returns 1

**A) Issues, Gaps, and Edge Cases**

1. `CRITICAL` The spec treats [`.vscode/tasks.json`](/Users/matkatmusicllc/Programming/authv5-workspace/review-agent-orchestrator/.vscode/tasks.json) and [`.claude/settings.json`](/Users/matkatmusicllc/Programming/authv5-workspace/review-agent-orchestrator/.claude/settings.json) as disposable scaffold, but in this repo both are tracked files. `has_non_scaffold_changes()` would ignore real tracked changes there, and cleanup could silently delete user work. Fix: do not overwrite/delete tracked repo files as ephemeral scaffold; use a separate managed file or inject a marked snippet and only remove the marked/generated section.

2. `CRITICAL` The daemon discovers all linked worktrees from `git worktree list` and has no ownership marker, so it can act on user-created worktrees that were never spawned by this system. Fix: mark managed worktrees explicitly, ideally in per-worktree Git config (`git config --worktree ...`) or another persistent marker outside tracked files, and ignore everything else.

3. `CRITICAL` `runOn: "folderOpen"` does not guarantee first-launch automation. VS Code explicitly prompts the first time a folder with automatic tasks is opened, so R3, R6, and R10 will not reliably happen on first open without prior permission. Fix: use an explicit bootstrap command, or preconfigure `task.allowAutomaticTasks` for the managed worktree, or do not make critical behavior depend solely on `folderOpen`.

4. `HIGH` The worktree task graph is wrong for R6. All three tasks auto-run on folder open, and two of them also `dependsOn` `npm install`, which creates duplicate/racy startup paths rather than one deterministic sequence. Fix: make exactly one compound task auto-run, set `dependsOrder: "sequence"`, and keep child tasks non-auto-run.

5. `CRITICAL` The daemon can delete a merged worktree while its IDE window, Claude process, tmux session, or `npm install` is still active. The spec only kills tmux when IDE is already dead, but cleanup runs regardless of IDE liveness. Fix: never remove a worktree while the managed IDE/processes are alive; mark it pending cleanup and wait until the window is closed.

6. `HIGH` The merge-parent scan is not sufficient to avoid false positives. I reproduced a fresh branch being misclassified as merged when it was created at a commit that had already appeared historically as a non-first merge parent. Fix: persist branch-birth metadata such as creation commit/time and only consider merge commits newer than that birth point, or use stored spawn metadata in worktree config.

7. `HIGH` The merge logic breaks on detached worktrees and branch-deleted-after-detach cases. `git worktree list --porcelain` can emit `detached` instead of `branch ...`, and your function returns false if `refs/heads/<branch>` is gone. Fix: handle detached entries explicitly; either declare them unsupported and skip with warning, or store the managed branch name and original tip independently of current checkout state.

8. `HIGH` `pgrep -f` with basename/path substrings is too weak for both daemon singleton and IDE liveness. On macOS `pgrep -f` is substring-based unless `-x` is used, so false positives are easy, and concurrent spawns can race into multiple daemons. Fix: use canonical absolute paths with exact matching plus an atomic singleton mechanism; if “no lockfiles” only applies to business state, still use a lock for synchronization.

9. `HIGH` `tmux new-session -s __NAME__` is not idempotent and does not satisfy reopen semantics. If the session already exists, it errors instead of attaching, and CWD is only implicit. Fix: use `tmux new-session -A -s __NAME__ -c "${workspaceFolder}"` or `has-session || new-session -d ...; attach-session ...`.

10. `MEDIUM` Case B uses `git worktree prune` globally, which can prune unrelated stale worktrees and surprise the user. Git already supports targeted recovery for missing assigned paths via `git worktree add --force`. Fix: avoid global prune in spawn; repair or force-add only the target worktree.

11. `MEDIUM` The `git worktree list --porcelain` parsing requirements are incomplete. You need `-z` and handling for `detached`, `locked`, `prunable`, `bare`, and paths containing unusual characters; “skip the first entry” is not enough of a parser contract. Fix: parse `--porcelain -z` record-by-record and ignore unsupported states safely.

12. `MEDIUM` Running `npm install` on every worktree-window open is a side effect, not a benign bootstrap step. It can mutate lockfiles, run postinstall scripts, fail offline, and block Claude/tmux indefinitely on network or disk failures. Fix: gate it, add retry/failure handling, or explicitly document that worktree open is stateful and may fail for environmental reasons.

13. `MEDIUM` The dirty-check algorithm is underspecified for renames, submodules, and tracked scaffold files. A simple porcelain filter is not enough when the ignored files are tracked or when status lines are not plain `?? path`. Fix: use `git status --porcelain=v1 -z`, parse status codes correctly, and only ignore exact generated files that are verified to match the generated template.

14. `MEDIUM` Git edge cases are not addressed: locked worktrees, submodules, shallow clones, moved worktrees needing `git worktree repair`, bare repos, and symlinked paths. Git’s own docs still call multi-checkout submodule support incomplete. Fix: either explicitly declare these unsupported and fail fast, or add detection and conservative skip/manual-cleanup behavior.

15. `LOW` The signal model is optimistic. The spec says the daemon “dies naturally via SIGHUP,” but the script only traps `SIGINT`/`SIGTERM`, and VS Code task shutdown/background-shell behavior is not reliable enough to treat HUP as guaranteed. Fix: trap `SIGHUP` and `EXIT`, and prefer explicit parent-process monitoring over terminal-close assumptions.

16. `LOW` The path-state matrix only checks `-d`. A symlink to a directory, a plain file at the target path, or a non-canonical path can all break later steps in confusing ways. Fix: use `-e`, canonicalize with `pwd -P`, reject symlink targets, and emit precise recovery instructions.

**B) Accuracy / Completeness Verdict**

The spec is not ready for implementation as-is. R2 is straightforward, and I locally verified that `claude --permission-mode plan` is a valid CLI flag, but R1 and R3-R10 still have correctness gaps around ownership, task automation, merge detection, liveness detection, and destructive cleanup behavior.

The biggest blockers are the tracked-file scaffold design, the assumption that `folderOpen` tasks will auto-run on first launch, the ability to delete active worktrees, and the fact that the merge-detection algorithm cannot distinguish “branch created now at old commit” from “branch merged earlier” without extra metadata.

**C) Final Recommendation**

Revise the spec before implementation. Minimum changes:
1. Add a managed-worktree marker and stop using tracked repo files as disposable scaffold.
2. Replace the three auto-run tasks with one compound bootstrap task.
3. Make cleanup wait until the worktree IDE/processes are gone.
4. Add branch-birth metadata for merge detection.
5. Replace `pgrep` substring matching with exact canonical matching plus an atomic singleton lock.
