DEBATE SYNTHESIS: Worktree Spawner Rewrite — Completeness & Edge Case Review
=============================================================================
Agents: Claude (security-auditor), Codex (gpt-5.4), Gemini (gemini-3.1-pro-preview)

UNANIMOUS VERDICT (3/3): NOT READY FOR IMPLEMENTATION AS-IS. Revisions needed.
All three agents independently reached the same conclusion. The core design is
sound, but there are blocking issues around process management, VS Code task
automation reliability, and merge detection edge cases.

=============================================================================
CRITICAL ISSUES (must fix before implementation)
=============================================================================

1. PGREP SUBSTRING FALSE POSITIVES [Claude #1, Codex #5, Gemini #1]
   All three agents flagged pgrep as the most likely real-world bug source.
   pgrep -f "agy.*reponame" will match ANY process containing that
   substring (other editors, grep commands, file paths). Can cause:
   - Daemon thinks main IDE is alive when it's dead (or vice versa)
   - Worktree IDE misdetected, orphaning tmux sessions
   GEMINI ESCALATION: The daemon singleton check will match ITS OWN
   PROCESS — meaning the daemon can never launch. This is a day-one
   showstopper that neither Claude nor Codex caught.
   FIX: (a) Use the bracket trick: pgrep -f "[w]orktreeDaemon.sh.*${REPO_ROOT}"
   to exclude the pgrep command itself. (b) Use canonical/absolute paths.
   (c) Best: PID file + kill -0 for singleton, PID capture for IDE detection.

2. SCAFFOLD FILES ARE TRACKED REPO FILES [Codex #1]
   .vscode/tasks.json and .claude/settings.json are tracked in git.
   The spec treats them as disposable scaffold (overwrite on spawn,
   delete on cleanup). This silently destroys real tracked changes.
   FIX: Use a separate managed file (.worktree-managed marker) or
   use per-worktree git config. Never delete tracked files as cleanup.

3. NO OWNERSHIP MARKER ON WORKTREES [Codex #2]
   Daemon discovers ALL linked worktrees via git worktree list and
   has no way to distinguish managed vs user-created worktrees.
   Could accidentally clean up manual worktrees.
   FIX: Write a marker file (e.g., .managed-worktree) during scaffold.
   Only act on worktrees that have this marker.

4. runOnFolderOpen FIRST-LAUNCH TRUST PROMPT [Codex #3, Claude #7]
   VS Code prompts the user to trust automatic tasks the first time
   a folder with runOnFolderOpen is opened. R3, R6, and R10 will NOT
   auto-run on first open without prior permission. Also, dependsOn
   behavior with runOnFolderOpen background tasks is unreliable.
   FIX: Either (a) pre-configure task.allowAutomaticTasks in workspace
   settings during scaffold, or (b) replace three auto-run tasks with
   one compound bootstrap script that handles sequencing internally.

5. RACE CONDITION: SPAWN vs DAEMON [Claude #2]
   If daemon polls during the window between branch creation and
   scaffold completion, it may see an "unmanaged" worktree and
   attempt cleanup before spawn finishes.
   FIX: Write the managed-worktree marker FIRST (before scaffold),
   or have spawn signal the daemon to skip the new worktree for one
   cycle. Alternatively, marker-based ownership (issue #3) solves this.

6. MERGE DETECTION: BRANCH-CREATED-AT-OLD-COMMIT [Codex #4]
   If a branch is created at an ancestor of an existing merge commit
   (e.g., git worktree add -b foo path OLD_COMMIT), the parent scan
   may find a historical merge commit whose parent matches the new
   branch tip — false positive.
   FIX: Persist branch-birth metadata (creation timestamp or commit)
   during spawn. Only consider merge commits newer than birth.

=============================================================================
HIGH ISSUES (should fix — high risk of user-facing bugs)
=============================================================================

7. DAEMON EXITS WHEN MAIN IDE CLOSES BUT WORKTREE IDE STILL OPEN [Gemini #5]
   R10 says daemon exits when main IDE closes. But if a user closes the
   main IDE while worktree IDEs are still open, the daemon dies. Merge
   detection ceases, and closing the worktree IDE later won't kill tmux
   (violating R7). UNIQUE FINDING — neither Claude nor Codex caught this.
   FIX: Daemon should check if ANY relevant agy process is running
   (main IDE OR any worktree IDE) and only exit when all are gone.

8. isBackground + EMPTY problemMatcher HANGS TASK ENGINE [Gemini #6]
   The daemon task sets isBackground: true with empty problemMatcher: [].
   VS Code background tasks without beginsPattern/endsPattern are never
   marked "ready," which can hang the task engine and prevent other
   runOnFolderOpen tasks from starting.
   FIX: Don't use isBackground for the daemon launcher. Make it a
   standard task that forks with nohup and exits immediately:
   "command": "nohup ${workspaceFolder}/worktreeDaemon.sh '${workspaceFolder}' > /dev/null 2>&1 &"

9. MERGE SCAN CPU BOTTLENECK — BASH WHILE-READ LOOP [Gemini #2]
   Piping git log --all --merges into a bash while-read loop processes
   thousands of lines in bash every 5 seconds per worktree. This will
   heavily peg CPU on any non-trivial repo.
   FIX: Replace with awk one-liner (C-based, processes in milliseconds):
   git log --all --merges --format='%P' | \
     awk -v tip="${branch_tip}" '{for(i=2;i<=NF;i++) if($i==tip) exit 0} END{exit 1}'

10. set -e IN DAEMON LOOP [Claude #6]
    If the spec uses set -e (errexit), ANY command failure in the
    daemon loop kills the daemon silently. git commands, pgrep, tmux
    commands can all return non-zero legitimately.
    FIX: Do NOT use set -e in the daemon. Use explicit error checking
    on every command, or || true on commands that may legitimately fail.
    Add trap on EXIT (not just SIGINT/SIGTERM).

11. git worktree remove --force DATA LOSS [Claude #8, Gemini #7]
    --force bypasses the dirty-worktree check built into git. Even
    though the spec checks git status first, there's a TOCTOU race.
    Gemini: "If your custom git status check has a bug, --force will
    irreversibly delete user work."
    FIX: Use git worktree remove (without --force) first. Only escalate
    to --force if it fails AND the spec's own dirty check passed.

12. TMUX SESSION NAME COLLISION [Claude #4, Gemini #4]
    If a tmux session with the worktree name already exists (from a
    previous unclean shutdown), tmux new-session -s NAME will fail.
    GEMINI FIX (better): Use tmux new-session -A -s NAME (the -A flag
    attaches to existing session or creates new one automatically).

13. SINGLETON DETECTION WITHOUT PID FILE [Claude #5]
    Using pgrep for singleton detection is less reliable than a PID
    file with kill -0 validation. PID reuse on long-running systems
    can cause false positives.
    FIX: Keep a PID file (write $$ at start, validate with kill -0
    on detection). This is a regression from the current design.

14. SCAFFOLD COMPARISON IN CASE A IS UNDERSPECIFIED [Claude #15]
    .vscode/tasks.json is generated via sed with __NAME__ substitution.
    Comparing against the template (which has __NAME__ placeholders)
    will ALWAYS show a diff. The comparison needs to reverse-substitute
    or compare against a rendered version.
    FIX: Generate the expected version (sed the template with the
    current name) into a temp file and diff against that.

15. CLEANUP OF ACTIVE WORKTREES [Codex, Gemini #5]
    Nothing prevents the daemon from cleaning up a worktree where
    the IDE or Claude is still running (merge detected = instant
    cleanup). Could kill active work sessions.
    FIX: Always check IDE/tmux/Claude process liveness BEFORE
    cleanup. If any process is alive, warn but do not clean up.

16. INTERACTIVE PROMPT HIDDEN IN AUTOMATED TASK [Gemini #8]
    Case A prompts "Overwrite? [y/N]". If spawnWorktree.sh is triggered
    via a VS Code task or Command Palette shortcut that defaults to a
    silent terminal, the script hangs indefinitely waiting for invisible
    input.
    FIX: Ensure the spawn task sets presentation.reveal: "always" and
    presentation.focus: true. Or accept a --force flag to skip prompts.

=============================================================================
MEDIUM ISSUES (should address for robustness)
=============================================================================

17. MERGE SCAN PERFORMANCE [Claude #3, Gemini #2]
    git log --all --merges --format='%H %P' scans ENTIRE repo history
    every 5 seconds. On repos with thousands of merge commits this
    is expensive. FIX: Limit with --since or --ancestry-path, or
    cache the last-checked merge commit hash. Also use awk instead
    of bash while-read (see HIGH issue #9).

18. MERGE DETECTION: NO SQUASH/REBASE SUPPORT [Claude #12, Gemini #3]
    The parent scan only detects --no-ff merges. If someone squash-
    merges or rebases, the worktree will never be auto-cleaned.
    This is acceptable per R8 (which specifies --no-ff), but should
    be documented as a known limitation.
    GEMINI FIX: Add fallback — check if git branch -vv shows the
    tracking branch as [gone] (indicating PR merged + remote branch
    deleted). This covers squash/rebase-merged PRs.

19. RE-VERIFY BRANCH TIP BEFORE DELETION [Claude #13]
    Between merge detection and cleanup, a new commit could be pushed
    to the branch. FIX: Re-check branch tip matches the detected
    merged tip before running git branch -D.

20. git worktree list --porcelain PARSING [Claude #17]
    Porcelain format edge cases: detached HEAD worktrees have no
    branch field; prunable entries have extra fields. FIX: Handle
    these cases explicitly in get_worktrees().

21. macOS SIGHUP BEHAVIOR [Claude implicit]
    On macOS, terminal SIGHUP behavior differs from Linux. The
    daemon may not receive SIGHUP when the IDE terminal closes if
    VS Code manages the process differently. The M.1 pgrep check
    is the actual safety net, not SIGHUP.

=============================================================================
LOW ISSUES
=============================================================================

22. Claude CLI flag syntax unverified [Claude #24] — spec notes this.
23. WORKTREE_PATH uses relative ../ which could resolve unexpectedly
    with symlinks. FIX: Use realpath/readlink -f.
24. No log rotation or max log size for daemon output.
25. No timeout on npm install — could block Claude/tmux indefinitely.
26. MKDIR SYNTAX ERROR IN SPEC [Gemini #10]
    Step 4 says "mkdir -p .claude, .vscode" — trailing comma after
    .claude will create a directory literally named ".claude," on most
    systems. FIX: "mkdir -p .claude .vscode"
27. WEAK INPUT VALIDATION [Gemini #11]
    Spec says validate [a-zA-Z0-9_-]+ but only performs space-to-
    underscore replacement. "my.branch/name" passes the replacement
    but breaks tmux/path logic. FIX: Add explicit enforcement check:
    if [[ ! "$WORKTREE_NAME" =~ ^[a-zA-Z0-9_-]+$ ]]; then exit 1; fi

=============================================================================
RECOMMENDED ACTION (consolidated from all 3 agents)
=============================================================================
Revise the spec to address at minimum:
  1. Process detection: Replace all pgrep -f with PID-file + kill -0 approach.
     Fix self-detection bug with bracket trick or PID exclusion.
  2. Scaffold design: Use untracked marker files, not tracked repo files.
  3. Ownership: Add .managed-worktree marker during spawn.
  4. Task automation: Single compound bootstrap script (npm install && tmux
     && claude). Remove isBackground from daemon task — use nohup fork.
  5. Merge detection: Add branch-birth metadata, limit git log scan window,
     replace bash while-read with awk. Add squash-merge fallback via
     tracking-branch [gone] detection.
  6. Daemon safety: No set -e, explicit error handling, re-verify before
     destructive ops. Remove --force from git worktree remove.
  7. Daemon lifecycle: Watch ALL IDE windows (main + worktree), not just main.
  8. Cleanup guards: Check all processes dead before removing worktree.
  9. Input validation: Actually enforce the [a-zA-Z0-9_-]+ regex.
 10. tmux: Use new-session -A flag for idempotent session creation.

Then re-review before implementation.
