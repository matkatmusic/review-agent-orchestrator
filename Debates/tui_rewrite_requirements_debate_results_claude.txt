1. COMPONENT ARCHITECTURE REQUIREMENTS
================================================================================

1.1 Component Tree Restructuring
---------------------------------

The current tree has unnecessary indirection caused by class components needing
bridge components to access hooks. The restructured tree should be:

  run.tsx
    AppWrapper (functional, exists already)
      NavigationProvider (new context)
        AppStateProvider (new context)
          AppShell (new: wraps Header + ViewRouter + Footer)
            Header (functional, already memo'd -- keep)
            ViewRouter (new: switch on currentView.type)
              HomeView (rewrite from class to functional)
              DetailView (rewrite from class to functional)
                IssueHeader (functional -- keep, add memo)
                ResponseChain (rewrite from class to functional)
                  ResponseContainer (rewrite from class to functional, memo)
                InputBox (functional -- keep, add memo)
                GroupPicker (rewrite from class to functional)
              IssueListPicker (rewrite from class to functional)
              NewIssue (functional -- keep)
              AgentStatus (functional, memo'd -- keep)
              BlockingMap (functional -- keep)
              GroupView (functional -- keep)
            Footer (functional, memo'd -- keep)

Key structural changes:
  - Eliminate all InputBridge components (DetailInputBridge, HomeViewInputBridge,
    GroupPickerInputBridge, IssueListInputBridge). These exist only because class
    components cannot call useInput. With functional components, useInput is called
    directly in the owning component.
  - Extract ViewRouter from the monolithic App.render() switch statement.
  - Introduce NavigationProvider and AppStateProvider to eliminate prop drilling.

1.2 Custom Hooks to Extract
-----------------------------

HOOK: useNavigation()
  Purpose: Manages the view stack with push/pop/replace/home semantics.
  Backed by useReducer with actions: PUSH_VIEW, POP_VIEW, REPLACE_VIEW, GO_HOME.
  Returns: { currentView, viewStack, navigateTo, goBack, replaceView, goHome }
  Provided via NavigationContext.

HOOK: useGlobalKeys(currentViewType)
  Purpose: Handles the 6 global shortcuts (Esc, q, s, b, g, n).
  Calls useInput internally.
  Returns boolean indicating whether the key was consumed.
  Must check currentViewType to prevent pushing duplicate views.

HOOK: useViewport(messages, selectedIndex, height, columns)
  Purpose: Encapsulates the scroll anchor logic currently in ResponseChain.
  Tracks lastFirstVisible, lastHeight, lastRootResponse via useRef.
  Returns: { firstVisible, lastVisible, visibleMessages }
  Must reset anchor when messages array identity changes (thread enter/exit).

HOOK: useThreadNavigation(rootResponse)
  Purpose: Manages the thread stack currently stored as a class instance array.
  State: threadStack via useState with immutable updates.
  Returns: {
    currentMessages: Response[],
    isInThread: boolean,
    threadParent: Response | null,
    displayedRoot: Response | null,
    enterThread: (selectedIndex: number) => void,
    exitThread: () => { restoredIndex: number } | null,
    resolveThread: (selectedIndex: number) => void,
    headerLineCount: number
  }

HOOK: useFocusRing(slotCount, options)
  Purpose: Manages circular Tab focus cycling with a dynamic number of slots.
  The slot count changes based on thread mode and whether the selected message
  has replies.
  Returns: {
    focusedSlot: number | null,
    cycleFocus: (direction: 1 | -1) => void,
    resetFocus: () => void,
    isSlotFocused: (slot: number) => boolean
  }
  Should integrate with Ink's useFocusManager if possible, but the current focus
  ring includes non-component slots (header fields, footer items) that useFocus
  cannot manage. A custom implementation may be necessary.

HOOK: useSavedSelection()
  Purpose: Persists the selected message index per issue inum across navigations.
  Backed by useRef<Map<number, number>>.
  Returns: {
    getSavedIndex: (inum: number) => number | undefined,
    saveIndex: (inum: number, index: number) => void
  }

HOOK: useCursorNavigation(listLength, options?)
  Purpose: Reusable up/down cursor for HomeView, AgentStatus, BlockingMap, etc.
  Returns: { cursor, moveCursor, resetCursor }
  Accepts optional wrap-around configuration.

1.3 State Management Architecture
-----------------------------------

NavigationContext:
  - viewStack: View[] managed by useReducer
  - savedSelections: Map<number, number> managed by useRef
  - focusedFooterIndex: number | null
  - threadInfo: { inThread: boolean; threadResolved?: boolean; selectedHasReplies?: boolean }
  - groupMode: GroupMode (persists across navigation)

AppStateContext:
  - issues: Issue[]
  - unreadInums: Set<number>
  - maxAgents: number
  - detailData: Record<number, DetailMockData>
  - containers: Container[]
  - dependencies: Dependency[]
  - containerIssues: Record<number, Issue[]>
  - Mutation actions (dispatch or callbacks) for immutable updates

Critical state management rule: ALL state updates MUST be immutable. The current
codebase mutates arrays and objects in place (e.g., arr.splice(idx, 1), directly
setting issue.status) and then calls forceUpdate(). The rewrite MUST produce new
objects/arrays for every state change.

1.4 Input Bridge Elimination
------------------------------

The four bridge components exist solely because class components cannot call hooks:

  DetailInputBridge    -> useInput() directly in functional DetailView
  HomeViewInputBridge  -> useInput() directly in functional HomeView
  GroupPickerInputBridge -> useInput() directly in functional GroupPicker
  IssueListInputBridge -> useInput() directly in functional IssueListPicker

Each bridge receives an onKey callback and renders null. In the rewrite, these
disappear entirely. The parent functional component calls useInput((input, key) => { ... })
directly.

2.1 Home View
--------------

UI ELEMENTS:
  - Status filter tabs: horizontal row with "All" tab + one tab per IssueStatus
    enum value (Active, Awaiting, Blocked, Deferred, Resolved). Each tab shows
    the label and count in parentheses, e.g., "Active (3)". The active tab renders
    with bold + inverse styling. Tabs are separated by gap={2}.
  - Issue list: vertical list of issues filtered by the selected tab. Each row:
    - Column 1: Cursor indicator -- filled triangle (U+25B8) on selected row,
      3 spaces on unselected rows
    - Column 2: Issue number formatted as "I-<inum>" padded to 6 characters
    - Column 3: 2-space gap
    - Column 4: Title truncated to 40 characters (37 + "..." if over 40),
      padded to 40 characters
    - Column 5: 1-space gap
    - Column 6: Unread indicator -- yellow star (U+2731) if issue inum is in
      unreadInums set, space if not
    - Column 7: 2-space gap
    - Column 8: Status label padded to 10 characters, colored per statusToColor()
    - Selected rows: cyan color + bold on columns 1, 2, 4
  - Capacity warning: "Agents full (N/M)" in red+bold when active count >= maxAgents
  - Empty state: "No issues in this view." in dim text

KEYBOARD INTERACTIONS:
  - Down arrow or 'j': move cursor down by 1, clamped to listLength - 1
  - Up arrow or 'k': move cursor up by 1, clamped to 0
  - Tab: cycle status filter forward. Sequence: All -> Active -> Awaiting ->
    Blocked -> Deferred -> Resolved -> All. Resets cursor to 0.
  - Shift+Tab: cycle status filter backward. Same sequence reversed.
  - Return: call onSelect(inum) with the inum of the issue at clamped cursor position
  - 'a': call onActivate(inum) with the selected issue's inum
  - 'd': call onDefer(inum) with the selected issue's inum
  - 'r': call onResolve(inum) with the selected issue's inum
  - 'n': call onNewIssue()
  - Fallthrough to global key handler (Esc, q, s, b, g)

STATE TRANSITIONS:
  - useStatusFilter: boolean, starts false (showing All)
  - statusFilter: IssueStatus enum value, starts Active
  - cursor: number, starts 0, resets to 0 on tab change

DATA DEPENDENCIES:
  - issues: Issue[] (full list)
  - unreadInums: Set<number>
  - maxAgents: number


2.2 Detail View (MOST COMPLEX)
-------------------------------

UI ELEMENTS:

      Line 3: "Blocked by: <list>  |  Blocks: <list>" where each field is

    - Thread parent ResponseContainer: rendered with isThreadParent=true (gray styling),
      height constrained to its computed line count minus 1, overflow hidden
    - Replies separator: centered "[ Replies ]" with dash fill, gray text

  Conversation area:
    - ResponseChain component showing the current message chain (main chain or
      thread reply chain depending on threadStack state)
    - Height = rows - HEADER_LINES(3) - contextHeaderLines - INPUT_AREA_LINES(3) - footerLines
    - Minimum height: 1 line

  Input box:
    - 3 lines: separator (dashes), text input prompt, padding
    - "Enter response: > " prompt with cyan color
    - Uses ink-text-input component
    - Receives focus when no header field, view-replies, or footer item is focused

  GroupPicker overlay (when overlay === OverlayType.Group):
    - Replaces entire content area
    - Centered vertically and horizontally
    - Bordered box with single border style
    - First row: "Create new group" with green "+" prefix
    - Subsequent rows: container names, current group marked with yellow color and bullet
    - Create mode: "Create new group" title, "Name: " text input, hint text

KEYBOARD INTERACTIONS:
  Priority order (first match wins, overlay blocks all others when active):
  1. If overlay is active: all keys handled by overlay component, not DetailView
  2. Alt+H (or U+02D9): call onHome(selectedMessage) -- navigate to home
  3. Escape: call exitThread(). If no thread, calls onBack(selectedMessage)
  4. Ctrl+Shift+Right: call enterThread() on selected message (no-op if no replies)
  5. Ctrl+Shift+Left: call exitThread()
  6. Ctrl+R: call resolveIssue() -- toggles issue between Active/Resolved
     NOTE: Ctrl+R must NOT echo 'r' into the text input. The current code uses
     a suppressNextInput flag. The rewrite should prevent this by checking for
     modifier keys in the input handler before passing to TextInput.
  7. Shift+Left: navigate to previous issue (replaceCurrentView, bounds-checked)
  8. Shift+Right: navigate to next issue (replaceCurrentView, bounds-checked)
  9. Tab: cycle focus forward through the focus ring
  10. Shift+Tab: cycle focus backward through the focus ring
  11. Return + focusedViewReplies: call resolveThread() -- toggles thread_resolved_at
  12. Return + focusedField !== null: open overlay for that field
  13. Return + focusedFooterIndex !== null: dispatch the footer action
  14. Up arrow: decrement selectedMessage (clamped to 0), clear viewReplies focus
  15. Down arrow: increment selectedMessage (clamped to messages.length - 1),
      clear viewReplies focus

  GroupPicker keyboard (when overlay is active):
    - In list mode: j/k/Up/Down navigate, Enter selects or enters create mode,
      Esc closes overlay
    - In create mode: TextInput handles typing, Enter submits if non-empty,
      Esc cancels back to list mode

STATE TRANSITIONS:
  - inputValue: string, initially ""
  - focusedField: number | null (0=Group, 1=BlockedBy, 2=Blocks), initially null
  - focusedFooterIndex: number | null, initially null
  - focusedViewReplies: boolean, initially false
  - overlay: OverlayType enum (None, Group), initially None
  - blockedBySet: Set<number>, initialized from props.blockedBy
  - blocksSet: Set<number>, initialized from props.blocks
  - selectedMessage: number, initialized to last message index (or initialSelectedMessage)
  - threadStack: ThreadStackEntry[], initially empty
  - All focus state resets on thread enter/exit

DATA DEPENDENCIES:
  - inum: number
  - issue: Issue
  - rootResponse: Response | null
  - blockedBy: number[]
  - blocks: number[]
  - group: string
  - columns: number, rows: number (terminal dimensions)
  - containers: Container[] (for GroupPicker)
  - allIssues: Issue[] (for adjacent navigation and IssueListPicker)
  - unreadInums: Set<number>
  - userLastViewedAt: string | null (for "New Replies" separator)

================================================================================
3. CROSS-CUTTING CONCERNS
================================================================================

3.1 Navigation System
----------------------

The view stack is an ordered array of View objects. The stack always has at least
one entry (the root view, typically Home).

Operations:
  - PUSH_VIEW(view): append view to stack
  - POP_VIEW: remove last entry if stack.length > 1
  - REPLACE_VIEW(view): replace the last entry (used for Shift+Left/Right navigation)
  - GO_HOME: reset stack to [{ type: ViewType.Home }]

Screen clearing: The current code calls process.stdout.write('\x1B[2J\x1B[H') on
every navigation. The rewrite should avoid raw ANSI codes. Consider using Ink's
clearScreen or simply relying on the component tree changing to cause a re-render.
If screen artifacts persist, a single clear utility wrapped in a useEffect may
be acceptable.

Footer focus index resets to null on every navigation transition.
Saved selections persist across navigations via useRef.

3.2 Focus Management
----------------------

The current DetailView implements a manual focus ring with index arithmetic in
cycleFocus(). The ring includes:

  Slot 0: Input box (text entry)
  Slots 1-3: Header fields (Group, BlockedBy, Blocks) -- only when NOT in thread
  Slot N: "View replies" button -- only when selected message has replies
  Slots N+1...: Focusable footer shortcuts

The ring is dynamic: its size changes based on thread mode and selected message.
Ink's built-in useFocus/useFocusManager manages focus between components, but
the DetailView focus ring includes non-component targets (header field indices,
a conceptual "view replies" button). The rewrite has two options:

  Option A: Use Ink's useFocus for each focusable element and useFocusManager.focusNext/
    focusPrevious for Tab cycling. This requires making each header field and the
    "view replies" button actual focusable components.

  Option B: Keep a custom useFocusRing hook with manual index math (cleaner version
    of current cycleFocus). This is simpler and avoids potential conflicts with
    Ink's global focus system.

Recommendation: Option B with a clean hook interface. The current focus ring logic
is correct but tangled with class state. A custom hook isolates it cleanly.

3.3 Keyboard Handling Architecture
------------------------------------

Three tiers of keyboard handling:

  Tier 1 -- Global shortcuts: useGlobalKeys() hook consumed by views that want
    global navigation (Esc/q/s/b/g/n). Returns boolean. Called from HomeView,
    AgentStatus, BlockingMap, GroupView. NOT called from DetailView (which handles
    Esc and navigation itself).

  Tier 2 -- View-specific shortcuts: Each view calls useInput() with its own
    handler. The handler processes view-specific keys and optionally falls through
    to global keys.

  Tier 3 -- Overlay/modal keys: When an overlay (GroupPicker) or modal
    (IssueListPicker) is active, its keyboard handler takes priority. The parent
    view's handler should be disabled or guard against overlay state.

Key matching utilities (hotkeys.ts) should be preserved:
  - matchesKey(key, choice): single key check
  - matchesKeyCombination(combo, input, key): combo check
  - Key combination definitions: CTRL_SHIFT_LEFT/RIGHT, CTRL_R, ALT_H, SCROLL_UP_DOWN
  - ALT_H must also match '\u02D9' (terminal encoding variant)

3.4 Data Layer
---------------

The rewrite must address how mock data flows through the component tree.

Current problems:
  - Several components import mock data directly at module level (GroupView imports
    MOCK_CONTAINERS, MOCK_CONTAINER_ISSUES; BlockingMap imports MOCK_ISSUES, MOCK_DEPS;
    AgentStatus defines MOCK_AGENTS inline).
  - Data is mutated in place and forceUpdate() triggers re-renders.
  - No clear separation between data source and UI state.

Rewrite approach:
  - All data flows through props or context. No direct mock data imports in view
    components (except in the AppStateProvider which loads/holds the data).
  - Mutations produce new objects. For example, toggling a dependency:
    WRONG:  arr.splice(idx, 1); forceUpdate();
    RIGHT:  setBlockedBy(prev => prev.filter(n => n !== inum));
  - The mock store remains as the persistence layer. AppStateProvider loads from
    mock-store on mount and calls saveMockData() on changes if auto-save is enabled.
  - getNextResponseId() continues to provide unique IDs.

SECURITY NOTE: The mock-data system reads/writes files from a predictable path
(data/mock-data.json) using synchronous fs operations. This is acceptable for a
development tool but:
  - The JSON path should not be user-controllable (it isn't -- path.resolve from
    import.meta.dirname is safe)
  - File permissions should be checked (owner-read/write only if the data contains
    sensitive review content)
  - The resetMockData() function copies a default file over the active file. If
    the default file were tampered with, it would propagate. Low risk for a dev tool.

3.5 Theming and Styling
-------------------------

Current duplication:
  - statusToColor() is the single source for status colors -- good
  - Author colors (cyan for User, green for Agent) are hardcoded in
    ResponseContainer -- should be centralized
  - Selected item color (cyan) is hardcoded in multiple views -- should be a token
  - Thread parent colors (gray borders, #aaaaaa body) are hardcoded -- should be tokens
  - Various unicode characters (triangles, circles, dashes) are scattered --
    should be named constants

  colors.selected = 'cyan'
  colors.selectedBold = true

  colors.threadParentBody = '#aaaaaa'

  chars.cursor = '\u25B8'  // filled triangle

3.6 Error Handling
-------------------

Current gaps:
  - No error boundaries anywhere in the component tree
  - DetailView silently shows "Issue I-N not found" in red text for missing data
  - No try/catch around mock data file operations in the render path (file reads
    happen at module load time, not during render, so this is acceptable)

Rewrite requirements:
  - Add a top-level error boundary component wrapping the ViewRouter. On error,

Memoization priorities:
  - ResponseContainer should be React.memo'd with a custom comparator checking

4. INK-SPECIFIC IMPLEMENTATION REQUIREMENTS
================================================================================

4.1 Box/Text Composition
--------------------------

Strict rules from Ink:
  - ALL text must be inside <Text>. Raw strings outside <Text> will throw.
  - <Box> CANNOT be nested inside <Text>. Box is for layout; Text is for content.
  - Use <Text wrap="truncate"> for single-line content that might overflow.
  - Use <Box overflow="hidden"> for fixed-height content areas.

Current violations to fix:
  - None identified in the current code (it follows this rule already), but
    the rewrite must maintain this discipline.

Composition patterns:
  - Horizontal layout: <Box flexDirection="row"> with <Text> children
  - Vertical layout: <Box flexDirection="column"> with mixed children
  - Bordered areas: <Box borderStyle="single"> (used in GroupPicker, IssueListPicker)
  - Response containers: manual border characters (not Ink borders) because they
    need custom styling per segment. This is correct and should be preserved.

4.2 useFocus Integration
--------------------------

As discussed in 3.2, the DetailView focus ring is complex:
  - Input box focus is managed by passing a `focus` prop to ink-text-input
  - Header fields are not separate components -- they're inline <Text> elements
    with conditional inverse styling
  - Footer shortcuts have focusable items identified by action property

If using Ink's useFocus:
  - Each focusable element becomes a component calling useFocus()
  - useFocusManager().focusNext() / focusPrevious() handle Tab cycling
  - The id parameter of useFocus could identify each slot
  - Limitation: useFocus registers globally, so overlapping focus systems
    (DetailView vs. GroupPicker) need careful management

If using custom hook:
  - Single source of truth for focus state
  - No registration complexity
  - Explicit slot numbering with dynamic slot count

4.3 useInput Patterns
-----------------------

Each view calls useInput once with a handler that:
  1. Checks overlay/modal state first (return early if active)
  2. Checks key combinations (modifier keys first)
  3. Checks single special keys (Escape, Return, Tab, arrows)
  4. Checks character input ('j', 'k', 'a', 'd', etc.)
  5. Falls through to global key handler if nothing matched

Important: useInput handlers fire for EVERY keystroke. The handler must be
idempotent and not produce side effects unless a match occurs. Guard clauses
at the top prevent processing during overlay/modal states.

The `isActive` parameter of useInput should be used to disable input handling
when a view is not the active view or when an overlay is shown. Example:
  useInput(handler, { isActive: overlay === OverlayType.None });

4.4 Responsive Layout
-----------------------

Terminal dimensions come from useStdout() and resize events. The AppWrapper
already handles this correctly with useState + useEffect + resize listener.

Components that need dimensions:
  - App shell: constrains total height to rows
  - DetailView: computes conversation height from rows/columns
  - ResponseContainer: word-wraps body text at columns - 4
  - Header: fills horizontal rule to column width
  - Footer: wraps shortcuts into rows based on column width
  - IssueHeader: fills separator to column width

Width should flow through props (not context) because it changes on resize and
most components need it for layout calculations.

4.5 Border and Spacing Conventions
------------------------------------

Response containers use manual Unicode box-drawing:
  - Top-left: U+250C (light down and right)
  - Top-right: U+2510 (light down and left)
  - Bottom-left: U+2514 (light up and right)
  - Bottom-right: U+2518 (light up and left)
  - Horizontal: U+2500 (light horizontal)
  - Vertical: U+2502 (light vertical)
  - Inner width: columns - 4 (2 chars for left border+space, 2 for space+right border)

Overlays/pickers use Ink's borderStyle="single" for consistency.

Spacing: gap={2} between status tabs, marginBottom={1} for section headers,
paddingLeft/Right={1} for bordered boxes.

6. SECURITY-SPECIFIC OBSERVATIONS
================================================================================

While this is a local developer tool (not network-facing), security hygiene
matters for code quality and future-proofing:

6.1 Input Validation
  - The NewIssue form validates blocked-by format. Good. Ensure the rewrite
    preserves this validation and does not allow injection of arbitrary strings
    into data structures.
  - Text input in the InputBox and GroupPicker should sanitize or escape control
    characters before storing them in mock data JSON.

6.2 File System Operations
  - mock-store.ts uses path.resolve(import.meta.dirname, ...) which is safe against
    path traversal.
  - fs.readFileSync and fs.writeFileSync are synchronous and block the event loop.
    For a TUI this is acceptable (happens at startup/save), but should not happen
    during render cycles.
  - The --resetMockData flag is processed before app init (correct order).

6.3 Process Operations
  - process.stdout.write() with ANSI codes is used for screen clearing. The rewrite
    should minimize raw terminal control sequences.
  - process.argv is accessed for --resetMockData. No risk of injection since it's
    a simple includes() check.

6.4 Dependency Security
  - ink, ink-text-input, and react are the core dependencies. These should be
    pinned to known-good versions in package.json.
  - No network calls are made from the TUI layer (all data is local mock or DB).
  - No eval(), Function(), or dynamic code execution.

6.5 State Integrity
  - The current code mutates shared state objects directly (e.g., issue.status = ...).
    Multiple components holding references to the same object can see stale or
    inconsistent state. The rewrite's immutable update pattern eliminates this
    entire class of bugs.
  - Thread resolution mutates thread_resolved_at on Response objects in the linked
    list. With immutable updates, this requires creating new Response objects and
    relinking the chain, or using an external resolution state map keyed by
    response ID.
