SECURITY AUDITOR ANALYSIS: Worktree Test Harness Architecture Decisions
========================================================================
Reviewer: Security Auditor (Claude Opus 4.6)
Scope: test-worktree-scripts.sh, spawnBGClaudeWT.sh, worktreeDaemon.sh, worktreeLib.sh


POINT 1: pkill -f "worktreeDaemon.sh" kills ALL daemon processes system-wide
--------------------------------------------------------------------------

Verdict: REAL CONCERN
Severity: HIGH

The full_cleanup() function at line 216 of test-worktree-scripts.sh runs:

    pkill -f "worktreeDaemon.sh" 2>/dev/null || true

This matches any process whose command line contains the string "worktreeDaemon.sh",
which includes:

  - Daemons started by real, non-test worktrees the developer is actively using
  - Daemons running in other repositories on the same machine
  - Even unrelated processes that happen to have that string in their arguments

This is a textbook "blast radius" problem. A test harness should never have
side effects on production processes.

Recommended fix:
  (a) Track the PID of any daemon the test harness itself starts (the test already
      knows PID_FILE). Kill only that specific PID.
  (b) If the concern is orphaned processes the test did not track, use a process
      group: start the daemon in its own process group via setsid, record the PGID,
      and kill the group on cleanup.
  (c) As a last-resort fallback, if pkill must remain, scope it:

        pkill -f "worktreeDaemon.sh ${REPO_ROOT}"

      This still is not bulletproof (another test run against the same repo root),
      but it narrows the match significantly. The PID-based approach in (a) is
      strongly preferred.

The existing code already reads PID_FILE and kills the specific PID at lines 205-211.
The pkill at line 216 is the "stray" fallback. The fix is to remove the pkill entirely,
or gate it behind a confirmation prompt with a warning that it is system-wide.


POINT 2: Tests B7 and B10 merge into the user's ACTUAL working branch, then git reset --hard
---------------------------------------------------------------------------------------------

Verdict: REAL CONCERN
Severity: CRITICAL

This is the single most dangerous pattern in the test suite. Here is what happens:

  test_B7, line 634:
    git merge --no-ff "${TEST_WT_NAME}" -m "test merge for B7"

  test_B7, line 674:
    git reset --hard HEAD~1

  test_B10, lines 698 and 741: same pattern.

If the script is interrupted between the merge and the reset (Ctrl-C, terminal
crash, power loss, OOM kill), the user's working branch (tui-rewrite) now
contains a spurious merge commit. Worse, if the user has unstaged or staged
changes at the time the test runs, the git reset --hard destroys them
irrecoverably. There is no stash, no reflog warning, nothing.

This is not a theoretical risk. Interactive test suites invite Ctrl-C between
steps -- the script even has "Press Enter to continue" prompts.

Additionally, git reset --hard HEAD~1 assumes the merge was exactly one commit
ago. If the daemon's cleanup creates additional commits or if the user
accidentally runs a git command in another terminal between merge and reset,
HEAD~1 points to the wrong commit and the reset destroys real work.

Recommended fix:
  (a) Create a disposable branch for the merge target:

        throwaway="__test_merge_target_$(date +%s)"
        git checkout -b "${throwaway}"
        git merge --no-ff "${TEST_WT_NAME}" -m "test merge for B7"
        # ... daemon assertions ...
        git checkout "${original_branch}"
        git branch -D "${throwaway}"

      This completely isolates the test from the user's real branch. The daemon
      needs the lockfile's parentBranch to point to the throwaway branch, which
      the test already controls.

  (b) If the test MUST use the real branch (to verify real-world daemon behavior),
      record the exact commit hash before the merge:

        pre_merge_sha="$(git rev-parse HEAD)"
        git merge ...
        # ... assertions ...
        git reset --hard "${pre_merge_sha}"

      This is safer than HEAD~1 because it is immune to intervening commits.

  (c) In either case, the cleanup trap (see Point 5) should include the reset
      so that an interrupted test still restores the branch.

Approach (a) is strongly preferred. The daemon does not care whether the parent
branch is tui-rewrite or __test_merge_target -- it reads the name from the
lockfile. There is no loss of test fidelity.


POINT 3: Fixed sleep durations vs. polling with timeout
-------------------------------------------------------

Verdict: REAL CONCERN
Severity: MEDIUM

Multiple tests use "sleep 15" or "sleep 20" to wait for the daemon's 10-second
scan interval. The problems:

  - If the daemon scans immediately after the sleep starts, the test wastes
    14-19 seconds per test case. Across 19 tests, this adds minutes of dead time.
  - If the system is under load and the daemon's scan takes longer than expected
    (node startup latency, I/O contention), the fixed sleep may be too short,
    causing flaky failures.
  - The 10-second SCAN_INTERVAL in the daemon is a magic number that the test
    hardcodes knowledge of. If someone changes the daemon interval, every test
    sleep must be updated manually.

Recommended fix: Poll for the expected state with an exponential-backoff timeout.

    wait_for_condition() {
      local description="$1"
      local check_cmd="$2"
      local timeout="${3:-30}"
      local interval=1
      local elapsed=0
      while [ "${elapsed}" -lt "${timeout}" ]; do
        if eval "${check_cmd}"; then
          return 0
        fi
        sleep "${interval}"
        elapsed=$((elapsed + interval))
        # Optionally increase interval: interval=$((interval * 2))
      done
      fail "${description} (timed out after ${timeout}s)"
      return 1
    }

Usage example for B7:

    wait_for_condition \
      "daemon detects merge" \
      "tail -n +$((log_lines_before + 1)) '${LOCK_DIR}/daemon.log' | grep -q 'cleanup succeeded'" \
      30

This is faster when the daemon acts quickly, more resilient under load, and
decouples the test from the daemon's internal timing.

Downside: polling adds complexity and can mask "the daemon did this too fast"
type timing bugs. For this use case, the tradeoff clearly favors polling.


POINT 4: Daemon lifecycle bug -- B7 kills the daemon, B10 never restarts it
---------------------------------------------------------------------------

Verdict: REAL CONCERN (confirmed bug)
Severity: HIGH

The daemon exits when it finds zero lockfiles (worktreeDaemon.sh line 56-59):

    if [ ${#lockfiles[@]} -eq 0 ] || [ ! -e "${lockfiles[0]}" ]; then
      log "No lockfiles remain. Daemon exiting."

After B7 cleans up test_fresh_1 (the last remaining lockfile), the daemon exits.
B10 then creates a new worktree and lockfile but never calls ensure_daemon or
starts a new daemon. B10's assertions about the daemon detecting the merge and
running cleanup will fail because the daemon is dead.

The suggested test order is:
  A1 A2 B1 A3 B9 B2 B4 B5 B6 B8 B7 B10 A4 A5 A6 B3

The order is not "fundamentally flawed" in concept, but it has this specific
lifecycle gap between B7 and B10.

Recommended fix:
  (a) At the start of test_B10, after creating the worktree and lockfile, add:

        ensure_daemon  (or the equivalent inline logic)

      This is the minimal fix. The daemon is meant to be started by
      spawnBGClaudeWT.sh, but B10 bypasses the spawn script and creates the
      worktree manually. Manual creation must also manually ensure the daemon.

  (b) Better: B10 should use spawnBGClaudeWT.sh to create the worktree (like
      A1 does), which calls ensure_daemon internally. This also tests the full
      integration path rather than a hand-rolled partial setup.

  (c) Even better: Add a helper function ensure_daemon_for_test() that the test
      harness calls before any assertion that depends on daemon behavior. This
      makes the dependency explicit and prevents future similar bugs.

There is also a secondary concern: the daemon's "exit when no lockfiles" behavior
means that between B5 (phantom lockfile cleanup) and B8, the daemon may also exit
if B5's phantom was the only lockfile. The test only survives because A1's
test_fresh_1 lockfile still exists at that point. This fragile implicit dependency
reinforces the case for explicit daemon lifecycle management in the test harness.


POINT 5: No trap for SIGINT/SIGTERM/EXIT in the test script
------------------------------------------------------------

Verdict: REAL CONCERN
Severity: HIGH

The production daemon (worktreeDaemon.sh) has a trap:

    trap cleanup_daemon SIGINT SIGTERM

But the test harness (test-worktree-scripts.sh) has no trap at all. If the user
presses Ctrl-C during test execution, the following artifacts are orphaned:

  - Test worktrees (test_fresh_1, test_fresh_2, phantom_wt, orphan_parent_test)
  - Test branches in the git repository
  - Lockfiles in .worktree-locks/
  - The daemon process
  - tmux sessions
  - A merge commit on the user's working branch (if interrupted during B7/B10)

The user must then manually discover and clean up each of these, or re-run the
script with the "full cleanup" option and hope it catches everything.

Recommended fix: Add a trap at the top of the test script:

    cleanup_on_exit() {
      echo ""
      echo "Test interrupted. Cleaning up..."

      # Undo any in-progress merge (safe: no-op if HEAD is not a merge)
      if [ -n "${PRE_MERGE_SHA:-}" ]; then
        git reset --hard "${PRE_MERGE_SHA}" 2>/dev/null || true
      fi

      full_cleanup
    }

    trap cleanup_on_exit EXIT

Concerns about traps:
  - Cleanup running on normal exit: This is actually desirable. Use a flag
    variable (CLEANUP_DONE) to prevent double-cleanup.
  - Cleanup masking the real failure: Print the failure information BEFORE
    cleanup. Or capture it in a variable and print after cleanup.
  - Cleanup itself failing: Use || true liberally (which full_cleanup already
    does). A partial cleanup is better than no cleanup.

The trap should handle EXIT (which covers normal exit, Ctrl-C, and most
signals) rather than individual signals, to ensure maximum coverage.

One subtlety: with set -e (which the test script does NOT use, but the
production scripts do), a trap on ERR can interfere. Since test-worktree-scripts.sh
uses set -uo pipefail without -e, this is not a concern here.


POINT 6: Destructive cleanup with insufficient guardrails
---------------------------------------------------------

Verdict: REAL CONCERN
Severity: HIGH

full_cleanup() performs:

    git worktree remove --force "${wt_path}" || rm -rf "${wt_path}"
    git branch -D "${wt}" || true
    pkill -f "worktreeDaemon.sh" || true

These operations depend entirely on the variables TEST_WT_NAME and TEST_WT_NAME2
resolving correctly. If these variables were ever empty or contained unexpected
values:

  - "rm -rf" with an empty variable becomes "rm -rf /absolute/path/../" which
    could traverse upward. In this script, wt_path is constructed as
    "${REPO_ROOT}/../${wt}" -- if wt is empty, this becomes the parent of
    REPO_ROOT, which is the user's workspace directory.

  - "git branch -D" with the wrong name deletes a real branch.

The current code does mitigate this partially: TEST_WT_NAME and TEST_WT_NAME2
are hardcoded constants ("test_fresh_1" and "test_fresh_2"), not derived from
user input. This reduces the probability of variable resolution failures
significantly.

However, defense-in-depth demands explicit guardrails.

Recommended fix:
  (a) Prefix validation before destructive operations:

        for wt in "${TEST_WT_NAME}" "${TEST_WT_NAME2}"; do
          # Guard: only delete branches/worktrees matching the test prefix
          if [[ "${wt}" != test_* ]]; then
            echo "SAFETY: refusing to delete non-test artifact: ${wt}"
            continue
          fi
          ...
        done

  (b) Before rm -rf, validate the path is non-empty and within the expected
      parent directory:

        if [ -z "${wt_path}" ] || [[ "${wt_path}" != */test_* ]]; then
          echo "SAFETY: refusing to rm -rf suspicious path: '${wt_path}'"
          continue
        fi

  (c) Use git worktree remove without --force first. Only escalate to --force
      if the non-force removal fails AND the path matches the expected pattern.
      Never fall through to rm -rf without the path validation in (b).


POINT 7: Test isolation -- shared state creates implicit ordering dependencies
------------------------------------------------------------------------------

Verdict: REAL CONCERN
Severity: MEDIUM

The dependency graph is:

    A1 --> A2 (needs worktree from A1)
    A1 --> A3 (needs worktree from A1)
    A1 --> B1 (needs daemon from A1)
    A1 --> B8 (needs worktree from A1)
    A1 --> B7 (needs worktree from A1)
    B7 --> B10 (implicit: daemon lifecycle, see Point 4)
    A5 --> A6 (needs worktree from A5)

For a fully automated test suite, this would be unacceptable. Shared state makes
tests order-dependent, prevents parallel execution, and means a failure in an
early test cascades as false failures through all dependents.

However, this is explicitly a MANUAL test suite with an interactive runner that
prompts between each test. In this context:

  - The user is expected to run tests in the suggested order.
  - The interactive prompts provide natural breakpoints to inspect state.
  - The test descriptions document prerequisites ("run A1 first").
  - Full isolation would require creating and destroying worktrees for every
    single test, adding substantial setup time (git worktree creation, scaffolding
    copy, daemon startup) to each test.

For a manual test suite, shared state with documented ordering is an acceptable
tradeoff. The test runner already enforces the suggested order by default.

Recommended improvements (not a full rewrite):
  (a) Add precondition checks at the start of each dependent test:

        test_A2() {
          if [ ! -d "${TEST_WT_PATH}" ]; then
            skip "A2 requires A1 to have run first"
            return
          fi
          ...
        }

      Several tests already do this (B8 line 580), but the pattern should be
      consistent across all tests.

  (b) For the single-test execution mode (bash test-worktree-scripts.sh B7),
      consider auto-running prerequisites. This is complex but would make ad-hoc
      testing much more practical.


POINT 8: eval injection in worktreeDaemon.sh (PRODUCTION CODE BUG)
------------------------------------------------------------------

Verdict: REAL CONCERN
Severity: CRITICAL

This is not a test harness issue -- this is a code injection vulnerability in
production code. It is the most serious finding in this review.

worktreeDaemon.sh lines 66-77:

    node_output="$(node -e "
      const f = require('fs').readFileSync('${lockfile}', 'utf8');
      const d = JSON.parse(f);
      console.log('WORKTREE_NAME=' + JSON.stringify(d.worktreeName));
      console.log('WORKTREE_PATH=' + JSON.stringify(d.worktreePath));
      console.log('REPO_ROOT=' + JSON.stringify(d.repoRoot));
      console.log('PARENT_BRANCH=' + JSON.stringify(d.parentBranch));
    " 2>/dev/null)" || {
      ...
    }
    eval "${node_output}"

The node script wraps each value in JSON.stringify(), which produces
double-quoted strings. The eval then processes these as bash assignments.

Attack vector analysis:

  1. The node side is actually reasonably safe for simple payloads because
     JSON.stringify() escapes double quotes and backslashes. A value like:
       {"worktreeName": "foo\"bar"}
     becomes:
       WORKTREE_NAME="foo\"bar"
     which bash handles correctly within the double-quoted eval context.

  2. However, JSON.stringify does NOT escape the following characters that
     are significant in bash double-quoted strings:
       $  - enables command substitution: $(cmd) or ${var}
       `  - enables legacy command substitution: `cmd`
       !  - enables history expansion (in interactive shells)

     A lockfile containing:
       {"worktreeName": "$(rm -rf /tmp/important)"}

     becomes node output:
       WORKTREE_NAME="$(rm -rf /tmp/important)"

     When eval runs this, bash executes the command substitution.

  3. The lockfile path itself is also interpolated into the node -e string
     (line 67: readFileSync('${lockfile}', 'utf8')). A lockfile name containing
     a single quote would break out of the node string literal. Since lockfile
     names are derived from worktree names (which are user-provided arguments
     to spawnBGClaudeWT.sh), this is a secondary injection vector in the node
     evaluation itself.

Exploitability assessment:

  In the current workflow, lockfiles are only written by spawnBGClaudeWT.sh,
  which constructs the JSON from controlled variables using node's process.argv
  (not string interpolation). A local attacker who can write to the
  .worktree-locks/ directory can trivially exploit this. Since the directory
  is within the git repository, any process with write access to the repo can
  plant a malicious lockfile.

  The threat model is "local privilege escalation" -- if another user or
  compromised process can write to the lock directory, they can execute
  arbitrary commands as the user running the daemon.

Recommended fix (production code):

  (a) Eliminate eval entirely. Use read to parse the node output:

        node_output="$(node -e "
          const f = require('fs').readFileSync(process.argv[1], 'utf8');
          const d = JSON.parse(f);
          process.stdout.write(d.worktreeName + '\\0');
          process.stdout.write(d.worktreePath + '\\0');
          process.stdout.write(d.repoRoot + '\\0');
          process.stdout.write(d.parentBranch + '\\0');
        " "${lockfile}" 2>/dev/null)" || {
          log "WARNING: Failed to parse ${lockfile}. Skipping."
          continue
        }

        IFS= read -r -d '' WORKTREE_NAME <<< "${node_output}"
        # ... or use mapfile with null delimiter

      Better yet, use null-delimited output and read -d '':

        {
          IFS= read -r -d '' WORKTREE_NAME
          IFS= read -r -d '' WORKTREE_PATH
          IFS= read -r -d '' REPO_ROOT_PARSED
          IFS= read -r -d '' PARENT_BRANCH
        } < <(node -e "
          const d = JSON.parse(require('fs').readFileSync(process.argv[1], 'utf8'));
          process.stdout.write(d.worktreeName + '\0');
          process.stdout.write(d.worktreePath + '\0');
          process.stdout.write(d.repoRoot + '\0');
          process.stdout.write(d.parentBranch + '\0');
        " "${lockfile}" 2>/dev/null) || {
          log "WARNING: Failed to parse ${lockfile}. Skipping."
          continue
        }

  (b) Also fix the lockfile path injection: note the original code passes
      '${lockfile}' inside the node -e string literal. The fix above passes
      it as process.argv[1], which eliminates this vector entirely.

  (c) Add input validation after parsing: verify WORKTREE_NAME matches
      an expected pattern (alphanumeric + underscores), WORKTREE_PATH is
      an absolute path under the expected parent directory, etc.

  (d) The test suite SHOULD include a test case for malicious lockfile content.
      This would be a straightforward addition:

        test_B11_injection() {
          # Write lockfile with command substitution in worktreeName
          echo '{"worktreeName":"$(echo INJECTED > /tmp/worktree_injection_test)","worktreePath":"/tmp/fake","repoRoot":"/tmp","parentBranch":"main","spawnedAt":"2025-01-01"}' > "${LOCK_DIR}/injection.lock"
          sleep 15
          # Assert /tmp/worktree_injection_test does NOT exist
          assert_file_missing "/tmp/worktree_injection_test"
          rm -f "${LOCK_DIR}/injection.lock"
        }


ADDITIONAL FINDINGS (not in the debate prompt)
==============================================

FINDING A: Lockfile path injection in the node -e invocation
------------------------------------------------------------

Severity: MEDIUM

In worktreeDaemon.sh line 67:

    const f = require('fs').readFileSync('${lockfile}', 'utf8');

The lockfile variable is interpolated into a single-quoted JavaScript string
via bash. If the lockfile path contains a single quote (which is unlikely but
possible in filesystem paths), the JavaScript breaks out of the string literal,
enabling arbitrary JavaScript execution within the node -e context.

This is already addressed by the fix in Point 8(b) above (use process.argv).


FINDING B: spawnBGClaudeWT.sh has the same node string interpolation issue
--------------------------------------------------------------------------

Severity: LOW (values are more controlled here, but the pattern is still wrong)

spawnBGClaudeWT.sh lines 51-62 use process.argv for the values, which is the
correct approach. However, the reopen path (line 51) still works correctly.
No issue here -- this is already done right. Noted for completeness.


FINDING C: has_meaningful_changes uses a fragile parsing approach
----------------------------------------------------------------

Severity: LOW

worktreeLib.sh line 21:

    path="${line:3}"

This strips the first 3 characters of git status --porcelain output, which
works for the standard "XY path" format. However, for renamed files, git
status --porcelain outputs "R  old_path -> new_path", and for files with
spaces, the output may include quoting. The current approach would
misidentify renamed files or files with special characters.

In the current use case (checking for .claude/settings.json and
.vscode/tasks.json), this is unlikely to cause problems because those
paths contain no special characters. But it is worth noting as a latent
fragility.


FINDING D: test_C1 executes the subshell twice
-----------------------------------------------

Severity: LOW (test correctness, not security)

test_C1 lines 915-936 run the has_meaningful_changes check twice: once in a
subshell whose output is discarded (lines 915-925), and once in a command
substitution that captures the result (lines 926-936). The first invocation
is dead code. It should be removed to avoid confusion and wasted execution time.


SEVERITY SUMMARY
================

CRITICAL (fix before using in production):
  - Point 8: eval injection in worktreeDaemon.sh (code execution via crafted lockfile)
  - Point 2: git merge/reset on user's working branch (data loss on interrupt)

HIGH (fix before relying on test results):
  - Point 1: pkill kills all daemon processes system-wide (collateral damage)
  - Point 4: Daemon lifecycle gap between B7 and B10 (test will fail)
  - Point 5: No cleanup trap in test harness (orphaned artifacts on Ctrl-C)
  - Point 6: Destructive cleanup without path validation (safety guardrail gap)

MEDIUM (improve when practical):
  - Point 3: Fixed sleep vs polling (flakiness and wasted time)
  - Point 7: Shared test state (acceptable for manual suite, add precondition checks)

LOW (informational):
  - Finding C: Fragile porcelain parsing
  - Finding D: Dead code in test_C1


PRIORITIZED ACTION PLAN
========================

1. Fix the eval injection in worktreeDaemon.sh immediately. This is a code
   execution vulnerability in production code, not just a test issue. Use
   null-delimited output and read, not eval. Pass the lockfile path via
   process.argv, not string interpolation.

2. Add a trap to test-worktree-scripts.sh that calls full_cleanup on EXIT.
   Store the pre-merge SHA before B7/B10 merges and restore it in the trap.

3. Change B7 and B10 to merge into a throwaway branch instead of the user's
   working branch. Update the lockfile's parentBranch to reference the
   throwaway branch.

4. Replace pkill -f "worktreeDaemon.sh" with PID-specific kill using the
   recorded PID file.

5. Add ensure_daemon call at the start of B10 (or use spawnBGClaudeWT.sh).

6. Add path-prefix validation before rm -rf and git branch -D in full_cleanup.

7. Replace fixed sleeps with a poll-with-timeout helper.

8. Add a test case for malicious lockfile content (injection test).
