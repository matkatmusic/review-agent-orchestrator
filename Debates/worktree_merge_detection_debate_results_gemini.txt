- POSITION: 
The most robust programmatic method relies on a tiered approach, prioritizing the Git plumbing command `git merge-base --is-ancestor` for topological merges, followed by patch equivalence checks for rebases, and heuristic checks for squash merges.

- ARGUMENTS: 
`git merge-base --is-ancestor <source> <target>` is the absolute best primary method. 
*   **Plumbing vs Porcelain:** Unlike `git branch --merged` (which requires parsing string output and is meant for human consumption), `merge-base --is-ancestor` is a low-level plumbing command that silently returns an exit code (`0` for true, `1` for false). This makes it highly reliable for scripting.
*   **Reliability:** It traverses the Directed Acyclic Graph (DAG) to definitively prove if the tip commit of the worktree branch is part of the target branch's history. This natively and perfectly handles **Fast-forward merges**, standard **Merge commits**, and **Octopus merges**.
*   **Performance:** Graph traversal for ancestry is heavily optimized in Git, making it extremely fast even in large repositories with thousands of branches, as it doesn't need to load file contents or compute diffs.
*   **Worktree Existence:** This method is entirely agnostic to whether the physical worktree directory still exists. As long as the branch reference (or the commit hash you are checking) still exists in the `.git` directory, the graph traversal works perfectly.

- EDGE CASES: 
Because Git's DAG strictly tracks parent-child commit relationships, ancestry checks fail when commits are rewritten.

1.  **Rebase and Merge:** When a branch is rebased onto the target and then merged, the commit hashes change. Topological ancestry (`merge-base`) will fail.
    *   **How to handle:** Use `git cherry <target> <source>`. This command compares the "patch IDs" (the actual diffs) of the commits. If every commit in the source branch has an equivalent patch applied to the target branch, `git cherry` will output lines starting with `-` (meaning the patch is already present) and no lines starting with `+` (meaning a missing patch).
2.  **Squash Merge:** All commits from the worktree branch are condensed into a single, entirely new commit on the target branch. Both topological ancestry and patch equivalence will fail because the individual patches no longer exist, and the combined patch hash differs from the individual ones.
    *   **How to handle:** This is a fundamental limitation of distributed version control without centralized metadata. You must rely on conventions and heuristics. The most common approach is checking if the target branch's commit messages since the merge-base contain the source branch's name (e.g., standard GitHub/GitLab behavior). If a strict API is available (like the GitHub CLI `gh pr view`), querying the PR status is the only 100% foolproof way to detect a squash merge.
3.  **Deleted Branch:** If the worktree was removed and the branch was subsequently deleted, you can no longer reference `wt1-branch` by name. You must have saved the tip commit hash (SHA-1) beforehand to perform these checks.

- SCRIPT EXAMPLE: 
Here is a bash snippet demonstrating the tiered approach, prioritizing speed and accuracy:

```bash
#!/bin/bash

# Can be a branch name or a raw commit hash
SOURCE="wt1-branch" 
TARGET="B"

# Ensure references exist
if ! git rev-parse --verify "$SOURCE" >/dev/null 2>&1 || ! git rev-parse --verify "$TARGET" >/dev/null 2>&1; then
    echo "Error: One or both references do not exist."
    exit 2
fi

# 1. Check true topological merge (Merge commit, Fast-forward, Octopus)
# Fast, reliable, script-friendly (exit code 0 = true)
if git merge-base --is-ancestor "$SOURCE" "$TARGET"; then
    echo "Status: Merged (Topological)"
    exit 0
fi

# 2. Check Rebase and Merge (Patch equivalence)
# git cherry outputs '+' for missing patches, '-' for applied patches.
# If grep doesn't find any '+', all patches from SOURCE are in TARGET.
if ! git cherry "$TARGET" "$SOURCE" | grep -q '^+'; then
    echo "Status: Merged (Rebased)"
    exit 0
fi

# 3. Check Squash Merge (Heuristic approach)
# Look for the source branch name in the commit history of the target branch 
# that occurred after the branches diverged.
MERGE_BASE=$(git merge-base "$SOURCE" "$TARGET")
if git log "${MERGE_BASE}..${TARGET}" --format="%B" | grep -qi "$SOURCE"; then
    echo "Status: Merged (Likely Squashed - branch name found in commit message)"
    exit 0
fi

echo "Status: Not Merged"
exit 1
```
