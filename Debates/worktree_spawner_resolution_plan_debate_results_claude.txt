ISSUE 1: pgrep substring false positives + self-detection
  Synthesis ref: Critical #1, Consolidated Rec #1

  VERDICT: CONFIRMED. This is a show-stopper.

  pgrep -f "worktreeDaemon.sh.*reponame" will match itself (the pgrep process
  contains the pattern in its own cmdline), match substrings of longer repo names,
  and is fragile across platforms. The daemon cannot reliably start or detect its
  singleton status.

  EXACT CHANGE — worktreeDaemon.sh INIT section:

  Replace:
    # Singleton: pgrep for other daemon instances (no PID file)

  With PID-file + kill -0 approach:

    PIDFILE="${REPO_ROOT}/.worktree-daemon.pid"

    daemon_already_running() {
      if [ -f "${PIDFILE}" ]; then
        local old_pid
        old_pid=$(<"${PIDFILE}")
        if kill -0 "${old_pid}" 2>/dev/null; then
          # Verify it is actually our daemon, not a recycled PID
          if [ -d "/proc/${old_pid}" ] && grep -q "worktreeDaemon" "/proc/${old_pid}/cmdline" 2>/dev/null; then
            return 0
          elif [[ "$OSTYPE" == darwin* ]]; then
            if ps -p "${old_pid}" -o command= 2>/dev/null | grep -q "worktreeDaemon"; then
              return 0
            fi
          fi
        fi
        # Stale PID file — remove it
        rm -f "${PIDFILE}"
      fi
      return 1
    }

    if daemon_already_running; then
      log "Daemon already running (PID $(cat "${PIDFILE}")). Exiting."
      exit 0
    fi

    echo $$ > "${PIDFILE}"

    cleanup_pidfile() {
      rm -f "${PIDFILE}"
    }
    trap 'cleanup_pidfile; exit 0' EXIT SIGINT SIGTERM

  Also add .worktree-daemon.pid to .gitignore.

  EXACT CHANGE — spawnWorktree.sh STEP 6:

  Replace:
    pgrep -f "worktreeDaemon.sh.*$(basename "${REPO_ROOT}")"

  With:
    PIDFILE="${REPO_ROOT}/.worktree-daemon.pid"
    daemon_is_running() {
      [ -f "${PIDFILE}" ] && kill -0 "$(<"${PIDFILE}")" 2>/dev/null
    }
    if ! daemon_is_running; then
      nohup "${REPO_ROOT}/worktreeDaemon.sh" "${REPO_ROOT}" >> "${REPO_ROOT}/.worktree-daemon.log" 2>&1 &
    fi

  For M.1 (CHECK MAIN IDE ALIVE) and M.3.1 (CHECK WORKTREE IDE ALIVE), replace
  pgrep with a function that uses exact-match:

    ide_is_running_for_path() {
      local target_path="$1"
      local canonical
      canonical=$(cd "${target_path}" 2>/dev/null && pwd -P)
      # Use pgrep with full match but filter out our own PID
      pgrep -f "agy" | while read -r pid; do
        [ "${pid}" = "$$" ] && continue
        local cmdline
        if [[ "$OSTYPE" == darwin* ]]; then
          cmdline=$(ps -p "${pid}" -o command= 2>/dev/null)
        else
          cmdline=$(tr '\0' ' ' < "/proc/${pid}/cmdline" 2>/dev/null)
        fi
        if echo "${cmdline}" | grep -qF "${canonical}"; then
          return 0
        fi
      done
      return 1
    }

  Complexity: MEDIUM. Touches daemon init, spawner step 6, and all IDE checks.

ISSUE 5: Race condition — spawn vs daemon on freshly created worktrees
  Synthesis ref: Critical #5

  VERDICT: CONFIRMED. Daemon polls every 5s. A freshly created worktree could be
  scanned before the IDE opens, and if the branch happens to already be reachable
  from another branch (e.g., created from a commit already on master), the merge
  detection could trigger premature cleanup.

  EXACT CHANGE — The ownership marker (Issue 3 below) combined with a grace period
  addresses this. In worktreeDaemon.sh M.3, before checking merge status:

    # Skip worktrees younger than 30 seconds (creation grace period)
    MARKER="${worktree_path}/.managed-worktree"
    if [ -f "${MARKER}" ]; then
      local created_at
      created_at=$(stat -f "%m" "${MARKER}" 2>/dev/null || stat -c "%Y" "${MARKER}" 2>/dev/null)
      local now
      now=$(date +%s)
      if (( now - created_at < 30 )); then
        vlog "Skipping ${worktree_path}: within creation grace period"
        continue
      fi
    fi

  Complexity: SMALL. Stat check on the marker file.


ISSUE 6: Merge detection false positive on branches created at old commits
  Synthesis ref: Critical #6, Consolidated Rec #5

  VERDICT: CONFIRMED. This is the core bug that motivated the rewrite. The current
  is_worktree_merged() scans ALL merge commits ever. If a branch was created from
  a commit that was already merged somewhere, it matches. The fix requires recording
  the branch birth commit and constraining the scan window.

  EXACT CHANGE — Record branch birth in spawnWorktree.sh STEP 4 (after worktree add):

    # Record branch birth commit for merge detection
    BIRTH_COMMIT=$(git -C "${REPO_ROOT}" rev-parse HEAD)
    echo "${BIRTH_COMMIT}" > "${WORKTREE_PATH}/.worktree-birth-commit"

  EXACT CHANGE — Replace is_worktree_merged() in worktreeLib.sh:

    is_worktree_merged() {
      local branch_name="$1"
      local worktree_path="$2"
      local repo_root="${3:-${REPO_ROOT}}"

      # Get current branch tip
      local branch_tip
      branch_tip=$(git -C "${repo_root}" rev-parse "refs/heads/${branch_name}" 2>/dev/null) || return 1

      # Load birth commit for scan window
      local birth_file="${worktree_path}/.worktree-birth-commit"
      local since_commit=""
      if [ -f "${birth_file}" ]; then
        since_commit=$(<"${birth_file}")
      fi

      # Only scan merge commits AFTER branch birth
      local log_args=(--all --merges --format='%H %P')
      if [ -n "${since_commit}" ]; then
        log_args+=(--ancestry-path "${since_commit}..HEAD")
      fi

      # Use awk instead of bash while-read for performance
      git -C "${repo_root}" log "${log_args[@]}" 2>/dev/null | \
        awk -v tip="${branch_tip}" '{
          for (i = 3; i <= NF; i++) {
            if ($i == tip) { exit 0 }
          }
        } END { exit 1 }'
    }

  Note on --ancestry-path: this limits the scan to commits that are descendants of
  the birth commit AND ancestors of HEAD. This dramatically reduces the search space
  and eliminates false positives from pre-existing merge commits.

  ALSO ADD squash merge fallback (Consolidated Rec #5):

    is_worktree_squash_merged() {
      local branch_name="$1"
      local repo_root="${2:-${REPO_ROOT}}"

      # Check if the branch's diff is already contained in any target branch
      # by comparing tree objects after squash
      local branch_tip
      branch_tip=$(git -C "${repo_root}" rev-parse "refs/heads/${branch_name}" 2>/dev/null) || return 1

      local merge_base
      merge_base=$(git -C "${repo_root}" merge-base HEAD "${branch_tip}" 2>/dev/null) || return 1

      # If branch tip is ancestor of HEAD, the changes are incorporated
      if git -C "${repo_root}" merge-base --is-ancestor "${branch_tip}" HEAD 2>/dev/null; then
        # But this could just mean no divergence. Check there are actual commits.
        local ahead
        ahead=$(git -C "${repo_root}" rev-list --count "${merge_base}..${branch_tip}" 2>/dev/null)
        if [ "${ahead:-0}" -gt 0 ]; then
          return 0
        fi
      fi

      # Cherry-pick detection: check if all commits on branch exist on target
      local unmerged
      unmerged=$(git -C "${repo_root}" cherry HEAD "${branch_name}" 2>/dev/null | grep -c '^+' || true)
      if [ "${unmerged}" -eq 0 ]; then
        return 0
      fi

      return 1
    }

  Update daemon M.3.2 to call both:

    if is_worktree_merged "${branch_name}" "${worktree_path}" "${REPO_ROOT}" || \
       is_worktree_squash_merged "${branch_name}" "${REPO_ROOT}"; then
      # proceed to cleanup
    fi

  Complexity: LARGE. Core algorithm rewrite plus new squash detection function.

ISSUE 2: Scaffold files are tracked repo files used as disposable scaffold
  Synthesis ref: Critical #2, Consolidated Rec #2

  VERDICT: CONFIRMED. If .claude/settings.json and .vscode/tasks.json are tracked
  in the worktree's branch, then git status will show them as modified (since they
  were overwritten by scaffold), and git worktree remove --force could discard
  actual user changes. Worse, "discarding" scaffold on merge means running git
  checkout on tracked files, which is semantically wrong.

  EXACT CHANGE — The scaffold files must be untracked. In spawnWorktree.sh STEP 4,
  add them to the worktree's local git exclude:

    scaffold_worktree() {
      local wt_path="$1"
      local wt_name="$2"

      mkdir -p "${wt_path}/.claude" "${wt_path}/.vscode"
      cp "${REPO_ROOT}/v1/templates/settings.json" "${wt_path}/.claude/settings.json"
      sed "s/__NAME__/${wt_name}/g" "${REPO_ROOT}/v1/templates/tasks.json" > "${wt_path}/.vscode/tasks.json"

      # Ensure scaffold files are git-ignored in this worktree
      local exclude_file="${wt_path}/.git/info/exclude"
      # Note: in a worktree, .git is a file pointing to the real gitdir.
      # We need the actual gitdir path.
      local gitdir
      gitdir=$(git -C "${wt_path}" rev-parse --git-dir 2>/dev/null)
      exclude_file="${gitdir}/info/exclude"
      mkdir -p "$(dirname "${exclude_file}")"

      for pattern in ".claude/settings.json" ".vscode/tasks.json" ".managed-worktree" ".worktree-birth-commit" "bootstrap.sh"; do
        if ! grep -qxF "${pattern}" "${exclude_file}" 2>/dev/null; then
          echo "${pattern}" >> "${exclude_file}"
        fi
      done
    }

  This makes the scaffold files invisible to git status in the worktree, so:
  - has_non_scaffold_changes() no longer needs special filtering
  - git worktree remove --force won't see them as dirty
  - The daemon's "discard scaffold" step is just rm, not git checkout

  Complexity: MEDIUM. Changes scaffold_worktree and simplifies has_non_scaffold_changes.

ISSUE 11: git worktree remove --force data loss (TOCTOU race)
  Synthesis ref: High #11, Consolidated Rec #6, #8

  VERDICT: CONFIRMED. Between checking "is clean" and running "remove --force",
  the user could save a file. The --force flag bypasses git's own safety checks.

  EXACT CHANGE — In worktreeDaemon.sh M.3.3, replace the cleanup sequence:

    cleanup_worktree() {
      local worktree_path="$1"
      local branch_name="$2"

      # Step 1: Kill IDE and tmux first
      # (IDE kill is best-effort; user may have already closed it)
      if ide_is_running_for_path "${worktree_path}"; then
        log "WARNING: IDE still open for ${worktree_path}. Skipping cleanup."
        return 1
      fi
      tmux kill-session -t "${branch_name}" 2>/dev/null || true

      # Step 2: Wait briefly for any file writes to flush
      sleep 2

      # Step 3: Re-check for dirty files AFTER killing processes
      # Remove scaffold files first (they are untracked, safe to delete)
      discard_scaffold_files "${worktree_path}"

      if has_non_scaffold_changes "${worktree_path}"; then
        log "WARNING: Dirty files in merged worktree ${worktree_path}. Skipping removal."
        log "  Manual cleanup: git worktree remove ${worktree_path}"
        return 1
      fi

      # Step 4: Re-verify branch tip hasn't moved since merge check
      local current_tip
      current_tip=$(git -C "${REPO_ROOT}" rev-parse "refs/heads/${branch_name}" 2>/dev/null) || {
        log "WARNING: Branch ${branch_name} disappeared during cleanup. Removing worktree path only."
        git worktree remove "${worktree_path}" 2>/dev/null || rm -rf "${worktree_path}"
        git worktree prune
        return 0
      }

      # Step 5: Use non-force remove first, fall back to force only if truly clean
      if ! git worktree remove "${worktree_path}" 2>/dev/null; then
        log "WARNING: git worktree remove failed for ${worktree_path}. May have new changes."
        log "  Manual cleanup: git worktree remove --force ${worktree_path}"
        return 1
      fi

      # Step 6: Delete branch
      git branch -D "${branch_name}" 2>/dev/null || true
      git worktree prune

      log "Cleaned up merged worktree: ${branch_name}"
      return 0
    }

  Key changes:
  - No --force on worktree remove (let git's own safety check be the final guard)
  - Re-verify after killing processes
  - Sleep 2 to let filesystem flush
  - Re-check dirty status
  - Graceful fallback with manual instructions

  Complexity: MEDIUM. Restructures the entire cleanup flow.

ISSUE 20: git worktree list --porcelain parsing edge cases
  Synthesis ref: Medium #20

  VERDICT: CONFIRMED. Paths with spaces or special characters can break naive
  parsing. The --porcelain format is well-defined but needs careful handling.

  EXACT CHANGE — In worktreeLib.sh get_worktrees():

    get_worktrees() {
      local repo_root="${1:-${REPO_ROOT}}"
      local current_path="" current_branch=""

      git -C "${repo_root}" worktree list --porcelain 2>/dev/null | while IFS= read -r line; do
        case "${line}" in
          "worktree "*)
            current_path="${line#worktree }"
            ;;
          "branch "*)
            current_branch="${line#branch refs/heads/}"
            ;;
          "")
            # Empty line = end of record
            if [ -n "${current_path}" ] && [ -n "${current_branch}" ]; then
              # Use null byte as delimiter for paths with spaces
              printf '%s\t%s\n' "${current_path}" "${current_branch}"
            fi
            current_path=""
            current_branch=""
            ;;
        esac
      done
    }

  Callers must use:
    while IFS=$'\t' read -r wt_path wt_branch; do
      ...
    done < <(get_worktrees)

  Complexity: SMALL. Parser rewrite with tab delimiters.

    validate_name() {
      local name="$1"
      if [ -z "${name}" ]; then
        echo "ERROR: Worktree name cannot be empty." >&2
        return 1
      fi
      if ! [[ "${name}" =~ ^[a-zA-Z0-9_-]+$ ]]; then
        echo "ERROR: Worktree name '${name}' contains invalid characters." >&2
        echo "  Allowed: letters, numbers, underscore, hyphen." >&2
        return 1
      fi

        echo "ERROR: Worktree name exceeds 50 characters." >&2
        return 1
      fi
      # Prevent names that collide with git internals
      if [[ "${name}" == "HEAD" || "${name}" == "master" || "${name}" == "main" ]]; then
        echo "ERROR: '${name}' is a reserved branch name." >&2
        return 1
      fi
      return 0
    }

    WORKTREE_NAME="${1// /_}"
