DEBATE TOPIC: Architecture decisions in a bash test harness for git worktree management scripts

You are reviewing test-worktree-scripts.sh — an interactive bash test harness (~1170 lines)
that tests three worktree management scripts: spawnBGClaudeWT.sh, worktreeDaemon.sh, worktreeLib.sh.

The test script creates/destroys git worktrees, branches, daemon processes, and lockfiles.
It runs 19 test cases interactively, automating setup/assertions and prompting for manual IDE checks.

CONTEXT: The scripts manage git worktrees for a development workflow where:
- spawnBGClaudeWT.sh creates worktrees with scaffolding (.claude/settings.json, .vscode/tasks.json)
- worktreeDaemon.sh polls lockfiles every 10 seconds, cleaning up worktrees after branch merge
- worktreeLib.sh provides shared cleanup functions that distinguish "ignored" scaffolding files from real changes

KEY ARCHITECTURE DECISIONS TO DEBATE:

1. SAFETY: The test script uses `pkill -f "worktreeDaemon.sh"` in its cleanup function.
   This kills ALL worktreeDaemon processes system-wide, not just the test's daemon.
   Is this acceptable? What's the safer alternative?

2. SAFETY: Tests B7 and B10 do `git merge --no-ff` on the user's ACTUAL working branch,
   then `git reset --hard HEAD~1` to undo it afterward. If the script is interrupted mid-test,
   the merge remains. Is this an acceptable tradeoff for test fidelity, or should tests
   operate on a throwaway branch?

3. TIMING: Tests wait fixed durations (15-20 seconds) for the daemon's 10-second scan interval.
   Should the test instead poll for expected state with a timeout? What are the tradeoffs
   between fixed sleep and polling in bash test harnesses?

4. DAEMON LIFECYCLE BUG: After test B7 merges and daemon cleans up the last lockfile, the daemon
   exits ("No lockfiles remain"). Test B10 then creates a new worktree+lockfile but never
   restarts the daemon. B10's assertions about daemon behavior will fail because the daemon
   is dead. Is the suggested test order (A1 A2 B1 A3 B9 B2 B4 B5 B6 B8 B7 B10 A4 A5 A6 B3)
   fundamentally flawed, or just needs an ensure_daemon call in B10?

5. NO TRAP ON EXIT: The test script has no `trap` for SIGINT/SIGTERM/EXIT. If the user Ctrl-C's
   mid-test, orphaned worktrees, branches, and daemon processes remain. Should a bash test
   harness always have a cleanup trap? What are the downsides (cleanup running on normal exit,
   double-cleanup, cleanup masking the real failure)?

6. DESTRUCTIVE CLEANUP: `full_cleanup()` uses `git worktree remove --force || rm -rf`,
   `git branch -D`, and `pkill`. These are appropriate for test cleanup but could destroy
   real work if variables resolve incorrectly. Should there be guardrails (e.g., only
   delete branches/worktrees matching a test prefix)?

7. TEST ISOLATION: Tests share state — A1 creates a worktree that A2, A3, B1, B8, B7 all
   depend on. This creates implicit ordering dependencies. Is this acceptable for a manual
   test suite, or should each test be fully self-contained?

8. eval INJECTION IN DAEMON (not test bug, but test doesn't cover it):
   worktreeDaemon.sh line 77 does `eval "${node_output}"` where node_output comes from
   parsing a lockfile. If a lockfile contains crafted JSON like {"worktreeName": "$(rm -rf /)"},
   the eval executes it. Should the test suite include a test case for this?

For each point, state whether you agree it's a real concern, its severity (critical/high/medium/low),
and your recommended fix. If you disagree with a concern, explain why.
