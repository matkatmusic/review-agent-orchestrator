DEBATE RESPONSE: Methods to determine if a git worktree branch has been merged into another branch
Agent: Claude (Security Auditor perspective)

================================================================================
POSITION
================================================================================

My recommended primary method is `git merge-base --is-ancestor`, supplemented
by `git branch --merged` for batch queries and `git cherry` for squash-merge
detection. No single command covers all merge strategies reliably, so a layered
detection approach is essential -- much like defense-in-depth in security.

================================================================================
ARGUMENTS
================================================================================

1. `git merge-base --is-ancestor` (PRIMARY -- true merge and fast-forward)

   Command:
     git merge-base --is-ancestor wt1-branch B

   This is a plumbing-level command. It returns exit code 0 if the tip of
   wt1-branch is an ancestor of B (meaning all commits on wt1-branch are
   reachable from B), and exit code 1 otherwise. It produces no stdout output,
   making it ideal for scripting.

   Why this is best:
   - Plumbing command: stable interface, not subject to UI changes across git
     versions. Porcelain commands like `git branch --merged` wrap this logic
     but add formatting overhead and are harder to parse reliably.
   - O(1)-ish performance: it walks the commit graph using the commit-graph
     file (if present) and stops as soon as ancestry is confirmed or denied.
     It does not enumerate all branches.
   - Exit-code driven: perfect for conditionals in shell scripts. No output
     parsing needed.
   - Works regardless of whether the worktree still exists: it operates on
     refs, not worktree state. As long as the branch ref (wt1-branch) still
     exists in the repo, this works. If the branch was deleted after merge,
     you need the commit SHA instead (see Edge Cases).

   This correctly detects:
   - Standard merge commits (the M1 in the ASCII diagram)
   - Fast-forward merges (wt1-branch tip becomes B tip)
   - Octopus merges (multiple branches merged in one commit)

2. `git branch --merged B` (SECONDARY -- batch detection)

   Command:
     git branch --merged B

   This lists all branches whose tips are ancestors of B. Useful when you want
   to check many worktree branches at once rather than querying one at a time.
   Under the hood it performs the same ancestor check, but outputs branch names
   as a list.

   Trade-off: porcelain command, output format may vary (e.g., leading spaces,
   asterisk for current branch). Requires text parsing. For single-branch
   checks, --is-ancestor is strictly superior.

3. `git cherry` / `git log --cherry-pick` (TERTIARY -- squash merge detection)

   Squash merges are the hard case. When someone does `git merge --squash
   wt1-branch`, git creates a NEW commit on B that contains all the changes
   from wt1-branch but has NO parent link back to wt1-branch. The commit graph
   shows no ancestry relationship. `--is-ancestor` will return exit code 1
   (not merged) even though the content was integrated.

   Command:
     git log --cherry-pick --right-only --no-merges --oneline B...wt1-branch

   This uses git's patch-id mechanism to compare the actual diffs (not commit
   hashes) between the two branches. If wt1-branch was squash-merged, the
   patch-ids of the individual commits on wt1-branch will match the squashed
   commit on B, and --cherry-pick will filter them out. If the output is
   empty, all changes from wt1-branch are present in B.

   The underlying plumbing equivalent is:
     git cherry B wt1-branch

   Which outputs lines prefixed with `-` (already in B, by patch-id) or `+`
   (not in B). If all lines are `-`, the branch is effectively merged.

   Trade-offs:
   - Patch-id comparison is content-based, not graph-based. It can produce
     false positives if someone independently wrote identical changes.
   - It is slower than --is-ancestor because it must compute patch-ids for
     commits on both sides.
   - It does NOT detect rebase-and-merge perfectly if commits were reordered
     or amended during rebase (patch-ids change if the diff changes).

================================================================================
EDGE CASES
================================================================================

1. SQUASH MERGES
   As described above, --is-ancestor fails here. Use git cherry as a fallback.
   Accept that squash detection is heuristic, not definitive.

2. REBASE-AND-MERGE
   If the branch was rebased onto B before merging, the original commits are
   rewritten with new hashes. --is-ancestor works if the rebase was followed
   by a fast-forward merge (the new commits are directly on B). git cherry
   also works if the patch content was preserved. It fails if commits were
   squashed or edited during interactive rebase.

3. BRANCH DELETED AFTER MERGE
   If wt1-branch was deleted after merging, the ref no longer exists. You need
   the commit SHA. If you do not have it, you can search the reflog:
     git reflog show --all | grep wt1-branch
   Or look for merge commits that reference the branch name:
     git log --merges --grep="wt1-branch" B
   From a security/reliability standpoint, the best practice is to record the
   branch tip SHA before deletion (e.g., in a log file or tag).

4. WORKTREE REMOVED BUT BRANCH EXISTS
   No issue. git merge-base operates on the object database and refs, not the
   filesystem worktree. The branch ref persists independently of the worktree
   lifecycle. You can verify with:
     git branch --list wt1-branch
   or:
     git rev-parse --verify wt1-branch

5. FAST-FORWARD MERGES
   --is-ancestor handles this correctly. The branch tip is literally on B's
   history. However, there is no merge commit to grep for, so approaches that
   search for merge commits will miss this.

6. OCTOPUS MERGES
   --is-ancestor handles this correctly. Each merged branch tip is an ancestor
   of the octopus merge commit.

7. RACE CONDITIONS IN AUTOMATION
   If B is being updated concurrently (e.g., CI merging other branches), your
   check could see a stale state. For critical automation, lock the ref or
   check against a specific commit SHA rather than a branch name:
     git merge-base --is-ancestor wt1-branch $(git rev-parse B)
   This pins B to a point-in-time value.

8. SHALLOW CLONES
   In CI environments with shallow clones, the commit graph is truncated.
   --is-ancestor may incorrectly return "not ancestor" because the connecting
   commits were not fetched. Ensure sufficient depth:
     git fetch --deepen=100
   Or better, use full clones for merge detection logic.

================================================================================
SCRIPT EXAMPLE
================================================================================

#!/usr/bin/env bash
set -euo pipefail

# Determine if a branch has been merged into a target branch.
# Handles true merges, fast-forwards, and squash merges.
#
# Usage: is_branch_merged <branch> <target>
# Exit codes: 0 = merged, 1 = not merged, 2 = error

is_branch_merged() {
    local branch="$1"
    local target="$2"

    # Validate that both refs exist
    if ! git rev-parse --verify "$branch" >/dev/null 2>&1; then
        echo "ERROR: branch '$branch' does not exist" >&2
        return 2
    fi
    if ! git rev-parse --verify "$target" >/dev/null 2>&1; then
        echo "ERROR: target '$target' does not exist" >&2
        return 2
    fi

    # Layer 1: Graph-based ancestor check (covers true merge, ff, octopus)
    if git merge-base --is-ancestor "$branch" "$target"; then
        echo "MERGED (ancestor check): $branch is reachable from $target"
        return 0
    fi

    # Layer 2: Patch-id check for squash merges
    # If git cherry shows all commits as '-' (present in target), it was
    # squash-merged or cherry-picked.
    local unmerged_count
    unmerged_count=$(git cherry "$target" "$branch" 2>/dev/null \
                     | grep -c '^+' || true)

    if [ "$unmerged_count" -eq 0 ]; then
        echo "MERGED (squash/cherry-pick): all patches from $branch exist in $target"
        return 0
    fi

    echo "NOT MERGED: $unmerged_count commit(s) from $branch not found in $target"
    return 1
}

# --- Batch check for all worktree branches ---
check_all_worktree_branches() {
    local target="${1:-HEAD}"

    echo "Checking worktree branches against $target..."
    echo "----------------------------------------------"

    # List all worktrees and extract branch names
    git worktree list --porcelain | while read -r line; do
        case "$line" in
            "branch refs/heads/"*)
                local wt_branch="${line#branch refs/heads/}"
                # Skip if this is the target branch itself
                if [ "$wt_branch" = "$target" ]; then
                    continue
                fi
                printf "  %-30s -> " "$wt_branch"
                is_branch_merged "$wt_branch" "$target" 2>/dev/null || true
                ;;
        esac
    done
}

# If run directly (not sourced), execute the single-branch check
if [ "${BASH_SOURCE[0]}" = "$0" ]; then
    if [ $# -lt 2 ]; then
        echo "Usage: $0 <branch> <target>" >&2
        echo "       $0 --all [target]    (check all worktree branches)" >&2
        exit 2
    fi

    if [ "$1" = "--all" ]; then
        check_all_worktree_branches "${2:-HEAD}"
    else
        is_branch_merged "$1" "$2"
    fi
fi

================================================================================
SECURITY AND RELIABILITY CONSIDERATIONS
================================================================================

From a security auditor's standpoint, the following concerns apply when using
merge detection in automated pipelines (CI/CD, deployment gates, cleanup
scripts):

1. REF INTEGRITY: Branch names can be force-pushed, rewritten, or deleted.
   If merge detection gates a deployment or cleanup action, pin to commit SHAs
   and verify signatures where possible:
     git verify-commit $(git rev-parse wt1-branch)

2. TOCTOU (Time-of-Check-Time-of-Use): Between checking merge status and
   acting on it, the branch state can change. For critical operations (like
   deleting a worktree or deploying), re-verify immediately before the action,
   or use atomic operations where git supports them.

3. TRUST BOUNDARY: If branch names come from user input (e.g., a webhook
   payload), sanitize them. A malicious branch name like
   `--upload-pack=malicious-command` could exploit git argument injection.
   Always use `--` to separate options from arguments:
     git merge-base --is-ancestor -- "$branch" "$target"

4. AUDIT TRAIL: Log merge detection results with timestamps and commit SHAs
   for post-incident investigation. Do not rely solely on branch names in logs
   since they are mutable references.

5. FAIL-CLOSED: If the merge detection script encounters an error (exit code
   2 in the script above), the default behavior should be to treat the branch
   as NOT merged. Never assume merge status on error -- that could lead to
   premature worktree cleanup or skipped validation.

================================================================================
SUMMARY OF RECOMMENDED APPROACH
================================================================================

  Check Type          Command                          Detects
  ------------------  -------------------------------  --------------------------
  Primary (graph)     git merge-base --is-ancestor     true merge, ff, octopus
  Secondary (batch)   git branch --merged              same as above, multi-branch
  Tertiary (content)  git cherry / --cherry-pick        squash merge, cherry-pick
  Ref existence       git rev-parse --verify           branch still exists
  Worktree status     git worktree list --porcelain    worktree exists or removed
  Merge commit search git log --merges --grep=         find merge commit by name

Use the layered approach: try --is-ancestor first (fast, definitive for graph
merges), fall back to git cherry for squash detection (slower, heuristic). Log
results and fail closed on errors.
