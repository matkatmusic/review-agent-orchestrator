Architectural Discussion
================================================================================

C++ key event dispatch design, using top-level key press receiver that is the only place that key events arrive at: 
  key comes in.
    is it a tab key? 
      yes -> dispatch to 'cycle focus handler(currentView, key)', end.
      no -> continue
    is it a footer hotkey?
      yes -> dispatch to 'footer hotkey handler(currentView, key)', end.
      no -> continue
    dispatch to 'focused component keypress handler(focusedComponent, key)', end.

  cycle focus handler(currentView, key): 
    get list of focusable components in currentView
    cycle through the list, refreshing the view each time 'tab' is pressed so the focused component is visually differentiated from other tabbable components. 

  footer hotkey handler(currentView, key): 
    get list of footer hotkeys for currentView
    execute action for matching hotkey. 

  focused component keypress handler(focusedComponent, key):
    forward key event to focusedComponent.

  Question: Should TextInput be a tabbable component so that it is included in the focus cycle, and the focus cycle is how we stop specific keyboard input from arriving at the TextInput?
  Answer: Yes. In ink, useInput has an isActive option. When TextInput is not
  the focused element, pass isActive: false and it stops processing keypresses.
  Focus is declarative state, not imperative routing.

  Question: Which keys need to never be treated as meant for the TextInput?
  Answer: Tab, Escape, Ctrl+R, Ctrl+Shift+Arrow, Alt+H, Alt+Q, Ctrl+Alt+Q,
  Ctrl+Alt+R, Shift+Left/Right. In React/Ink these are handled by a useInput
  hook at a higher level that processes them and updates state. TextInput
  ignores them because it only receives input when isActive is true, and
  modifier-key combos are intercepted before TextInput sees them.

React/Ink key event dispatch design (replaces the C++ imperative model above):

  Ink broadcasts every keypress to ALL useInput callbacks simultaneously.
  There is no single dispatcher — components self-subscribe. The design uses
  declarative state (focus, isActive) instead of imperative forwarding.

  Architecture:
    1. Single useInput hook at the view level (e.g., DetailView)
    2. Checks keys in priority order using early return:
       a. Tab/Shift+Tab -> call cycleFocus(), return
       b. Modifier combos (Ctrl+R, Alt+Q, etc.) -> dispatch action, return
       c. Footer hotkeys (single-char like 'q', 'd', 'b') -> dispatch, return
       d. Navigation keys (Up/Down/Escape) -> handled by view state, return
    3. TextInput has its own internal useInput with isActive={inputFocused}
       - When inputFocused is false, TextInput ignores all keypresses
       - When inputFocused is true, only unhandled printable chars reach it
       - Modifier combos (Ctrl+*, Alt+*) are consumed in step 2 before
         TextInput can see them, because the view-level handler runs first
         and the key combinations don't produce printable input characters

  Focus model (declarative, not imperative):
    - focusedSlot: number (managed by useFocusRing hook)
    - Slot 0 = InputBox, Slots 1-3 = header fields, Slot 4+ = footer items
    - Each component reads focusedSlot and renders accordingly:
      const inputFocused = focusedSlot === 0;
      const groupFocused = focusedSlot === 1;
      <TextInput isActive={inputFocused} />
      <HeaderField highlighted={groupFocused} />
    - No key forwarding needed — components observe state, not events

  Key priority (within the single useInput handler):
    Priority 1: Tab         -> cycleFocus(+1), Shift+Tab -> cycleFocus(-1)
    Priority 2: Ctrl+Shift  -> thread navigation (enter/exit)
    Priority 3: Ctrl+R      -> resolve issue/thread
    Priority 4: Alt+Q       -> start quoting
    Priority 5: Escape       -> cancel quote / exit thread / go back
                               (priority: quoting > thread > goBack;
                                see Step 2.2 and Steps 8.7/8.9)
    Priority 6: Up/Down     -> scroll conversation
    Priority 7: Return      -> dispatch based on focusedSlot
                               (send if input, open overlay if header field,
                                trigger action if footer item)
    Priority 8: Single chars -> footer hotkeys (only when input not focused)

  Why this differs from the C++ design:
    - No centralized dispatcher — useInput is per-component, broadcast-based
    - No "forwarding" of keys — each component decides if it should act
    - Focus is state, not routing — components read focusedSlot, not events
    - "Blocking" keys from TextInput is achieved by isActive, not filtering

<Rebuttal>
App owns/shows a ViewBase instance.
All available views are instances of ViewBase.
in C++, ViewBase would be defined like this:
struct ViewBase {
  std::unique_ptr<HeaderBase> header;
  std::unique_ptr<ContentBase> content;
  std::unique_ptr<FooterBase> footer;

  virtual InkComponent render();
};

Every visible object inherits from RenderBase, which defines a single pure virtual function that returns the Component that would be passed into ink's render() function:
struct RenderBase {
  virtual InkComponent getRenderContent() = 0;
  virtual std::vector<TabbableComponents*> getTabbableComponents() = 0
};

HeaderBase as C++:
struct HeaderBase : RenderBase {
  virtual InkComponent getRenderContent(); - returns the <Component /> that gets passed into ink's render() function.
  virtual std::vector<TabbableComponents*> getTabbableComponents() override; - returns the list of tabbable components for the tab focus handler.
};

ContentBase as C++:
struct ContentBase : RenderBase {
public:
  void setNumLinesToShow(int numLinesToShow); // sets the num lines to show when producing the render content.  must be called before getRenderContent();
  InkComponent getRenderContent() override {
    //returns only the visible portion of the content, ContentBase keeps track internally of which part of its view is visible.
    assert(numDisplayLines != 0); //call setNumLinesToShow() first
    createScrolledAndClippedViewContent();
    return getClippedRenderContent();
  }

  //the content has no tabbable components
  std::vector<TabbableComponents*> getTabbableComponents() override { return {}; }
protected:
  virtual void createScrolledAndClippedViewContent() = 0;
  virtual InkComponent getClippedRenderContent() = 0;
};

FooterBase as C++:
struct FooterBase : RenderBase {
  std::vector<TabbableComponents*> getTabbableComponents() override;
  virtual std::vector<HotKeyComponent*> getHotKeyComponents(); //gets all the hot keys for the footer.
  virtual int getNumLinesNeeded(); // computes the number of lines needed to show all hotkeys, uses getHotKeyComponent() to figure out if a component can fit on a line or not.
};

//ViewBase class rendering approach:
InkComponent ViewBase::render() {
  auto headerBlock = header->getRenderContent();
  auto footerBlock = footer->getRenderContent();
  auto contentLinesAvailable = computeLinesAvailable(headerBlock, footerBlock, windowHeight);
  content->setNumLinesToShow(contentLinesAvailable);
  auto content = content->getRenderContent();
  return
  {
    <Box>
      <headerBlock/>
      <content/>
      <footerBlock/>
    </Box>
  };
}

Base keyhandling approach:
    each ViewBase has a reference to the App.
    ViewBase.useInput() receives key input.
      if key is a view-changing key, ViewBase.app.changeView(key, this) is called, passing the key and current view up to the app, so the app can

  App::changeView(key, ViewBase)
    destruct the current view, removing all listeners of key events
    use 'key' to determine which view should be created.
    construct new view (passing in app as reference for view changing).
    make new view visible.

Architectural decision — View Configuration Pattern:

  The C++ rebuttal above proposed each view owning its own Header/Content/Footer
  via inheritance (ViewBase -> HeaderBase -> ContentBase -> FooterBase). The
  goals were correct (each view controls its header/footer behavior) but the
  mechanisms (inheritance, imperative render ordering, manual destruction) don't
  translate to React. The React-idiomatic solution:

  - App owns fixed Header and Footer instances (one each, always rendered)
  - Header derives its content from currentView (Phase 1 via prop, Phase 2+ via nav-driven app state)
  - Footer derives shortcuts from viewType + footerOptions props
  - Views configure the footer via a setFooterOptions callback from AppShell
  - Most views don't call setFooterOptions (their shortcuts are static per type)
  - DetailView calls setFooterOptions when focus state changes
  - footerOptions resets to {} on view change
  - Each view owns its own useInput for key handling
  - useGlobalKeys at app level handles cross-view shortcuts (q/s/b/g/n)
    with inputActive guard to suppress single-char keys during text entry

  Component tree and data flow:

    NavigationProvider  (provides currentView, navigateTo, goBack, etc.)
      └── AppShell  (calls useNavigation() to read currentView,
                     owns footerOptions state, owns useGlobalKeys hook)
            │
            ├── Header
            │     receives currentView from AppShell
            │     derives subtitle and view label from that value
            │
            ├── ViewRouter  (receives setFooterOptions callback as prop)
            │     │
            │     ├── HomeView        — owns useInput, does NOT call setFooterOptions
            │     ├── AgentStatus     — owns useInput, does NOT call setFooterOptions
            │     ├── BlockingMap     — owns useInput, does NOT call setFooterOptions
            │     ├── NewIssue        — owns useInput, does NOT call setFooterOptions
            │     ├── GroupView       — owns useInput, does NOT call setFooterOptions
            │     └── DetailView      — owns useInput, CALLS setFooterOptions
            │           useEffect: when focus state changes, calls
            │           setFooterOptions({ inThread, responseSelected,
            │                              hasReplies, isQuoting,
            │                              focusedAction, inputFocused })
            │
            └── Footer  (receives viewType + footerOptions as props)
                  viewType derived from AppShell's useNavigation()
                  footerOptions comes from AppShell state
                  calls getFooterShortcuts(viewType, footerOptions)

  Callback flow (downward):
    AppShell --setFooterOptions--> ViewRouter --setFooterOptions--> DetailView

  Data flow (upward via state):
    DetailView --calls setFooterOptions()--> AppShell state updates
                                             --> Footer re-renders with new props

  useGlobalKeys(currentViewType, inputActive):
    Lives in AppShell. Handles q/s/b/g/n/Esc.
    inputActive = true when DetailView's TextInput is focused.
    Suppresses single-char shortcuts during text entry.

     - Tests use Phase 3 mock data where applicable (otherwise local fixtures)
     - Tests use ink-testing-library for component tests
     - Tests use plain function calls for hook/utility tests

  2. Write minimal implementation (GREEN)
     - Make tests pass with simplest possible code
     - No over-engineering, no "future-proofing"

  3. Refactor (REFACTOR)
     - Clean up only if needed
     - Extract hooks/utilities only when duplication appears
     - Run all tests after every refactor

  4. Verify integration
     - Run full test suite: npx vitest run
     - Launch app if applicable: node dist/tui/run.js
     - Spot-check visual output in terminal


================================================================================
GATE TEST RULE
================================================================================

Every step and substep MUST include an explicit `Gate test` block with at
least one named test/check. ALL tests for a step MUST pass before the next
step/substep begins. No implicit or deferred-only gates are allowed. No step
may be marked complete with failing tests.

  - Implementation steps: at least one NEW test per step (required by TDD above).
  - Schema/design steps: gate test must still be explicit and local to the step.
    (Exception: design-only steps may use a commit gate instead of a runtime test.)
  - Integration/wiring steps: at least one integration test verifying the wiring.
  - Polish steps: at least one visual/behavioral regression test.

Enforcement:
  1. Run `npx vitest run` after completing each step. Zero failures required.
  2. Run `npx tsc --noEmit` after each step to verify structural type safety.
  3. Each step's git commit MUST include at least one new test case or test file.
     Running only the pre-existing test suite does not satisfy the gate.
  4. Record pass count in the step's checkbox line.
  5. All previous steps' tests MUST remain green (regression clause).
