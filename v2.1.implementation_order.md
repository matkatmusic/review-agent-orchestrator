## Phase 11: Resolve — Apply Worktree to Active Branch
> Build phase: "Issue -> resolve: check if rebase needed, then rebase, then apply worktree to current active branch in repo, changes are unstaged"

### 11.1 Evaluate apply strategies
V1 evolved through 5 approaches before stabilizing on rebase-then-format-patch. V2.1 differs: agents may have multiple commits (v1 amended to one). Test each strategy against a multi-commit worktree branch to determine the correct approach:
- [ ] **Strategy A — rebase + `git diff HEAD..worktree-I<#> | git apply`**: rebase worktree onto active HEAD, then diff captures all agent changes as unstaged. Simplest, but untested with multi-commit branches.
- [ ] **Strategy B — rebase + `git format-patch <merge-base>..HEAD --stdout | git apply`**: rebase worktree, then format-patch all commits since divergence. Closest to v1's stable approach, extended for multiple commits.
- [ ] **Strategy C — rebase + `git format-patch -1 --stdout | git apply` with pre-squash**: squash all worktree commits into one first (`git reset --soft <merge-base> && git commit`), then use v1's exact single-patch approach. Most proven, but modifies branch history.
- [ ] **Strategy D — rebase + `git format-patch <merge-base>..HEAD --stdout | git apply --3way`**: like B but with 3-way merge fallback for context mismatches.
- [ ] Test each strategy: create a worktree branch with 3+ commits, commit new work on active branch to force divergence, then apply. Measure: clean apply rate, conflict handling, unstaged result correctness.
- [ ] Select the most stable strategy and document the choice.

### 11.2 Resolve flow (using selected strategy)
- [ ] User selects Resolve from TUI (Detail view or Dashboard)
- [ ] Agent rebases onto active branch HEAD (agent resolves its own rebase conflicts, as in v1 — agent has context about its changes)
- [ ] If agent cannot resolve rebase conflicts: report conflicting files to user via TUI, pause resolve
- [ ] Apply worktree changes to active branch as unstaged using the strategy selected in 11.1
- [ ] If apply fails: report conflicting files to user, user fixes or cancels
- [ ] Kill agent pane, delete agent_sessions row
- [ ] Remove worktree (branch optionally kept for history)
- [ ] Set issue status to Resolved, set resolved_at timestamp
- [ ] Tests: changes appear unstaged in main repo, agent cleaned up, issue resolved

### 11.3 Conflict handling
- [ ] Rebase conflicts: agent resolves (has context about its own changes). If agent fails, escalate to user via TUI.
- [ ] Apply conflicts: report conflicting files to user, user fixes in editor or cancels resolve.
- [ ] Tests: conflict reported, resolve can be retried
