# v2.1 Design Discussion Summary

Context for debating the design decisions in `v2.1.md` and the implementation plan in `v2.1.implementation_order.md`. This captures the reasoning, tradeoffs, alternatives considered, user constraints, and corrections that shaped the current design.

---

## Architecture: Single Process vs. Two Processes

The v1 system ran the TUI and daemon as separate processes communicating through the filesystem (`.pending/` JSON files, lockfiles) and a shared SQLite database. The v2.1 redesign evaluated whether to keep that separation or merge them.

**Tradeoffs discussed:**

| Concern | Single Process | Two Processes |
|---------|---------------|---------------|
| Launch complexity | One command | Must launch both; partial state if one dies |
| State sync | Direct — instant UI refresh | TUI polls DB; 1-3s delay |
| Crash blast radius | TUI crash kills orchestrator | Each survives the other |
| Code complexity | ~50 fewer lines, no coordination | Daemon entry point, TUI entry point, lifecycle management |
| Risk (single) | Ink render loop could delay scan cycle | N/A |
| Risk (two) | N/A | Process lifecycle coordination bugs |

**Decision:** Single process. The user's key addition: session recovery on startup — on launch, check for existing tmux panes matching the naming convention, reconcile against the `agent_sessions` DB table, re-adopt alive panes, clean up dead ones. This mitigates the crash blast radius concern: if the orchestrator crashes, agents survive in tmux and can be re-adopted on restart.

**Open question for debate:** Whether the Ink render loop could meaningfully delay the 1-2s scan cycle under load (many active agents writing to DB simultaneously).

## Issue Statuses: 5-State Model

The user defined each status with specific semantics:

- **Awaiting** — "This issue is a candidate for being worked on, because nothing is blocking it." No agent, no worktree.
- **Active** — "An agent is actively working on the issue and a worktree has been created." User-triggered from TUI, capped by MAX_AGENTS.
- **Blocked** — Depends on another issue. System-managed: auto-transitions to Awaiting when all blockers resolve. Agent can self-detect via `aidi block`.
- **Deferred** — User chose to postpone. Only user can move it back. Explicitly distinct from Blocked.
- **Resolved** — Done. Pane killed, worktree applied. Can be reopened (→ Awaiting).

**Why Blocked and Deferred are separate:** The user was emphatic that "Deferred is something that the user chooses" while "Blocked means another Issue needs to be resolved first." Blocked is system-driven with automatic unblocking; Deferred is purely user-controlled with manual unblocking. Combining them (as the original draft implied) would conflate system state with user intent.

**Key constraint — user-triggered activation:** Unlike v1 (where the daemon auto-spawned agents for any open question), v2.1 requires the user to explicitly mark an issue as Active from the TUI. The TUI disables the "make active" action when active count >= MAX_AGENTS. This gives the user control over what gets worked on and when.

## Worktree Lifecycle: Commit-to-Branch Strategy

When an Active issue becomes Blocked or Deferred, the agent's work needs to be preserved.

**Options considered:**
1. Keep worktree, kill pane only — simplest, but disk accumulates with unused worktrees
2. Agent commits to branch, worktree removed — enables cross-machine resumption
3. Agent commits to branch, worktree kept — hybrid, safest but most disk
4. Git stash + metadata — fragile stash management

**Decision:** Option 2. The user's reasoning: "It allows resumption of worktrees on different computers. The downside is a non-issue, because that is how you use git." Committing to a named branch (`worktree-I<#>`) before removing the worktree is standard git workflow and enables picking up work on a different machine.

**On resolve:** Rebase check against the active branch, apply changes as unstaged (like a pull request for review), kill agent, remove worktree.

**Rebasing is manual only:** A `rebase-worktree` TUI command, not automatic. The user added this as an explicit build phase. No auto-rebase on main branch commits — the scan cycle flags "needs rebase" but takes no action.

**Rejected:** Viewing worktree changes from the TUI — "I'll just use a dedicated git GUI to inspect worktree changes."

## Agent-User Communication: Two-Channel Model

The v1 system had a single communication path: verbose tmux send-keys re-prompts that ate context and tokens. The user described the problem: "the agent was prompted as 'new user response added to <Qnum>.md, review response. if it says implement,...' which was very verbose."

**The user's metaphor:** "The goal is metaphorically identical to adding a reply to a ticket on Jira, and also being able to walk over to the agent's desk and talk to them directly."

This produced a strict two-channel design:

### Ticket channel (formal, logged)
- User writes response in TUI → saved to DB → orchestrator rewrites `issue.md` in worktree with full issue state → sends short tmux signal (e.g., `"I7: new reply"`) → agent reads `issue.md`
- Agent writes responses to DB via `aidi respond <inum> "(type) message"`
- Everything persists in the database

### Direct channel (informal, ephemeral)
- User types directly in tmux pane, agent responds in pane
- NOT logged unless user explicitly says "log that," at which point the agent uses `aidi` to write to DB
- Analogous to walking over to someone's desk for a quick chat

**Strict separation:** Ticket channel always goes through DB. Direct channel always stays in the pane. The two never mix unless the user explicitly requests logging.

**Signal efficiency:** Only user responses trigger signals to the agent. Agent's own DB writes do not loop back as notifications. The signal is minimal (a few words via send-keys), not a verbose re-prompt. The agent then reads the full context from `issue.md`.

**Response tagging:** Agents prefix their DB responses with `(type)` tags — `(log)`, `(note)`, `(info)`, etc. The tag is part of the body text, not a separate column. Purpose: indicate the intent behind the response to the reader.

## Agent Context Delivery: File-Based, Not CLI-Based

**Decision:** Each worktree receives two untracked files:
- **CLAUDE.md** — agent behavioral instructions (v2.1 equivalent of v1's `review-agent.md`)
- **issue.md** — issue title, description, and full conversation history

**issue.md is the agent's source of truth** for its own issue. The agent does NOT read its own issue from DB. This avoids the need for the agent to parse CLI output for its primary context.

**issue.md is fully rewritten** (not appended) on every user response. This ensures the file always matches DB state. Appending was rejected because it could diverge from the database if edits or deletions occurred.

**aidi (agent-issue-db-interface) is used for:**
- Writing responses to the agent's own issue (`aidi respond`)
- Reading OTHER issues (blocking info, mentioned issues) (`aidi read`)
- Creating sub-issues (`aidi create`)
- Reporting blocks (`aidi block`)

**Naming:** Full name `agent-issue-db-interface`, symlinked as `aidi` for brevity. The user rejected shorter abstract names (rit, ix, rio, ait, itx, isst) because none of them described what the tool does. The final name is descriptive when read in full and short when invoked.

## Tmux & Pane Management

**Layout:** Horizontal split — TUI in top half, agent panes in bottom half. The user framed this as an experiment: "I would like to try out having the TUI be in the top half of the tmux window."

**Auto-launch:** The orchestrator automatically spawns a Terminal.app window (or IDE's integrated terminal) running a tmux session with the horizontal split when launched. This was a user-initiated addition to the spec.

**Pane visibility — designed, then revised:**
- Initial design: panes hidden by default, auto-shown when agent needs user permission. Reasoning: "Agentic output is very noisy and distracting."
- User caught a conflict: hidden-by-default prevents the "walk over to agent's desk" direct interaction pattern.
- Revised design: panes still hidden by default, but the TUI provides a "show pane" action from Dashboard (for selected Active issue) and Issue Detail view to manually reveal an agent's pane.

## Groups & Organizational Hierarchy

The user wanted multi-tiered organization similar to sprints/milestones in project management tools.

**Schema options considered:**
1. **Unified containers table** — self-referencing `parent_id`, tree structure, one table for both groups and sprints
2. **Separate tables** — explicit Groups table + Sprints table + two join tables

**User's deciding question:** "Which one allows for most variability of linking issues together (analogous to a tree data structure)?"

**Decision:** Unified containers table. A single `containers` table with `type` column (`group` or `sprint`) and `parent_id` for arbitrary nesting (Group → Sprint → sub-sprint → ...), plus an `issue_containers` join table for many-to-many relationships with per-container `sort_order`.

**Additional decisions:**
- Default "Inbox" group — all new issues auto-assigned, user can move them
- Group lifecycle — when all issues in a group resolve, prompt user: "close / add issues / defer?" (not automatic)
- Sort order — status first (Active → Awaiting → Blocked → Deferred → Resolved), then `sort_order` (per-container, if set), then inum as fallback
- Blocking references individual inums, not groups — to block on a group, expand to all inums in that group

## Agent Permissions: Per-Worktree Isolation

The user challenged the assumption that agents would inherit the project root's `.claude/settings.json`: "When an agent is spawned in a Worktree, why would it need the settings.json found in the $PROJECT_ROOT? Why wouldn't it get its own?"

**Decision:** Each worktree gets its own `/<worktree>/.claude/settings.json` written by the orchestrator on worktree creation. Not inherited from the project root.

**The exact permission set is deferred.** It will be determined empirically during implementation: start with v1's permission set as a baseline, spawn agents, observe what gets blocked, iterate. The user added a build phase specifically for this verification step.

## Installation & Configuration

- **Git submodule** — same as v1
- **Idiot-proof install.sh** — checks all runtime requirements (Node.js 18+, npm, tmux, Claude CLI, git), asks before attempting to install each missing dependency, reports failures, then runs npm install + build + DB init + config template
- **Config format: JSON** — `config.json` + `config.local.json` (gitignored). Chosen over v1's shell config (`config.sh`) because JSON.parse() is built-in, TypeScript interfaces provide type safety, and no custom parser is needed.
- **Auto-update:** After `git submodule update`, re-run `install.sh`

## TUI Design Decisions

- **6 views:** Dashboard, Issue Detail, New Issue, Agent Status, Blocking Map, Group View
- **Color-coded text** for agent vs. user responses — no markdown rendering
- **Manual title + description** at issue creation, with option for agent-generated auto-summary
- **Persistent header and footer** with per-view keyboard shortcuts
- **Both users and agents** can create issues (agents create sub-issues via `aidi create`)
- **Scan cycle:** 1-2 second interval via `useEffect` + `setInterval`

## User Constraints & Principles

1. **User controls activation** — Active is user-triggered, not auto-spawned
2. **MAX_AGENTS caps concurrency** — number of tmux panes = number of active issues
3. **Cross-machine resumption** — worktree changes must be committed to a branch
4. **Token efficiency** — signals must be minimal; no verbose re-prompts
5. **Strict channel separation** — ticket (DB) and direct (tmux) never mix unless explicitly requested
6. **issue.md is the agent's source of truth** for its own issue
7. **Only user responses trigger agent signals** — agent's own writes don't loop back
8. **Red/Green TDD at all stages** — tests first, then implementation
9. **Idiot-proof installation** — install.sh handles everything
10. **Empirical permissions** — determined during implementation, not pre-specified
11. **Incremental build phases** — TUI first (no agents, no DB), then DB, then worktrees, then agents, then instructions
