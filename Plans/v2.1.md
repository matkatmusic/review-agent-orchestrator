Plan: issue Review Orchestrator вҖ” v2

 Context

The v1 file-based issue system didn't work reliably вҖ” `is_blocked()` failed, agents spawned despite unresolved blockers, and the XML tag format was brittle. The v2 redesign replaces issue files entirely with a SQLite database and a TUI (terminal user interface) for user interaction, but was buggy.  This is a complete rewrite from scratch of v2, starting from a proper design phase before anything is built.

The v1 system has outgrown bash. Rewriting as a Node.js/TypeScript app gives type safety, native SQLite access via `better-sqlite3`, a TUI framework, proper error handling, and testability.

The result is a repo-level ticketing system вҖ” like Jira but offline, portable, and travels with the repo. Clone the repo, run setup, and all issues/responses/dependencies are immediately available.  Git worktrees are used to isolate agent workspaces and enable parallel work on multiple Issues.  When agents are done with their work, they ask to apply their changes to the current branch (not necessarily `main`).  when granted permission to apply, their changes are unstaged for review, similar to a pull request.

 Resolved Design Decisions

Stack
- [x] TypeScript app, using Ink for TUI, better-sqlite3 for SQLite access
- [x] Red/Green TDD at all stages of development
- [x] Config format: JSON (`config.json` + `config.local.json` for gitignored overrides)

Architecture
- [x] Single process вҖ” TUI + orchestrator loop in one Ink app. `scanCycle` runs on a `useEffect` timer (1-2s interval). One thing to launch, one thing to stop.
- [x] Session recovery on startup вҖ” on launch, if an existing tmux session with agent panes is detected: signal each agent to commit via `cancel_and_send`, wait for clean trees, kill the tmux session, start a fresh session, and relaunch Active agents from their worktree branches. Avoids pane identity problems entirely.
- [x] SQLite WAL mode вҖ” enables concurrent reads/writes from the main process and aidi (agent CLI).

Issue Statuses (5 states)
- [x] Awaiting вҖ” candidate for work, nothing blocking it. No agent, no worktree.
- [x] Active вҖ” user explicitly activated via TUI. Agent spawned, worktree created. Capped by `MAX_AGENTS`.
- [x] Blocked вҖ” depends on another issue. Background scan auto-transitions to Awaiting when all blockers resolve.
- [x] Deferred вҖ” user chose to postpone. Stays until user manually moves it.
- [x] Resolved вҖ” done. Pane killed, worktree applied. Can be reopened (вҶ’ Awaiting).

Status Transitions
- [x] Awaiting вҶ’ Active: user triggers from TUI. TUI disables "make active" when active count >= MAX_AGENTS.
- [x] Active вҶ’ Blocked: agent self-detects via aidi, or user sets manually. Agent commits all work to branch, worktree removed, pane killed.
- [x] Active вҶ’ Deferred: user choice. Agent commits to branch, worktree removed, pane killed.
- [x] Active вҶ’ Resolved: user choice. Rebase check, then apply worktree to active branch (unstaged), agent deleted.
- [x] Blocked вҶ’ Awaiting: automatic when all blockers resolve (background scan).
- [x] Deferred вҶ’ Awaiting: user manually moves it.
- [x] Resolved вҶ’ Awaiting: user reopens.
- [x] Any вҶ’ Deferred: user choice from TUI.

Agent вҶ” User Communication (two channels)
- [x] Ticket channel (formal, logged): user writes response in TUI вҶ’ saved to DB (increments `issue_revision`) вҶ’ orchestrator sends `C-c` to agent pane and polls until agent is idle (v1's `cancel_and_send` pattern) вҶ’ orchestrator rewrites `issue.md` via atomic write (write-to-temp + fsync + rename), then `chmod 444` вҶ’ sends short signal via `tmux send-keys` (e.g., `"I7: new reply"`) вҶ’ agent reads `issue.md` вҶ’ agent runs `aidi ack <inum> <revision>` to confirm receipt (sets `agent_last_read_at` in DB). Orchestrator compares ack revision against current `issue_revision`; retries signal if mismatch or no ack within N seconds. Agent writes responses to DB via `aidi respond <inum> --type <type> "message"`. Only user responses trigger a signal to the agent; agent's own writes do not.
- [x] Direct channel (informal, ephemeral): user types in tmux pane, agent responds in pane. NOT logged unless user says "log that", in which case agent uses aidi to write to DB.
- [x] Strict separation: ticket = always via DB. Direct = always in chat. Agent never writes to DB from direct chat unless explicitly told to log.

Agent Response Tagging
- [x] Agent responses have a `type` column: `log`, `note`, `info`, `analysis`, `implementation`, `fix`, etc. Stored as a separate field in the `responses` table, not embedded in the body text. Agent submits type via `aidi respond <inum> --type <type> "message"`. User responses have `type = NULL`. TUI renders the type as a colored label in the message box header.

Agent Prompt & Context
- [x] CLAUDE.md written (untracked) into each worktree вҖ” contains agent behavioral instructions. Includes: routine commit instructions, commit-on-signal instructions, "never modify issue.md directly", "do not modify files outside the worktree folder", and "on teardown signal, log current work via `aidi respond` before committing."
- [x] issue.md written (untracked, `chmod 444`) into each worktree вҖ” contains a structured header (inum, title, status, description, blocked_by, blocks) followed by full conversation history. Orchestrator rewrites the entire file via atomic write (write-to-temp + fsync + rename, then `chmod 444`) on every DB change (user responses only), then sends a short signal.
- [x] aidi for writes вҖ” agent uses `aidi respond`, `aidi block`, `aidi create` for all DB writes to its own issue and for creating sub-issues.
- [x] aidi for reads of other issues вҖ” agent reads other issues (blocking info, mentioned issues) via aidi, not issue.md.
- [x] issue.md is the source of truth for the agent's own issue context. Agent does not read its own issue from DB.

Agent Permissions
- [x] Each worktree gets its own `/<worktree>/.claude/settings.json` (NOT inherited from project root).
- [x] Exact permission set determined empirically during build phase (see build phase line 19).

Worktree Lifecycle
- [x] On activate: create worktree (or re-create from existing branch if resuming).
- [x] On block/defer: orchestrator signals agent to commit and stop (agent logs current work via `aidi respond` before committing), polls for clean working tree with configurable timeout (`teardownTimeout`, default 60s). On timeout: TUI notification with force-kill option. Agent handles all git operations in its worktree вҖ” orchestrator never touches worktree git state. Enables resumption on different computers.
- [x] On resolve: check if rebase needed, rebase, apply to active branch (unstaged), worktree removed, agent deleted.
- [x] Rebase: manual 'rebase-worktree' TUI command (not automatic). Available from Issue Detail view and Dashboard.

TUI
- [x] Launch: orchestrator automatically spawns a Terminal.app window (or IDE's integrated terminal) running a tmux session with horizontal split (top half: orchestrator TUI, bottom half: agent panes).
- [x] Pane visibility: agent panes hidden by default, auto-shown when agent needs user permission. TUI provides a "show pane" action from [Dashboard view (for selected Active issue) and Issue Detail view] to manually reveal an agent's pane for direct interaction. Agentic output is noisy вҖ” hiding by default reduces visual clutter.
- [x] Views (6): Dashboard, Issue Detail, New Issue, Agent Status, Blocking Map, Group View.
- [x] Display: color-coded text (agent vs user responses). No markdown rendering.
- [x] Summaries: manual title + description at creation. Option to replace description with agent-generated auto-summary of the conversation.
- [x] Persistent header and footer with per-page keyboard command options.

Groups & Sprints (hierarchical organization)
- [x] Unified `containers` table with `type` column (`group` or `sprint`). Self-referencing `parent_id` for nesting (Group вҶ’ Sprint вҶ’ sub-sprint, etc.).
- [x] Join table `issue_containers` links issues to containers (many-to-many). Per-container `sort_order`.
- [x] Default 'Inbox' group вҖ” new issues auto-assigned. User can move them.
- [x] Group lifecycle: when all issues in a group are Resolved, prompt user: "Group has no unresolved issues. Close / Add issues / Defer?"
- [x] Sort order within containers: status first (Active вҶ’ Awaiting вҶ’ Blocked вҶ’ Deferred вҶ’ Resolved), then `sort_order` (if set), then inum as fallback.
- [x] Group View: navigable вҖ” next/previous issue within a group.

Issue Creation
- [x] Both users and agents can create issues. Agents create sub-issues via `aidi create`.

Install & Setup
- [x] Git submodule вҖ” tool installed as submodule in target project.
- [x] Idiot-proof `install.sh` вҖ” checks all runtime requirements (Node.js 18+, npm, tmux, Claude CLI, git), asks user before attempting to install each missing dependency, reports failures. Then: `npm install`, `npm run build`, initialize DB, write config template.
- [x] Auto-update: after `git submodule update`, re-run `install.sh` to handle `npm install` + rebuild.

 Design Decisions (consolidated)

Data & Storage
- SQLite database (WAL mode) вҖ” single source of truth for issues, responses, dependencies, status, agent sessions, and container hierarchy
- SQL dump tracked in git вҖ” `issues.dump.sql` is the portable, diffable artifact. The `.db` file is gitignored (runtime artifact rebuilt from dump)
- Auto-export вҖ” orchestrator writes `issues.dump.sql` after every scan cycle that changes the DB
- JSON config вҖ” `config.json` + `config.local.json` (gitignored) for project-specific settings
- inum as primary key вҖ” stable, embedded in DB rows, never changes
- `lastIssueCreated` вҖ” metadata row tracking the highest I number assigned

Architecture
- Single process вҖ” TUI + orchestrator in one Ink app. Scan cycle runs on 1-2s `setInterval`. All shell interactions (git, tmux, cancel_and_send) are asynchronous (`exec`/`spawn` wrapped in Promises) to avoid blocking the Ink render loop.
- Session recovery вҖ” on startup, if existing agent panes detected: signal each to commit, wait for clean trees, kill tmux session, start fresh, relaunch Active agents from worktree branches
- Git submodule вҖ” tool installed as submodule, managed via `install.sh`

Issues & Statuses
- 5 statuses: `Awaiting` вҶ’ `Active` вҶ’ `Resolved`, plus `Blocked` (system) and `Deferred` (user)
- Active is user-triggered вҖ” user sets via TUI, capped by MAX_AGENTS config value
- Blocked is separate from Deferred вҖ” Blocked = dependency on another issue (auto-unblocks). Deferred = user choice (manual move only).
- Enforce blocked invariant вҖ” blocked issues cannot be `Awaiting` or `Active`; background scan enforces every cycle
- Auto-unblock вҖ” `Blocked` issues whose blockers are all `Resolved` auto-move to `Awaiting`
- Auto re-block вҖ” if a resolved blocker is reopened and the dependency row still exists, dependents auto-transition back to `Blocked`. Dependency rows are the source of truth.
- Cycle detection вҖ” reuse v2's `wouldCreateCycle` recursive CTE at `addBlock` time. Rejects direct and transitive cycles.
- Resolved can reopen вҖ” transitions back to `Awaiting` (may re-block dependents)

Agent Interaction
- Two communication channels вҖ” ticket (DB, formal, logged) and direct (tmux pane, informal, ephemeral). Strict separation.
- Agents interact via CLI tool вҖ” `agent-issue-db-interface` (Node script), symlinked as `aidi` for brevity. Agents invoke `aidi respond <inum> --type <type> "message"`, `aidi block`, `aidi create`, `aidi read <inum>`, `aidi ack <inum>` (confirm issue.md read)
- issue.md in worktree вҖ” on user response (which increments `issue_revision`), orchestrator sends `C-c` to interrupt agent, polls until idle, rewrites issue.md via atomic write (write-to-temp + fsync + rename, then `chmod 444`), then sends short tmux signal. Agent reads file, runs `aidi ack <inum> <revision>` to confirm receipt. Orchestrator validates revision match; retries if mismatch or no ack within N seconds. Agent reads from file, not DB, for its own issue. Interrupt-before-rewrite eliminates file read race conditions. Atomic write protects against orchestrator crash mid-write.
- CLAUDE.md in worktree вҖ” agent behavioral instructions (untracked): includes "never modify issue.md", "do not modify files outside worktree folder", "on teardown signal, log current work via aidi before committing"
- Response tagging вҖ” `type` column on responses table (`log`, `note`, `info`, etc.). Submitted via `--type` flag, not embedded in body text
- Per-worktree permissions вҖ” `/<worktree>/.claude/settings.json`, not inherited from project root. Scoped to the worktree folder вҖ” agent is not allowed to modify files outside its worktree.
- DB transactions вҖ” status transitions wrapped in `BEGIN IMMEDIATE` / `COMMIT` for DB state change only. All side effects (signals, polling, git, tmux) happen after the transaction commits successfully.

Groups & Organization
- Unified `containers` table вҖ” `type` = `group` or `sprint`, self-referencing `parent_id` for nesting
- Many-to-many вҖ” issues link to containers via `issue_containers` join table with per-container `sort_order`
- Default Inbox group вҖ” uncategorized issues auto-assigned
- Group lifecycle вҖ” prompt user when all issues resolved: close / add issues / defer
- `blocked_by`/`blocks` вҖ” reference individual inums. To block on a group, expand to all inums in that group
- `blocked_by` is purely DB-managed вҖ” agents call `aidi block`, no file-level annotations

Worktrees
- inum-based вҖ” `.claude/worktrees/I7`, branch `worktree-I7`
- On block/defer вҖ” agent commits to branch, worktree removed (enables cross-machine resumption)
- On resolve вҖ” rebase check, apply to active branch (unstaged), worktree removed
- Rebase is manual вҖ” `rebase-worktree` TUI command, not automatic

TUI
- Terminal.app window (or IDE's integrated terminal) running a Tmux session is automatically spawned with horizontal split (top half: orchestrator, bottom half: agent panes) when the orchestrator is launched.
- Pane visibility вҖ” hidden by default, auto-shown when agent needs user permission. TUI provides a "show pane" action from [Dashboard view (for selected Active issue) and Issue Detail view] to manually reveal an agent's pane for direct interaction.
- 6 views вҖ” Dashboard, Issue Detail, New Issue, Agent Status, Blocking Map, Group View
- Color-coded text вҖ” agent vs user responses distinguished by color, no markdown rendering
- Summaries вҖ” manual title + description, with option for agent auto-summary
- Sort order вҖ” status first, then `sort_order` (if set), then inum
- Logs вҖ” `[review] Spawned agent pane: I7 вҖ” Migrate ServerDerivedFields`
- Scan loop вҖ” DB-driven, 1-2s interval, checks for new responses and status invariants

 Database Schema

Schema source of truth: `sql/schema.sql` + `sql/seed.sql`

```sql
PRAGMA journal_mode=WAL;
PRAGMA foreign_keys=ON;
PRAGMA busy_timeout=5000;

CREATE TABLE IF NOT EXISTS metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS issues (
    inum              INTEGER PRIMARY KEY,
    title             TEXT NOT NULL,
    description       TEXT NOT NULL DEFAULT '',
    status            TEXT NOT NULL DEFAULT 'Awaiting'
                      CHECK (status IN ('Awaiting', 'Active', 'Blocked', 'Deferred', 'Resolved')),
    created_at        TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    resolved_at       TEXT,
    issue_revision    INTEGER NOT NULL DEFAULT 0,
    agent_last_read_at TEXT,
    user_last_viewed_at TEXT
);

CREATE TABLE IF NOT EXISTS responses (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    inum       INTEGER NOT NULL REFERENCES issues(inum),
    author     TEXT NOT NULL CHECK (author IN ('user', 'agent')),
    type       TEXT NOT NULL DEFAULT 'none'
               CHECK (type IN ('question', 'implementation', 'clarification', 'analysis', 'fix', 'other', 'none')),
    body       TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
CREATE INDEX IF NOT EXISTS idx_responses_inum ON responses(inum);

CREATE TABLE IF NOT EXISTS dependencies (
    blocker_inum INTEGER NOT NULL REFERENCES issues(inum),
    blocked_inum INTEGER NOT NULL REFERENCES issues(inum),
    PRIMARY KEY (blocker_inum, blocked_inum),
    CHECK (blocker_inum != blocked_inum)
);
CREATE INDEX IF NOT EXISTS idx_deps_blocked ON dependencies(blocked_inum);
CREATE INDEX IF NOT EXISTS idx_deps_blocker ON dependencies(blocker_inum);

CREATE TABLE IF NOT EXISTS containers (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL,
    type        TEXT NOT NULL CHECK (type IN ('group', 'sprint')),
    parent_id   INTEGER REFERENCES containers(id),
    description TEXT NOT NULL DEFAULT '',
    status      TEXT NOT NULL DEFAULT 'Open'
                CHECK (status IN ('Open', 'Closed')),
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    closed_at   TEXT
);

CREATE TABLE IF NOT EXISTS issue_containers (
    inum         INTEGER NOT NULL REFERENCES issues(inum),
    container_id INTEGER NOT NULL REFERENCES containers(id),
    sort_order   INTEGER,
    PRIMARY KEY (inum, container_id)
);

CREATE TABLE IF NOT EXISTS agent_sessions (
    inum        INTEGER PRIMARY KEY REFERENCES issues(inum),
    pane_id     TEXT NOT NULL,
    head_commit TEXT NOT NULL DEFAULT 'unknown',
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
```

Needs Decision

- [ ] P1-5. Ink's `useInput` has no event consumption вҖ” all handlers fire for every keypress.
  Discovered during Phase 1.11 manual UX testing. Ink's `useInput` hook registers handlers on a flat `process.stdin` listener list. Every registered handler fires for every keypress вҖ” there is no propagation, no bubbling, no way for a child component to "consume" an event and prevent parent components from also processing it. This caused multiple bugs:
  - Double-fire Esc: pressing Esc in Detail view fired both Detail's handler (calling `onBack`) and App's handler (calling `goBack`), causing two navigation pops in one keypress.
  - Key stealing: App-level global shortcuts (e.g., `n` for New Issue, `q` for Quit) fired even when a child view needed those keys (e.g., `n` for next issue in GroupView, any key for text entry in Detail).
  - Workaround fragility: the current `VIEW_OWNED_KEYS` pattern (a `Record<ViewType, ReadonlySet<string>>` declaring which keys each view "owns") prevents App from processing owned keys, but: (a) it must be manually maintained per view, (b) it doesn't prevent sibling or grandchild handlers from double-firing, (c) Detail view must list every typeable character to prevent global shortcuts from firing during text entry.

  Decision: Replace `useInput` with a JUCE-inspired key event system: `KeyEventContext` + `useKeyEvent` + focus tracking.

  Reference model: JUCE's `ComponentPeer::handleKeyPress` (juce_ComponentPeer.cpp:196-233). The core pattern is a single dispatch loop that walks from the focused component up the component tree. Each component gets a chance to consume the event via `keyPressed() вҶ’ bool`. Returns `true` = consumed (stop propagating). Returns `false` = pass to parent. Base `Component::keyPressed()` returns `false` by default, so unhandled keys naturally bubble. Tab handling is a framework-level fallback after all components pass.

  Design principles (adapted from JUCE for React/Ink):

  1. Boolean return = consumed (from JUCE `keyPressed вҶ’ bool`):
     Every `useKeyEvent` handler returns `true` (consumed, stop) or `false` (ignored, bubble up). This is the same convention as `Component::keyPressed`. No event object mutation, no `stopPropagation()` вҖ” just a return value. The base case is `false` (transparent to all keys), matching JUCE's `Component::keyPressed` default.

  2. Child-first dispatch, walking up the tree (from JUCE `handleKeyPress` loop):
     JUCE's dispatch is: `for (target = focused; target != null; target = target->getParentComponent())`. Our equivalent: a `KeyEventContext` provider at the App level intercepts `process.stdin` keypresses before Ink's `useInput` fires. It walks an ordered handler list (registered by `useKeyEvent` hooks). Handlers are ordered child-first by registration depth: deepest component registers first, App registers last. When a handler returns `true`, the walk stops. This replicates JUCE's parent-walk without needing actual DOM traversal.

     Implementation detail: each `useKeyEvent(handler, depth)` call registers the handler with a numeric depth (0 = App, 1 = view, 2 = sub-region like text input). The context sorts handlers by descending depth before dispatch. On each keypress, the context iterates handlers in order; first `true` return stops the loop.

  3. Focus tracking via context (from JUCE `getCurrentlyFocusedComponent` + `setWantsKeyboardFocus`):
     JUCE determines the dispatch target via `Component::getCurrentlyFocusedComponent()`. Components opt in via `setWantsKeyboardFocus(true)`. Our equivalent: `KeyEventContext` tracks a `focusedRegion: string` (e.g., `'text-input'`, `'conversation'`, `'dependency-links'`, `'list'`). Each `useKeyEvent` handler receives the current `focusedRegion` as a parameter and can decide whether to handle based on it. Only one region is focused at a time.

     Tab cycles focus between regions within a view (matching JUCE's tab-to-next-sibling fallback at line 223-230 of `handleKeyPress`). The Tab fallback is handled at the context level after all handlers return `false`, not inside any individual component вҖ” same as JUCE.

  4. Visual focus indicator:
     The focused region renders with a visual indicator (colored border, highlight). Unfocused regions render dimmed. This gives clear feedback about which component will receive keypresses. Each view reads `focusedRegion` from context and conditionally applies styling.

  5. TextInput coexistence:
     `ink-text-input` registers its own internal `useInput` that cannot be replaced. Two options:
     - (a) Coexist: TextInput keeps its own handler for text entry (characters, backspace, cursor). Our `useKeyEvent` handles navigation keys (Esc, Tab, global shortcuts). They don't conflict because TextInput only mutates text, never navigation. When `focusedRegion === 'text-input'`, the view's `useKeyEvent` handler returns `true` for all printable characters to suppress global shortcuts вҖ” analogous to how JUCE's `TextEditor::keyPressed` returns `true` for typeable characters (line 2182-2188) but `false` for unrecognized keys.
     - (b) Replace: Build a custom text input component that participates in `useKeyEvent` natively. Full control, like JUCE's `TextEditor` which is a Component with its own `keyPressed` override.
     - Preferred: (a) for now. Swap to (b) if TextInput's behavior conflicts with focus management (e.g., Tab-to-cycle-focus in Detail view вҖ” TextInput may swallow Tab).

  6. Eliminates VIEW_OWNED_KEYS:
     The per-view key ownership map (`Record<ViewType, ReadonlySet<string>>`) becomes unnecessary. Each component's handler naturally consumes only the keys it handles and returns `false` for everything else вҖ” exactly like JUCE components that only override `keyPressed` for their specific keys. App's handler at depth 0 acts as the root component, catching anything the active view didn't consume.

  Architecture mapping (JUCE вҶ’ TUI):

  | JUCE Concept | TUI Equivalent |
  |---|---|
  | `ComponentPeer::handleKeyPress` loop | `KeyEventContext` dispatch loop over sorted handler list |
  | `Component::keyPressed() вҶ’ bool` | `useKeyEvent(handler, depth) вҶ’ handler returns bool` |
  | `Component::keyPressed` default returns `false` | Handlers return `false` for unrecognized keys |
  | `getCurrentlyFocusedComponent()` | `KeyEventContext.focusedRegion` |
  | `setWantsKeyboardFocus(true)` | Registering a `useKeyEvent` handler at a given depth |
  | `KeyListener` (external observers) | Not needed вҖ” React's composition model handles this |
  | Tab fallback after all components pass | Context-level Tab handler after all `useKeyEvent` handlers return `false` |
  | `TextEditor::keyPressed` consuming typeable chars | TextInput region's handler returning `true` for printable chars when focused |
