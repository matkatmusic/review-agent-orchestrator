STEP 3: Build ResponseRow[] -> Response tree hydrator
========================================================================

The TUI already has thread-builders.ts that links in-memory Message
objects into a Response linked list. We need a similar function that
takes flat ResponseRow[] from the DB and produces a Response tree.

[ ] 3.1  Create src/db/response-tree.ts with:

         hydrateResponseTree(rows: ResponseRow[]): Response | null

         Algorithm:
         a) Build a Map<id, Response> from all rows (create Response
            nodes with null pointers).
         b) For each row with responding_to_id, set:
            node.responding_to = map.get(responding_to_id)
            parent.response = node
         c) For each row with replying_to_id, set:
            node.replying_to = map.get(replying_to_id)
            parent.reply = node (if parent.reply is null)
            OR append to end of reply's .response chain
         d) For continuations (is_continuation=1), set the flag.
         e) Find the root: the node with responding_to_id=null and
            replying_to_id=null. Return it.

         Handle edge cases:
         - Orphaned nodes (parent ID not in the set) -> log warning
         - Cycles (visited set guard)
         - Empty input -> return null

[ ] 3.2  Create src/db/response-tree.test.ts with tests:
         - Empty rows -> null
         - Single row -> single Response node
         - Linear chain (3 messages) -> linked via .response
         - Thread (main chain + reply branch) -> .reply links correct
         - Nested thread (reply to a reply) -> nested .reply
         - Continuations flagged correctly
         - Mixed: main chain + multiple threads + nesting
         - Orphan handling (missing parent ID)

[ ] 3.3  Add a convenience function:
         getResponseTree(db: DB, inum: number): Response | null
         that calls listByInum() then hydrateResponseTree().

[ ] 3.4  Run tests: `npx vitest run`

from a Response tree, following the debate-consensus format.

[ ] 4.1  Create src/serializer/issue-md.ts with:

         serializeIssueMd(opts: SerializeOpts): string

         interface SerializeOpts {
             inum: number;
             title: string;
             status: string;
             revision: number;
             agentLastAckRevision: number | null;
             blockedBy: number[];
             blocks: number[];
             group: string;
             rootResponse: Response | null;
             maxMainChainMessages?: number;   // default 30
             maxThreadMessages?: number;      // default 10
         }

[ ] 4.2  Implement header section:
           # I-{inum}: {title}
           Revision: {revision}
           Status: {status}
           Blocked by: {list or (none)}
           Blocks: {list or (none)}

[ ] 4.3  Implement continuation merging:
         Walk the response tree. When a node has is_continuation=true,
         merge its body into the preceding non-continuation node's
         body (double newline separator). Use the root node's ID,
         author, type, timestamp for the merged message.
         If a thread branches off a continuation, resolve the thread
         header to the root (non-continuation) message ID.

[ ] 4.4  Implement "What's New" summary section:
         Compare each message's timestamp against agentLastAckRevision.
         (Or use a simpler approach: mark messages added after the last
         ack as [NEW]. The revision number is on the issue, not per-
         message. Need to determine: which messages are "new"?)

         Approach: messages are "new" if they were created after the
         agent's last ack. Use agent_last_read_at timestamp on the
         issue to compare against each response's created_at.

         Alternative: track which response IDs existed at each revision.
         Simpler: pass in a cutoff timestamp, mark messages after it.

         Decision needed: clarify "new" definition. For now, accept an
         optional newSinceCutoff?: string (ISO timestamp). Messages with
         created_at > cutoff get [NEW] marker.

[ ] 4.5  Implement main conversation section:
         Walk rootResponse via .response chain (skip continuations,
         they were merged in 4.3). For each message:
           [#{id} {Author} - {Type} - {timestamp}]
           {body}
         If message has .reply, add thread reference line:
           Thread: #{id} has N replies (open/resolved) [NEW if applicable]

         Apply maxMainChainMessages windowing: if chain is longer than
         the limit, include only the last N and prepend:
           (... M earlier messages omitted. Earliest shown: #K)

[ ] 4.6  Implement thread sections:
         After the main chain, for each message that has .reply:

         Active threads (fully expanded):
           === Thread on #{id} ({Author}: "{body preview}") [N messages, open] ===
           [#{rid} {Author} - {timestamp} -> reply to #{parentId}] [NEW]
           {body}
           ...
           (sub-threads nested with --- Sub-thread on #{id} --- markers)

         Resolved threads (collapsed):
           === Thread on #{id} (...) [RESOLVED - N messages] ===
           (Resolved {timestamp}. Last message: {Author}: "{preview}")

         Apply maxThreadMessages windowing within each thread.

[ ] 4.7  Implement scalability windowing:
         - Main chain: last maxMainChainMessages (default 30)
         - Per-thread: last maxThreadMessages (default 10)
         - [NEW] messages always included regardless of window
         - Thread root (first reply) always included for context
         - Issue description (first message) always included

[ ] 4.8  Create src/serializer/issue-md.test.ts with tests:

STEP 5: Atomic file writer utility
========================================================================

issue.md must be written atomically: temp file + fsync + rename +
chmod 444. This protects against orchestrator crash mid-write.

[ ] 5.1  Create src/serializer/atomic-write.ts with:

         async function atomicWriteReadOnly(
             filePath: string,
             content: string
         ): Promise<void>

         Implementation:
         a) Write content to filePath + '.tmp'
         b) Open the file handle, call fsync, close
         c) Rename filePath.tmp -> filePath (atomic on POSIX)
