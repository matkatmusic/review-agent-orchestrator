Plan: issue Review Orchestrator — v2

 Context

The v1 file-based issue system didn't work reliably — `is_blocked()` failed, agents spawned despite unresolved blockers, and the XML tag format was brittle. The v2 redesign replaces issue files entirely with a SQLite database and a TUI (terminal user interface) for user interaction, but was buggy.  This is a complete rewrite from scratch of v2, starting from a proper design phase before anything is built.

The v1 system has outgrown bash. Rewriting as a Node.js/TypeScript app gives type safety, native SQLite access via `better-sqlite3`, a TUI framework, proper error handling, and testability.

The result is a repo-level ticketing system — like Jira but offline, portable, and travels with the repo. Clone the repo, run setup, and all issues/responses/dependencies are immediately available.  Git worktrees are used to isolate agent workspaces and enable parallel work on multiple Issues.  When agents are done with their work, they ask to apply their changes to the current branch (not necessarily `main`).  when granted permission to apply, their changes are unstaged for review, similar to a pull request. 

 Build phases:

- [ ] TUI first: flush out how UX feels to use (no agents spawned, menu nav only, no db changes)
- [ ] TUI db interaction: make sure tui and db communicate correctly, update UX after testing if needed.
- [ ] Issue -> Active: worktree creation or load from branch
- [ ] Issue -> defer/resolve/blocked_by: if worktree exists, commit to branch, delete worktree (background task?)
- [ ] Issue -> blocks I<#>: if I<#> worktree exists, commit to branch, delete worktree (background task?)
- [ ] Issue -> Active: Agent spawning (no instructions to agent)
- [ ] Issue -> Active: set permissions in /<worktree>/.claude/settings.json, verify agent has permission to do what it needs to do.
- [ ] Issue -> defer/resolve/blocked_by: agent deletion
- [ ] Git repo active branch commit: do nothing
- [ ] new TUI command: 'rebase-worktree': available from Issue view and Dashboard for selected Issue, tells the agent for an active worktree to rebase off of the current active branch in the repo. 
- [ ] Issue -> active: agent spawning with initial prompt ('read @issue.md')
- [ ] Issue -> resolve: check if rebase needed, then rebase, then apply worktree to current active branch in repo, changes are unstaged (v1's current behavior), agent deleted. 


 Resolved Design Decisions

Stack
- [x] TypeScript app, using Ink for TUI, better-sqlite3 for SQLite access
- [x] Red/Green TDD at all stages of development
- [x] Config format: JSON (`config.json` + `config.local.json` for gitignored overrides)

Architecture
- [x] Single process — TUI + orchestrator loop in one Ink app. `scanCycle` runs on a `useEffect` timer (1-2s interval). One thing to launch, one thing to stop.
- [x] Session recovery on startup — on launch, if an existing tmux session with agent panes is detected: signal each agent to commit via `cancel_and_send`, wait for clean trees, kill the tmux session, start a fresh session, and relaunch Active agents from their worktree branches. Avoids pane identity problems entirely.
- [x] SQLite WAL mode — enables concurrent reads/writes from the main process and aidi (agent CLI).

Issue Statuses (5 states)
- [x] Awaiting — candidate for work, nothing blocking it. No agent, no worktree.
- [x] Active — user explicitly activated via TUI. Agent spawned, worktree created. Capped by `MAX_AGENTS`.
- [x] Blocked — depends on another issue. Background scan auto-transitions to Awaiting when all blockers resolve.
- [x] Deferred — user chose to postpone. Stays until user manually moves it.
- [x] Resolved — done. Pane killed, worktree applied. Can be reopened (→ Awaiting).

Status Transitions
- [x] Awaiting → Active: user triggers from TUI. TUI disables "make active" when active count >= MAX_AGENTS.
- [x] Active → Blocked: agent self-detects via aidi, or user sets manually. Agent commits all work to branch, worktree removed, pane killed.
- [x] Active → Deferred: user choice. Agent commits to branch, worktree removed, pane killed.
- [x] Active → Resolved: user choice. Rebase check, then apply worktree to active branch (unstaged), agent deleted.
- [x] Blocked → Awaiting: automatic when all blockers resolve (background scan).
- [x] Deferred → Awaiting: user manually moves it.
- [x] Resolved → Awaiting: user reopens.
- [x] Any → Deferred: user choice from TUI.

Agent ↔ User Communication (two channels)
- [x] Ticket channel (formal, logged): user writes response in TUI → saved to DB (increments `issue_revision`) → orchestrator sends `C-c` to agent pane and polls until agent is idle (v1's `cancel_and_send` pattern) → orchestrator rewrites `issue.md` via atomic write (write-to-temp + fsync + rename), then `chmod 444` → sends short signal via `tmux send-keys` (e.g., `"I7: new reply"`) → agent reads `issue.md` → agent runs `aidi ack <inum> <revision>` to confirm receipt (sets `agent_last_read_at` in DB). Orchestrator compares ack revision against current `issue_revision`; retries signal if mismatch or no ack within N seconds. Agent writes responses to DB via `aidi respond <inum> "(type) message"`. Only user responses trigger a signal to the agent; agent's own writes do not.
- [x] Direct channel (informal, ephemeral): user types in tmux pane, agent responds in pane. NOT logged unless user says "log that", in which case agent uses aidi to write to DB.
- [x] Strict separation: ticket = always via DB. Direct = always in chat. Agent never writes to DB from direct chat unless explicitly told to log.

Agent Response Tagging
- [x] Agent prefixes DB responses with `(type)` tags: `(log)`, `(note)`, `(info)`, etc. Tag is part of the body text, not a separate column. Indicates intent behind the response to the reader.

Agent Prompt & Context
- [x] CLAUDE.md written (untracked) into each worktree — contains agent behavioral instructions. Includes: routine commit instructions, commit-on-signal instructions, "never modify issue.md directly", "do not modify files outside the worktree folder", and "on teardown signal, log current work via `aidi respond` before committing."
- [x] issue.md written (untracked, `chmod 444`) into each worktree — contains a structured header (inum, title, status, description, blocked_by, blocks) followed by full conversation history. Orchestrator rewrites the entire file via atomic write (write-to-temp + fsync + rename, then `chmod 444`) on every DB change (user responses only), then sends a short signal.
- [x] aidi for writes — agent uses `aidi respond`, `aidi block`, `aidi create` for all DB writes to its own issue and for creating sub-issues.
- [x] aidi for reads of other issues — agent reads other issues (blocking info, mentioned issues) via aidi, not issue.md.
- [x] issue.md is the source of truth for the agent's own issue context. Agent does not read its own issue from DB.

Agent Permissions
- [x] Each worktree gets its own `/<worktree>/.claude/settings.json` (NOT inherited from project root).
- [x] Exact permission set determined empirically during build phase (see build phase line 19).

Worktree Lifecycle
- [x] On activate: create worktree (or re-create from existing branch if resuming).
- [x] On block/defer: orchestrator signals agent to commit and stop (agent logs current work via `aidi respond` before committing), polls for clean working tree with configurable timeout (`teardownTimeout`, default 60s). On timeout: TUI notification with force-kill option. Agent handles all git operations in its worktree — orchestrator never touches worktree git state. Enables resumption on different computers.
- [x] On resolve: check if rebase needed, rebase, apply to active branch (unstaged), worktree removed, agent deleted.
- [x] Rebase: manual 'rebase-worktree' TUI command (not automatic). Available from Issue Detail view and Dashboard.

TUI
- [x] Launch: orchestrator automatically spawns a Terminal.app window (or IDE's integrated terminal) running a tmux session with horizontal split (top half: orchestrator TUI, bottom half: agent panes).
- [x] Pane visibility: agent panes hidden by default, auto-shown when agent needs user permission. TUI provides a "show pane" action from [Dashboard view (for selected Active issue) and Issue Detail view] to manually reveal an agent's pane for direct interaction. Agentic output is noisy — hiding by default reduces visual clutter.
- [x] Views (6): Dashboard, Issue Detail, New Issue, Agent Status, Blocking Map, Group View.
- [x] Display: color-coded text (agent vs user responses). No markdown rendering.
- [x] Summaries: manual title + description at creation. Option to replace description with agent-generated auto-summary of the conversation.
- [x] Persistent header and footer with per-page keyboard command options.

Groups & Sprints (hierarchical organization)
- [x] Unified `containers` table with `type` column (`group` or `sprint`). Self-referencing `parent_id` for nesting (Group → Sprint → sub-sprint, etc.).
- [x] Join table `issue_containers` links issues to containers (many-to-many). Per-container `sort_order`.
- [x] Default 'Inbox' group — new issues auto-assigned. User can move them.
- [x] Group lifecycle: when all issues in a group are Resolved, prompt user: "Group has no unresolved issues. Close / Add issues / Defer?"
- [x] Sort order within containers: status first (Active → Awaiting → Blocked → Deferred → Resolved), then `sort_order` (if set), then inum as fallback.
- [x] Group View: navigable — next/previous issue within a group.

Issue Creation
- [x] Both users and agents can create issues. Agents create sub-issues via `aidi create`.

Install & Setup
- [x] Git submodule — tool installed as submodule in target project.
- [x] Idiot-proof `install.sh` — checks all runtime requirements (Node.js 18+, npm, tmux, Claude CLI, git), asks user before attempting to install each missing dependency, reports failures. Then: `npm install`, `npm run build`, initialize DB, write config template.
- [x] Auto-update: after `git submodule update`, re-run `install.sh` to handle `npm install` + rebuild.

 Design Decisions (consolidated)

Data & Storage
- SQLite database (WAL mode) — single source of truth for issues, responses, dependencies, status, agent sessions, and container hierarchy
- SQL dump tracked in git — `issues.dump.sql` is the portable, diffable artifact. The `.db` file is gitignored (runtime artifact rebuilt from dump)
- Auto-export — orchestrator writes `issues.dump.sql` after every scan cycle that changes the DB
- JSON config — `config.json` + `config.local.json` (gitignored) for project-specific settings
- inum as primary key — stable, embedded in DB rows, never changes
- `lastIssueCreated` — metadata row tracking the highest I number assigned

Architecture
- Single process — TUI + orchestrator in one Ink app. Scan cycle runs on 1-2s `setInterval`. All shell interactions (git, tmux, cancel_and_send) are asynchronous (`exec`/`spawn` wrapped in Promises) to avoid blocking the Ink render loop.
- Session recovery — on startup, if existing agent panes detected: signal each to commit, wait for clean trees, kill tmux session, start fresh, relaunch Active agents from worktree branches
- Git submodule — tool installed as submodule, managed via `install.sh`

Issues & Statuses
- 5 statuses: `Awaiting` → `Active` → `Resolved`, plus `Blocked` (system) and `Deferred` (user)
- Active is user-triggered — user sets via TUI, capped by MAX_AGENTS config value
- Blocked is separate from Deferred — Blocked = dependency on another issue (auto-unblocks). Deferred = user choice (manual move only).
- Enforce blocked invariant — blocked issues cannot be `Awaiting` or `Active`; background scan enforces every cycle
- Auto-unblock — `Blocked` issues whose blockers are all `Resolved` auto-move to `Awaiting`
- Auto re-block — if a resolved blocker is reopened and the dependency row still exists, dependents auto-transition back to `Blocked`. Dependency rows are the source of truth.
- Cycle detection — reuse v2's `wouldCreateCycle` recursive CTE at `addBlock` time. Rejects direct and transitive cycles.
- Resolved can reopen — transitions back to `Awaiting` (may re-block dependents)

Agent Interaction
- Two communication channels — ticket (DB, formal, logged) and direct (tmux pane, informal, ephemeral). Strict separation.
- Agents interact via CLI tool — `agent-issue-db-interface` (Node script), symlinked as `aidi` for brevity. Agents invoke `aidi respond <inum> "(type) message"`, `aidi block`, `aidi create`, `aidi read <inum>`, `aidi ack <inum>` (confirm issue.md read)
- issue.md in worktree — on user response (which increments `issue_revision`), orchestrator sends `C-c` to interrupt agent, polls until idle, rewrites issue.md via atomic write (write-to-temp + fsync + rename, then `chmod 444`), then sends short tmux signal. Agent reads file, runs `aidi ack <inum> <revision>` to confirm receipt. Orchestrator validates revision match; retries if mismatch or no ack within N seconds. Agent reads from file, not DB, for its own issue. Interrupt-before-rewrite eliminates file read race conditions. Atomic write protects against orchestrator crash mid-write.
- CLAUDE.md in worktree — agent behavioral instructions (untracked): includes "never modify issue.md", "do not modify files outside worktree folder", "on teardown signal, log current work via aidi before committing"
- Response tagging — agent prefixes body with `(log)`, `(note)`, `(info)` etc. to indicate intent
- Per-worktree permissions — `/<worktree>/.claude/settings.json`, not inherited from project root. Scoped to the worktree folder — agent is not allowed to modify files outside its worktree.
- DB transactions — status transitions wrapped in `BEGIN IMMEDIATE` / `COMMIT` for DB state change only. All side effects (signals, polling, git, tmux) happen after the transaction commits successfully.

Groups & Organization
- Unified `containers` table — `type` = `group` or `sprint`, self-referencing `parent_id` for nesting
- Many-to-many — issues link to containers via `issue_containers` join table with per-container `sort_order`
- Default Inbox group — uncategorized issues auto-assigned
- Group lifecycle — prompt user when all issues resolved: close / add issues / defer
- `blocked_by`/`blocks` — reference individual inums. To block on a group, expand to all inums in that group
- `blocked_by` is purely DB-managed — agents call `aidi block`, no file-level annotations

Worktrees
- inum-based — `.claude/worktrees/I7`, branch `worktree-I7`
- On block/defer — agent commits to branch, worktree removed (enables cross-machine resumption)
- On resolve — rebase check, apply to active branch (unstaged), worktree removed
- Rebase is manual — `rebase-worktree` TUI command, not automatic

TUI
- Terminal.app window (or IDE's integrated terminal) running a Tmux session is automatically spawned with horizontal split (top half: orchestrator, bottom half: agent panes) when the orchestrator is launched.
- Pane visibility — hidden by default, auto-shown when agent needs user permission. TUI provides a "show pane" action from [Dashboard view (for selected Active issue) and Issue Detail view] to manually reveal an agent's pane for direct interaction.
- 6 views — Dashboard, Issue Detail, New Issue, Agent Status, Blocking Map, Group View
- Color-coded text — agent vs user responses distinguished by color, no markdown rendering
- Summaries — manual title + description, with option for agent auto-summary
- Sort order — status first, then `sort_order` (if set), then inum
- Logs — `[review] Spawned agent pane: I7 — Migrate ServerDerivedFields`
- Scan loop — DB-driven, 1-2s interval, checks for new responses and status invariants

 Database Schema

```sql
-- Enable WAL mode for concurrent access (orchestrator + aidi)
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;

CREATE TABLE metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Seed: INSERT INTO metadata VALUES ('lastIssueCreated', '0');

CREATE TABLE issues (
    inum              INTEGER PRIMARY KEY,
    title             TEXT NOT NULL,          -- short label (e.g., "migrate_ServerDerivedFields")
    description       TEXT NOT NULL DEFAULT '',  -- used in TUI list views, option for agent auto-summary
    status            TEXT NOT NULL DEFAULT 'Awaiting'
                      CHECK (status IN ('Awaiting', 'Active', 'Blocked', 'Deferred', 'Resolved')),
    created_at        TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    resolved_at       TEXT,
    issue_revision     INTEGER NOT NULL DEFAULT 0,  -- monotonic counter, incremented on each user response/issue.md rewrite
    agent_last_read_at TEXT                   -- set by `aidi ack <inum> <revision>` after agent reads issue.md
);

CREATE TABLE responses (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    inum       INTEGER NOT NULL REFERENCES issues(inum),
    author     TEXT NOT NULL CHECK (author IN ('user', 'agent')),
    body       TEXT NOT NULL,           -- agent responses prefixed with (log), (note), (info) etc.
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
CREATE INDEX idx_responses_inum ON responses(inum);

CREATE TABLE dependencies (
    blocker_inum INTEGER NOT NULL REFERENCES issues(inum),
    blocked_inum INTEGER NOT NULL REFERENCES issues(inum),
    PRIMARY KEY (blocker_inum, blocked_inum),
    CHECK (blocker_inum != blocked_inum)
);
CREATE INDEX idx_deps_blocked ON dependencies(blocked_inum);
CREATE INDEX idx_deps_blocker ON dependencies(blocker_inum);

-- Hierarchical organization: groups contain sprints, sprints contain issues
CREATE TABLE containers (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL,
    type        TEXT NOT NULL CHECK (type IN ('group', 'sprint')),
    parent_id   INTEGER REFERENCES containers(id),  -- sprints belong to groups, etc.
    description TEXT NOT NULL DEFAULT '',
    status      TEXT NOT NULL DEFAULT 'Open'
                CHECK (status IN ('Open', 'Closed')),
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    closed_at   TEXT
);

-- Many-to-many: issues can belong to multiple containers
CREATE TABLE issue_containers (
    inum         INTEGER NOT NULL REFERENCES issues(inum),
    container_id INTEGER NOT NULL REFERENCES containers(id),
    sort_order   INTEGER,              -- per-container sort position (NULL = sort by inum)
    PRIMARY KEY (inum, container_id)
);

-- Seed: default Inbox group for uncategorized issues
INSERT INTO containers (name, type, description)
VALUES ('Inbox', 'group', 'Default group for uncategorized issues');

-- Tracks active agent tmux panes (replaces filesystem lockfiles from v1)
CREATE TABLE agent_sessions (
    inum        INTEGER PRIMARY KEY REFERENCES issues(inum),
    pane_id     TEXT NOT NULL,
    head_commit TEXT NOT NULL DEFAULT 'unknown',
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
```

 Debate Issues (Q1 — Three-Model Review)

Issues identified by a three-way design review (Claude Opus 4.6, Gemini 3 Pro Preview, GPT-5.3 Codex). Each issue needs a resolution before implementation begins.

Unanimous (all 3 models)

- [x] D1. Scan cycle will block Ink rendering.
  - Resolution: Single-flight guard (skip cycle if previous still running) + debounce dump export (export after 5s of no further DB changes, not every cycle). DB queries stay synchronous (better-sqlite3 is <1ms at this scale). ~~Git/tmux calls stay synchronous for now.~~ Updated by E14: all shell interactions (git, tmux, cancel_and_send) are asynchronous from the start.
  - Status: RESOLVED

- [x] D2. aidi CLI is Phase 13 but needed by Phase 10.
  - Resolution: Move aidi to Phase 0.5 — build alongside the CRUD layer in Phase 0. It's a thin CLI wrapper over the same DB module. Enables integration testing of agent DB writes long before agents exist. Update `v2.1.implementation_order.md` accordingly.
  - Status: RESOLVED

- [x] D3. Resolve flow git mechanics are fragile.
  - Resolution: Pseudocode was a syntax error. V1 evolved through 5 approaches (direct edits → format-patch → --3way → rebase+format-patch → amend+rebase+format-patch). The final stable v1 approach was rebase-then-format-patch. V2.1 differs because agents have multiple commits (v1 amended to one). Phase 11 updated to include an evaluation of 4 apply strategies against multi-commit branches to empirically determine the correct approach. Agent handles its own rebase conflicts (has context), escalates to user only if it fails. Updated `v2.1.implementation_order.md` Phase 11 accordingly.
  - Status: RESOLVED

- [x] D4. Tmux signal delivery has no reliability guarantee.
  - Resolution: Two-part fix. (1) `cancel_and_send` pattern from v1 (D5 resolution) ensures signal is delivered to an idle agent, not mid-generation. (2) Agent runs `aidi ack <inum>` after reading issue.md, setting `agent_last_read_at` in the `issues` table. Orchestrator's scan cycle compares this against the latest user response timestamp and retries the signal if ack not received within N seconds. Added `agent_last_read_at` column to `issues` schema and `aidi ack` command.
  - Status: RESOLVED

Raised by 2 of 3 models

- [x] D5. issue.md rewrite race condition.
  - Resolution: Use v1's `cancel_and_send` pattern — orchestrator sends `C-c` to agent pane and polls until idle before rewriting `issue.md`. Agent is guaranteed stopped during the file rewrite, eliminating the race condition. No atomic file writes needed. Updated the ticket channel description in Resolved Design Decisions and Design Decisions (consolidated) to include the interrupt-before-rewrite step.
  - Status: RESOLVED 

- [x] D6. Dependency graph edge cases.
  - Resolution: (1) Cycle detection: reuse v2's `wouldCreateCycle` recursive CTE (rename qnum→inum). Rejects direct and transitive cycles at `addBlock` time. (2) Reopened blocker: auto re-block. If a resolved blocker is reopened and the dependency row still exists, the scan cycle re-evaluates and transitions the dependent back to Blocked. The dependency row is the source of truth, not the status. (3) Deferred blocker: treated same as unresolved — dependent stays Blocked.
  - Status: RESOLVED

- [x] D7. Phase 5 bare agent spawn has limited value.
  - Resolution: Accepted. Phase 5 is a tmux-only smoke test — validates pane creation, session recording, and session recovery mechanics. Agent behavioral testing deferred to Phase 10.
  - Status: RESOLVED 

- [x] D8. Phase 1 mock data then Phase 2 DB wiring may be double work.
  - Resolution: Not an issue. `better-sqlite3` is synchronous — there's no sync-to-async transition when swapping mock data for DB queries. Both are synchronous calls returning the same TypeScript types. If the schema doesn't change, views wire up with minimal changes (swap `getMockIssues()` for `issues.list(db)`).
  - Status: RESOLVED

Raised by 1 model

- [x] D9. Git index.lock contention across shared .git directory. Multiple agents in worktrees + orchestrator running git commands every 1-2 seconds risks `.git/index.lock` collisions. Worktrees share the same underlying `.git` directory object store. (Gemini)
  - Resolution: Not an issue at this scale. Each worktree has its own index file (`.git/worktrees/I<#>/index`), so `git status` and `git add` don't contend across worktrees. Shared resources (branch refs, packed-refs) use per-ref locks that rarely collide at MAX_AGENTS=6. If lock errors surface during implementation, a retry-with-backoff wrapper is a trivial addition.
  - Status: RESOLVED

- [x] D10. Branch accumulation — resolved branches never cleaned up.
  - Resolution: Branches are cheap (40 bytes), useful for bisect/audit — not inherently a problem. Added two TODOs: (1) TUI setting for auto-delete worktree branches on merge detection, (2) TUI key command to manually delete a merged worktree branch when viewing a Resolved issue.
  - Status: RESOLVED

- [x] D11. aidi doesn't know which agent is calling.
  - Resolution: Not a real problem — every `aidi` command takes `<inum>` as an explicit argument (`aidi respond 7 "..."`, `aidi block 7 --by 3`). The agent knows its inum from the issue.md header. Define an issue.md header spec: `inum`, `title`, `status`, `description`, `blocked_by`, `blocks`. This gives the agent all context it needs to identify itself and its issue.

Needs Decision

- [ ] P1-5. Ink's `useInput` has no event consumption — all handlers fire for every keypress.
  Discovered during Phase 1.11 manual UX testing. Ink's `useInput` hook registers handlers on a flat `process.stdin` listener list. Every registered handler fires for every keypress — there is no propagation, no bubbling, no way for a child component to "consume" an event and prevent parent components from also processing it. This caused multiple bugs:
  - Double-fire Esc: pressing Esc in Detail view fired both Detail's handler (calling `onBack`) and App's handler (calling `goBack`), causing two navigation pops in one keypress.
  - Key stealing: App-level global shortcuts (e.g., `n` for New Issue, `q` for Quit) fired even when a child view needed those keys (e.g., `n` for next issue in GroupView, any key for text entry in Detail).
  - Workaround fragility: the current `VIEW_OWNED_KEYS` pattern (a `Record<ViewType, ReadonlySet<string>>` declaring which keys each view "owns") prevents App from processing owned keys, but: (a) it must be manually maintained per view, (b) it doesn't prevent sibling or grandchild handlers from double-firing, (c) Detail view must list every typeable character to prevent global shortcuts from firing during text entry.

  Decision: Replace `useInput` with a JUCE-inspired key event system: `KeyEventContext` + `useKeyEvent` + focus tracking.

  Reference model: JUCE's `ComponentPeer::handleKeyPress` (juce_ComponentPeer.cpp:196-233). The core pattern is a single dispatch loop that walks from the focused component up the component tree. Each component gets a chance to consume the event via `keyPressed() → bool`. Returns `true` = consumed (stop propagating). Returns `false` = pass to parent. Base `Component::keyPressed()` returns `false` by default, so unhandled keys naturally bubble. Tab handling is a framework-level fallback after all components pass.

  Design principles (adapted from JUCE for React/Ink):

  1. Boolean return = consumed (from JUCE `keyPressed → bool`):
     Every `useKeyEvent` handler returns `true` (consumed, stop) or `false` (ignored, bubble up). This is the same convention as `Component::keyPressed`. No event object mutation, no `stopPropagation()` — just a return value. The base case is `false` (transparent to all keys), matching JUCE's `Component::keyPressed` default.

  2. Child-first dispatch, walking up the tree (from JUCE `handleKeyPress` loop):
     JUCE's dispatch is: `for (target = focused; target != null; target = target->getParentComponent())`. Our equivalent: a `KeyEventContext` provider at the App level intercepts `process.stdin` keypresses before Ink's `useInput` fires. It walks an ordered handler list (registered by `useKeyEvent` hooks). Handlers are ordered child-first by registration depth: deepest component registers first, App registers last. When a handler returns `true`, the walk stops. This replicates JUCE's parent-walk without needing actual DOM traversal.

     Implementation detail: each `useKeyEvent(handler, depth)` call registers the handler with a numeric depth (0 = App, 1 = view, 2 = sub-region like text input). The context sorts handlers by descending depth before dispatch. On each keypress, the context iterates handlers in order; first `true` return stops the loop.

  3. Focus tracking via context (from JUCE `getCurrentlyFocusedComponent` + `setWantsKeyboardFocus`):
     JUCE determines the dispatch target via `Component::getCurrentlyFocusedComponent()`. Components opt in via `setWantsKeyboardFocus(true)`. Our equivalent: `KeyEventContext` tracks a `focusedRegion: string` (e.g., `'text-input'`, `'conversation'`, `'dependency-links'`, `'list'`). Each `useKeyEvent` handler receives the current `focusedRegion` as a parameter and can decide whether to handle based on it. Only one region is focused at a time.

     Tab cycles focus between regions within a view (matching JUCE's tab-to-next-sibling fallback at line 223-230 of `handleKeyPress`). The Tab fallback is handled at the context level after all handlers return `false`, not inside any individual component — same as JUCE.

  4. Visual focus indicator:
     The focused region renders with a visual indicator (colored border, highlight). Unfocused regions render dimmed. This gives clear feedback about which component will receive keypresses. Each view reads `focusedRegion` from context and conditionally applies styling.

  5. TextInput coexistence:
     `ink-text-input` registers its own internal `useInput` that cannot be replaced. Two options:
     - (a) Coexist: TextInput keeps its own handler for text entry (characters, backspace, cursor). Our `useKeyEvent` handles navigation keys (Esc, Tab, global shortcuts). They don't conflict because TextInput only mutates text, never navigation. When `focusedRegion === 'text-input'`, the view's `useKeyEvent` handler returns `true` for all printable characters to suppress global shortcuts — analogous to how JUCE's `TextEditor::keyPressed` returns `true` for typeable characters (line 2182-2188) but `false` for unrecognized keys.
     - (b) Replace: Build a custom text input component that participates in `useKeyEvent` natively. Full control, like JUCE's `TextEditor` which is a Component with its own `keyPressed` override.
     - Preferred: (a) for now. Swap to (b) if TextInput's behavior conflicts with focus management (e.g., Tab-to-cycle-focus in Detail view — TextInput may swallow Tab).

  6. Eliminates VIEW_OWNED_KEYS:
     The per-view key ownership map (`Record<ViewType, ReadonlySet<string>>`) becomes unnecessary. Each component's handler naturally consumes only the keys it handles and returns `false` for everything else — exactly like JUCE components that only override `keyPressed` for their specific keys. App's handler at depth 0 acts as the root component, catching anything the active view didn't consume.

  Architecture mapping (JUCE → TUI):

  | JUCE Concept | TUI Equivalent |
  |---|---|
  | `ComponentPeer::handleKeyPress` loop | `KeyEventContext` dispatch loop over sorted handler list |
  | `Component::keyPressed() → bool` | `useKeyEvent(handler, depth) → handler returns bool` |
  | `Component::keyPressed` default returns `false` | Handlers return `false` for unrecognized keys |

  | `setWantsKeyboardFocus(true)` | Registering a `useKeyEvent` handler at a given depth |
  | `KeyListener` (external observers) | Not needed — React's composition model handles this |
  | Tab fallback after all components pass | Context-level Tab handler after all `useKeyEvent` handlers return `false` |
  | `TextEditor::keyPressed` consuming typeable chars | TextInput region's handler returning `true` for printable chars when focused |
