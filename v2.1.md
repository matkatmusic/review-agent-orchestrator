# Plan: Question Review Orchestrator — v2

## Context

The v1 file-based question system didn't work reliably — `is_blocked()` failed, agents spawned despite unresolved blockers, and the XML tag format was brittle. The v2 redesign replaces question files entirely with a **SQLite database** and a **TUI** (terminal user interface) for user interaction, but was buggy.  This is a complete rewrite from scratch of v2, starting from a proper design phase before anything is built.

The v1 system has outgrown bash. **Rewriting as a Node.js/TypeScript app** gives type safety, native SQLite access via `better-sqlite3`, a TUI framework, proper error handling, and testability.

The result is a **repo-level ticketing system** — like Jira but offline, portable, and travels with the repo. Clone the repo, run setup, and all questions/responses/dependencies are immediately available.  Git worktrees are used to isolate agent workspaces and enable parallel work on multiple Issues.  When agents are done with their work, they ask to apply their changes to the current branch (not necessarily `main`).  when granted permission to apply, their changes are unstaged for review, similar to a pull request.

## Design Decisions (consolidated)

- **SQLite database** — single source of truth for questions, responses, dependencies, and status
- **SQL dump tracked in git** — `questions.dump.sql` is the portable, diffable artifact. The `.db` file is gitignored (runtime artifact rebuilt from dump)
- **Auto-export** — daemon writes `questions.dump.sql` after every scan cycle that changes the DB
- **TUI** — primary user interface. No more editing question files directly
- **No question files** — question content lives in the database, not markdown files on disk
- **qnum as primary key** — stable, embedded in DB rows, never changes
- **`lastQuestionCreated`** — metadata row tracking the highest Q number assigned
- **Agents interact via CLI tool** — `qr-tool read <qnum>`, `qr-tool respond <qnum> "..."`, etc.
- **Groups** — optional grouping of related questions. A group is resolved when ALL members are resolved
- **`blocked_by`/`blocks`** — reference individual qnums. To block on a group, expand to all qnums in that group
- **`blocked_by` is purely DB-managed** — agents call `qr-tool block-by`, no file-level annotations
- **Worktrees**: qnum-based (`.claude/worktrees/Q7`, branch `worktree-Q7`)
- **Logs**: `[review] Spawned agent pane: Q7 — Migrate ServerDerivedFields`
- **Scan loop**: DB-driven (query entries with status=Active)
- **Statuses**: `Awaiting` (unblocked backlog) → `Active` (agent working) → `Resolved` / `Deferred` (user-deferred or blocked)
- **Enforce blocked invariant**: blocked questions cannot be `Awaiting` or `Active`; daemon enforces every cycle
- **Auto-unblock**: `Deferred` questions whose blockers are all `Resolved` auto-move to `Awaiting`
- **User-deferred vs system-blocked**: distinguished by presence of `blocked_by` entries. System-blocked auto-unblock; user-deferred stay until manually moved

## Database Schema

```sql
CREATE TABLE metadata (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Seed: INSERT INTO metadata VALUES ('lastQuestionCreated', '0');

CREATE TABLE questions (
    qnum        INTEGER PRIMARY KEY,
    title       TEXT NOT NULL,          -- short label (e.g., "migrate_ServerDerivedFields")
    description TEXT NOT NULL DEFAULT '',  -- first ~200 chars, used in TUI list views
    "group"     TEXT,                   -- optional group name for related questions
    status      TEXT NOT NULL DEFAULT 'Awaiting'
                CHECK (status IN ('Awaiting', 'Active', 'Deferred', 'Resolved')),
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    resolved_at TEXT
);

CREATE TABLE responses (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    qnum       INTEGER NOT NULL REFERENCES questions(qnum),
    author     TEXT NOT NULL CHECK (author IN ('user', 'agent')),
    body       TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
CREATE INDEX idx_responses_qnum ON responses(qnum);

CREATE TABLE dependencies (
    blocker_qnum INTEGER NOT NULL REFERENCES questions(qnum),
    blocked_qnum INTEGER NOT NULL REFERENCES questions(qnum),
    PRIMARY KEY (blocker_qnum, blocked_qnum),
    CHECK (blocker_qnum != blocked_qnum)
);
CREATE INDEX idx_deps_blocked ON dependencies(blocked_qnum);
CREATE INDEX idx_deps_blocker ON dependencies(blocker_qnum);
```
