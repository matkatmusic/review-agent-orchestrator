## ALL THREE PROVIDERS AGREE (highest confidence)

### 1. HIGH — addResponse() lacks atomicity
# Providers: Claude, Gemini, Codex
# Location: responses.ts:10-23
# Issue: addResponse() does INSERT into responses + UPDATE on questions
# tracking columns as two separate db.run() calls. If the UPDATE fails,
# the response is inserted but last_responder/last_user_response are
# permanently desynced. needsReprompt() will return wrong answers.
# Fix: Wrap both calls in db.transaction(), same pattern as createQuestion().
DECISION: Wrap both calls in db.transaction()
STATUS: IMPLEMENTED — Wrapped INSERT + UPDATE in db.transaction() in addResponse(). 18/18 responses tests pass.

### 2. MEDIUM — cmdStatus() missing User_Deferred
# Providers: Claude, Gemini, Codex
# Location: qr-tool-commands.ts:97-118
# Issue: cmdStatus() initializes counts with only 4 statuses (Awaiting,
# Active, Deferred, Resolved). User_Deferred questions are counted in Total
# but never displayed in the breakdown. Users see mismatched numbers.
# Fix: Add User_Deferred: 0 to counts initializer and add output line.
DECISION: Add User_Deferred: 0 to counts initializer and add output line
STATUS: IMPLEMENTED — Added User_Deferred to counts and output in cmdStatus(). Updated test assertion. 19/19 qr-tool tests pass.

### 3. HIGH — blockByGroup() bypasses cycle detection
# Providers: Claude, Gemini, Codex
# Location: dependencies.ts:81-88
# Issue: blockByGroup() does a bulk INSERT OR IGNORE directly via SQL,
# bypassing the wouldCreateCycle() check enforced by addBlocker(). Example:
# Q1 blocks Q2, Q2 is in group "G". blockByGroup(1, "G") creates Q1->Q2->Q1 cycle.
# Fix: Replace bulk SQL with iteration through addBlocker() which includes
# cycle detection, or replicate the CTE check in the bulk path.
DECISION: Replace bulk SQL with iteration through addBlocker()
STATUS: IMPLEMENTED — blockByGroup() now iterates candidates through addBlocker() which includes cycle detection. New test verifies cycles are skipped. 19/19 dependency tests pass.
